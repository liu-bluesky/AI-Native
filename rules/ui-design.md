# UI 设计规则

> 本文件以 `/intro` 页面为当前项目的视觉基准。
> 后续涉及页面布局、样式、组件皮肤、动效和设计 token 时，默认先遵守这里，再做场景化变体。

## 1. 设计目标

- 整体气质必须接近 `/intro`：克制、先进、轻盈、可信，不做传统后台卡片堆叠风。
- 视觉重点必须服务信息判断，不允许为了“丰富”堆砌模块、装饰和高饱和颜色。
- 页面需要体现“AI Operating System”式的系统感：统一上下文、连续流程、低噪音界面。
- 同一产品内的公共页、工作台页、设置页都应共享同一套色彩、圆角、边框、动效语言。

## 2. 核心风格

### 2.1 关键词

- 克制
- 轻玻璃感
- 浅色底
- 空气感留白
- 冷静科技感
- 品牌化但不浮夸

### 2.2 禁止事项

- 禁止大面积纯色重底或厚重深色面板
- 禁止后台式统计卡、说明卡、功能入口卡无节制堆叠
- 禁止高饱和蓝紫霓虹铺满主界面
- 禁止把多个区域都做成视觉主角
- 禁止通篇强阴影、重描边、厚玻璃、重拟物
- 禁止动效频繁、急促、跳跃，破坏稳定感

## 3. 视觉基准

### 3.1 页面背景

页面背景默认采用 `/intro` 的浅色渐变和环境光，而不是纯白或纯灰：

```css
background:
  radial-gradient(circle at 18% 0%, rgba(125, 211, 252, 0.16), transparent 26%),
  radial-gradient(circle at 82% 14%, rgba(103, 232, 249, 0.12), transparent 22%),
  linear-gradient(180deg, #f5f4ef 0%, #f8fafc 38%, #edf2f7 100%);
```

补充要求：

- 可局部叠加弱光晕、网格或径向高光，但必须透明、稀薄、不可抢正文。
- 环境装饰应放在页面最底层，且 `pointer-events: none`。
- 除特定沉浸式场景外，不允许直接切到深色主题。

### 3.2 色彩角色

- 主文字：`#0f172a`
- 次级正文：`#475569`
- 弱说明：`#7c8aa0`
- 细边框：`rgba(15, 23, 42, 0.08)`
- 强边框：`rgba(15, 23, 42, 0.12)`
- 浅面板：`rgba(255, 255, 255, 0.72)`
- 强面板：`rgba(255, 255, 255, 0.84)`
- 深色强调：`#0f172a`
- 蓝色高光：`#38bdf8`
- 冰青高光：`#67e8f9`

规则：

- 高光色只用于边缘、轨迹、按钮 hover、局部标签、进入动效。
- 深色强调按钮是少数主焦点，不允许全页到处都是深色实心按钮。
- 功能色仍只用于语义提示，不允许承担品牌主色职责。

### 3.3 材质与阴影

- 容器默认使用轻玻璃感，不做厚玻璃。
- 基础卡片使用浅透明白底、细边框、轻阴影和 `backdrop-filter: blur(...)`。
- 阴影必须轻，优先靠透明度、边界光和层级建立结构。

当前基准：

- 主阴影：`0 24px 64px rgba(15, 23, 42, 0.08)`
- 次阴影：`0 14px 34px rgba(15, 23, 42, 0.06)`
- 大圆角：`34px`
- 中圆角：`24px`
- 小圆角：`18px`
- 胶囊圆角：`999px`

## 4. 版式系统

### 4.1 页面宽度

- 主内容容器宽度优先采用：`min(1180px, calc(100% - 40px))`
- 页面区块之间依赖留白建立节奏，不依赖粗边框和厚背景分隔
- 默认使用大区块 + 中等间距，而不是密集网格

### 4.2 信息层级

一个页面通常只允许以下 3 层：

```text
页面背景层
内容区块层
局部强调层
```

补充要求：

- 不要额外制造第 4 层“套壳容器”
- 一个屏幕只保留一个绝对视觉焦点
- 次要说明必须退后，不与标题和主操作竞争

### 4.3 区块结构

推荐沿用 `/intro` 的区块节奏：

- `eyebrow`：小号大写引导词
- `title`：短标题，强判断
- `text`：1 到 2 句解释
- `actions`：最多 1 个主按钮 + 1 个次按钮
- `surface`：玻璃容器或信息卡

## 5. 字体与文案

### 5.1 字体层级

- `eyebrow` 使用 `12px` 左右、小写转大写、较大字距
- 主标题必须短，优先 1 行到 2 行
- 正文不写成说明书，每段控制在 2 到 3 行内
- 卡片标题保持短句，不写长标题解释

### 5.2 文案风格

- 先给判断，再给解释
- 多用短句，少用从句
- 文案应像产品叙事，不像后台字段说明
- 避免“平台可支持”“系统能够实现”这类空泛表达

示例风格：

- `让 AI 真正进入生产。`
- `更短的路径，而不是更多页面。`
- `四步进入生产。`

## 6. 导航与头部

### 6.1 顶部导航

- 顶部导航应轻薄、悬浮、带轻毛玻璃感
- 允许 `sticky`，但要避免变成传统后台顶栏
- 品牌区要小，不做大 logo 强压正文

当前基准：

- 顶部间距：`top: 12px`
- 内边距：`14px 18px`
- 胶囊形导航
- 半透明浅底 + 轻阴影 + blur

### 6.2 导航项

- 默认透明文字按钮
- hover 只做轻微底色和文字增强
- 数量要少，只保留真正核心锚点或入口

## 7. Hero 与主视觉

### 7.1 Hero 原则

- Hero 要在 5 秒内回答“这是什么”和“为什么值得继续看”
- 左侧负责价值判断，右侧负责系统感展示
- 右侧主视觉必须像智能工作台，不是后台截图拼贴

### 7.2 主视觉结构

推荐三层：

- 底层：柔和光晕 / mesh / 背景场
- 中层：主中枢装置或核心内容块
- 上层：连接线、节点、轻标签、局部状态点

### 7.3 禁忌

- 禁止把 Hero 做成功能清单
- 禁止直接塞 dashboard 截图
- 禁止赛博朋克式霓虹满屏发光
- 禁止复杂 3D 连续旋转

## 8. 卡片与容器

### 8.1 基础容器

适用于内容卡、工作流卡、说明卡、CTA 容器：

```css
border: 1px solid rgba(255, 255, 255, 0.84);
border-radius: 30px;
background: rgba(255, 255, 255, 0.72);
box-shadow: 0 14px 34px rgba(15, 23, 42, 0.06);
backdrop-filter: blur(20px);
```

### 8.2 卡片规则

- 卡片内部信息要少，避免密集字段
- 卡片标题、正文、辅助标签要有明确优先级
- hover 只允许轻微上浮、边框增强或阴影增强
- 不允许 hover 时剧烈缩放、旋转或大面积颜色变化

### 8.3 卡片分组

- 优先使用 2 到 4 列均衡布局
- 不要用太多小卡片切碎页面
- 当信息不足时，宁愿减少卡片数量，也不要硬凑栅格

## 9. 按钮与交互控件

### 9.1 主按钮

- 主按钮使用深色实心，承担明确转化动作
- 需要轻微扫光或边缘提亮，但不能浮夸

当前基准：

- 背景：`#0f172a`
- 文字：白色
- 圆角：胶囊或大圆角
- hover：轻微上浮 + 阴影增强 + 高光扫过

### 9.2 次按钮

- 次按钮使用浅底、轻边框或透明底
- 不与主按钮争夺视觉焦点
- hover 时只做轻背景与边框增强

### 9.3 图标按钮

- 图标按钮默认透明或极浅底
- 用于设置、次级入口、卡片局部动作
- 不允许一排都做成实心圆按钮

## 10. 动效语言

### 10.1 原则

- 动效必须慢、轻、稳
- 作用是传达“系统在运行”，不是让页面显得热闹
- 任何动效都不能妨碍阅读

### 10.2 推荐动效

- 模块进入视口：淡入 + 上浮
- 环境光晕：缓慢呼吸
- 连线或按钮：极弱扫光
- 局部节点：轻微脉冲

当前基准：

- 进入动画：`600ms ~ 800ms`
- 位移：不超过 `24px`
- 光晕周期：`8s ~ 18s`
- 连线流光周期：`3s ~ 4s`

### 10.3 不建议动效

- 高频跳动
- 大面积粒子雨
- 连续旋转光环
- 夸张弹跳
- 整屏闪烁或持续扫描

## 11. 响应式规则

### 11.1 桌面端

- 优先保留主标题宽度、Hero 结构和区块留白
- 桌面端允许双栏和非对称布局，但必须均衡

### 11.2 平板端

- 当宽度收缩时，优先把并列区块改为纵向堆叠
- 导航链接数量减少，避免横向拥挤
- 保留主视觉层次，但简化装饰

### 11.3 小屏端

- 页面容器左右留白收敛
- 所有栅格退化为单列
- 装饰性背景和轻视差可以降级或关闭
- 保留标题、正文、主 CTA 的垂直节奏，避免压扁

## 12. 不同页面的继承方式

### 12.1 公共展示页

- 直接继承 `/intro` 的完整语言
- 可以使用 Hero、工作流、品牌化 CTA、可视化装置

### 12.2 产品工作页

- 继承同一套颜色、圆角、边框、阴影和动效语言
- 但减少营销化 Hero，强化任务路径和内容密度控制
- 不得回退成老式后台系统风格

### 12.3 设置与列表页

- 保持轻玻璃感和浅色底
- 结构更克制，强调信息组织
- 列表、表单、筛选区应是 `/intro` 语言的功能化变体，而不是另一套设计系统

## 13. 开发禁忌清单

以下改动默认视为错误：

- 把页面重新做成厚重后台工作台
- 随意引入高饱和紫色、霓虹蓝、大片深底
- 到处堆 Hero、统计卡、入口卡、说明卡
- 主按钮过多，导致全页多个主焦点
- 所有卡片都加重阴影、重边框、重毛玻璃
- 文案写成长段帮助文档
- 动效速度过快、数量过多、持续干扰阅读
- 做出和 `/intro` 完全无关的新视觉语言

## 14. 落地检查清单

设计或实现新页面后，至少自检以下项目：

- 页面是否仍然保留 `/intro` 的浅色渐变和空气感
- 是否只有一个主视觉焦点
- 主标题是否够短、够强
- 卡片是否轻而不是厚
- 高光色是否只用于点缀
- 是否存在多余统计卡、说明卡、入口卡
- 主按钮是否只有一个主要转化动作
- 动效是否慢、轻、稳
- 平板和移动端是否自然退化
- 页面是否看起来像同一产品，而不是新拼出来的后台模板

## 15. 一句话原则

以 `/intro` 为准，做“被光轻轻点亮的 AI 产品界面”，而不是“功能很多的后台页面”。
