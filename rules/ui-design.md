# /ai/chat UI 设计规则

> 本文件直接覆盖旧版通用 UI 规范。
> 后续只要涉及 `/ai/chat` 或同类 AI 对话页，必须优先遵守这里，不允许回退成“后台卡片堆叠式”设计。

## 1. 设计目标

- 页面气质必须接近 ChatGPT / DeepSeek 这类“内容优先”的对话界面，不做传统后台工作台。
- 首要任务是让用户快速进入对话，而不是先阅读一堆说明、指标卡、信息卡。
- 页面必须克制，优先做减法；任何新增模块都要先证明不是重复信息。
- 信息层级只保留 3 层：页面骨架、消息内容、输入动作。不要再人为制造第 4 层大容器。

## 2. 核心风格

### 2.1 关键词

- 简洁
- 通透
- 中性
- 低噪音
- 内容驱动

### 2.2 禁止事项

- 禁止大面积后台式卡片、统计卡、说明卡、英雄区堆叠
- 禁止在右侧聊天区再套一层厚重背景容器
- 禁止同一信息在侧栏、头部、空状态、输入区重复出现
- 禁止为了“丰富”而添加多余图标、标签、辅助文案
- 禁止高饱和主色大面积铺底

## 3. 页面骨架

页面结构固定为两栏：

```text
左侧：会话与项目侧栏
右侧：上下文栏 + 提示横幅（可选）+ 消息流 + 输入区
```

布局规则：

- 外层使用双栏布局，不再额外引入大面板包裹右侧内容
- 左栏固定窄宽度，右栏自适应撑满剩余空间
- 当前基准值：
  - 侧栏宽度：`272px`
  - 栏间距：`18px`
  - 页面内边距：`14px 16px 16px`
- 页面背景统一由最外层提供，右侧聊天区背景必须继承，不再单独做灰底/白底大面板

### 3.1 派生界面继承规则

- 只要仍属于 `/ai/chat` 的体验范围，例如设置中心、技能资源、员工选择、会话辅助面板，都视为聊天页派生界面。
- 派生界面必须优先继承主聊天页现有的布局语汇和视觉语言，包括侧栏结构、上下文栏节奏、列表选中态、内容区宽度、输入区风格。
- 禁止把派生界面单独设计成另一套“后台设置页”或“管理工作台”风格。
- 做减法时，不能只停留在“把卡片变淡、把边框变细”，必须先判断它是否仍然像同一个产品页面自然延展出来的区域。
- 如果主聊天页已经存在成熟样式，优先复用或局部变体，禁止重新发明一套新的导航、列表、头部、面板语言。

## 4. 背景与色彩

### 4.1 页面背景

使用轻微暖灰渐变，不是纯白，也不是深色：

```css
background:
  radial-gradient(
    circle at top left,
    rgba(255, 244, 214, 0.5),
    transparent 24%
  ),
  linear-gradient(180deg, #f6f3ee 0%, #f7f7f8 32%, #f5f5f6 100%);
```

### 4.2 颜色角色

- 主标题：`#111827` ~ `#171717`
- 正文：`#2b2f36`
- 次级说明：`#6b7280` / `#737373`
- 占位与弱提示：`#9ca3af` / `#a3a3a3`
- 细边框：`rgba(17, 24, 39, 0.06 ~ 0.10)`
- 浅灰交互底：`#f4f4f5` / `#eceff3`
- 用户气泡底：`#e9edf2`
- 白色实体仅用于输入容器、下拉浮层、空状态按钮、草稿卡片内子块

### 4.3 阴影原则

- 页面主骨架不要大阴影
- `chat-stage` 这类中间包裹层禁止阴影、禁止描边、禁止独立背景
- 输入框允许使用非常轻的阴影，作为唯一明确的操作焦点
- 其余容器默认 `box-shadow: none`

## 5. 左侧侧栏规则

### 5.1 视觉原则

- 侧栏背景与页面底色保持一致，不能看起来像另一张卡片
- 侧栏允许有结构，但不能过度装饰
- 品牌区保持轻量，只做页面身份，不做视觉主角

### 5.2 品牌区

- 左上角 `AI` 标记允许保留，但必须很小
- 品牌文案两行以内：主标题 + 极短副标题
- 设置按钮使用透明圆形图标按钮，不做主按钮样式

### 5.3 项目选择器

- 选择项目组件只展示项目名称
- 禁止展示 agent 信息、附加描述、冗余标签
- 宽度必须与下方“新对话”按钮一致
- 文案垂直居中
- 无数据时，菜单里明确展示：`暂无项目`

### 5.4 会话列表

- 列表是侧栏重点，不要再外套多层卡片
- 当前会话高亮可以使用浅灰底，不要使用高饱和主色块
- 一条会话只显示必要信息：标题、极少量辅助时间/状态

## 6. 右侧头部与提示区

### 6.1 上方上下文栏

- 只保留当前会话标题和极少量元信息
- 元信息用小字、居中、点分隔
- 不能做成工具栏式密集按钮区
- padding 保持轻量：`8px 24px 6px`

### 6.2 横幅提示

- 仅在连接器未选、路径未配、路径未保存等阻断场景下出现
- 横幅宽度跟随内容区居中，不铺满整屏
- 基准宽度：`min(100%, 780px)`
- 横幅是提示，不是页面主角，圆角和颜色都要克制

## 7. 消息流规则

### 7.1 整体宽度

- 消息区是页面核心，必须优先保证阅读舒适度
- 内容容器居中
- 消息流内层最大宽度：`920px`
- AI 消息正文最大宽度：`860px`
- 输入区宽度必须与消息区阅读宽度同一视觉体系

### 7.2 空状态

- 空状态必须像“开始一轮对话”，不是系统空白页
- 标题可以较大，但只允许一句
- 描述文字控制在短段落，不写长说明书
- 推荐问题使用轻按钮，不做主按钮堆叠

当前基准：

- 空状态最小高度：`420px`
- 标题字号：`30px`
- 推荐问题按钮最小宽度：`180px`

### 7.3 AI 消息

- AI 消息背景默认透明，不强制做白卡片
- AI 正文直接铺开阅读，重点是文本本身
- 文本字号：`15px`
- 行高：`1.9`
- 不显示冗余消息 meta 行

### 7.4 用户消息

- 用户消息靠右
- 使用浅灰底胶囊气泡
- 不能过窄，也不能固定死宽

当前基准：

- 最大宽度：`min(760px, 78%)`
- 最小宽度：`min(240px, 100%)`
- 圆角：`22px`
- 内边距：`12px 16px`

### 7.5 衍生消息块

适用于：

- thinking / 过程面板
- 审计信息
- 员工草稿卡片
- 附件
- 图片预览

规则：

- 这些块必须比正文弱一级，只作为辅助层
- 使用极浅灰底、轻边框、小圆角
- 禁止强阴影
- 禁止和主正文竞争视觉重心

## 8. 输入区规则

### 8.1 输入区定位

- 输入区是页面第二核心
- 必须固定在消息区下方，并居中
- 视觉上接近 OpenAI 的宽输入壳，而不是传统表单 textarea

### 8.2 宽度

当前强制规则：

```css
width: clamp(500px, 72vw, 800px);
min-width: 500px;
max-width: 100%;
```

补充要求：

- 桌面端输入区不要明显窄于消息区
- 没选项目时，输入区宽度也不能额外缩窄
- 小屏下再退化为 `width: 100%; min-width: 0;`

### 8.3 输入壳层

- 外壳圆角：`28px`
- 背景：半透明白
- 细边框 + 极轻阴影
- 聚焦后只允许轻微增强，不允许发光感过强

### 8.4 文本域

- 最小高度：`88px`
- 上内边距大于下内边距，让输入更像对话工具，而不是表单框
- placeholder 必须简短直接
- hint 文案如果噪音高，可以隐藏；不要为“解释功能”牺牲输入效率

## 9. 能力栏规则

这里指 `.composer-assist-strip` 一排功能按钮。

### 9.1 角色定义

- 它属于输入区的一部分，不是独立功能模块
- 作用是快速写入输入意图，而不是展示系统能力全貌

### 9.2 样式

- 使用胶囊按钮
- 小尺寸、浅灰底、轻边框
- 默认不抢眼，激活后仅轻微加深

当前基准：

- 高度：`32px ~ 34px`
- 圆角：`999px`
- 图标 + 文案横排
- 按钮间距：`8px`

### 9.3 交互要求

- 点击后必须立即更新输入框内容
- 同一时刻只允许一个激活项高亮
- 不再展示 `composer-assist-active-banner` 这类重复提示区
- 常用项直出，溢出项收进 `更多`

## 10. 选择器与轻操作控件

### 10.1 下拉选择器

- 选择器触发器要简洁，单行展示
- 当前页项目选择器只保留项目名
- 宽度必须与同区块主要按钮对齐
- 下拉菜单宽度尽量与触发器同步

### 10.2 图标按钮

- 设置、附件、次级操作一律走轻图标按钮
- 默认透明或极浅底
- hover 才给轻背景
- 不允许多个实心图标按钮并排抢焦点

### 10.3 发送按钮

- 发送按钮可以是页面里少数深色实心按钮之一
- 但尺寸必须小，不做大号 CTA

当前基准：

- 尺寸：`36px * 36px`
- 背景：`#111827`
- 禁用态：`#d4d4d8`

## 11. 响应式规则

### 11.1 中屏

- 在 `1120px` 左右开始收敛双栏布局
- 优先保护消息内容和输入区宽度
- 允许侧栏下移或压缩，但不要先压缩正文阅读宽度

### 11.2 小屏

- `640px` 以下输入区宽度切到 `100%`
- 减少左右 padding
- 允许隐藏非必要 meta、减少装饰元素
- 优先保留文本宽度，不要让气泡变得特别窄

## 12. 文案规则

- 所有文案优先短句
- 项目选择器、快捷动作、空状态按钮都必须是动作导向
- 不要写“平台说明式”长文
- 同一语义只出现一次，不重复解释

## 13. 开发禁忌清单

以下改动默认视为错误：

- 给右侧聊天区重新套一个带背景的大面板
- 重新引入 Hero、数据卡、说明卡到 `/ai/chat`
- 在项目下拉中塞 agent 信息、状态标签、描述文本
- 让输入框宽度明显小于消息阅读区
- 把用户气泡写成固定窄宽度
- 用高饱和蓝色大面积覆盖按钮或容器
- 为能力条增加额外说明 banner
- 增加 `chat-stage` 这类只有视觉装饰作用的包裹层样式

## 14. 落地检查清单

设计或改动 `/ai/chat` 后，至少自检以下项目：

- 右侧聊天区背景是否完全继承页面底色
- 侧栏是否仍然像独立卡片，如果像，需要继续减
- 项目选择器是否只显示项目名称
- 项目选择器宽度是否与下方主按钮一致
- 无项目时是否明确显示 `暂无项目`
- 消息区是否足够宽，AI 正文是否在舒适阅读宽度内
- 用户气泡是否不再窄小、固定死宽
- 输入框是否在桌面端保持 `500 ~ 800px` 的合理宽度
- 能力按钮点击后是否立即写回输入框
- 页面是否还存在重复说明、重复状态、重复模块

## 15. 一句话原则

`/ai/chat` 的设计不是“把更多功能摆出来”，而是“把对话做成唯一主角”。
