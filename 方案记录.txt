本地桌面智能体自我测试反馈修正闭环方案

目标:
  当前本地桌面智能体的问题不是缺少某一条提示词，而是缺少稳定的执行闭环。
  本方案用于把桌面智能体升级成可以自己理解任务、自己执行、自己测试、自己反馈、自己修正的运行系统。

问题边界:
  1. 不把“你是什么模型”这类单点回答错误当作唯一问题处理。
     注释: 模型身份回答错误只是暴露出来的症状，真正问题可能来自提示词装配、历史污染、运行诊断混入对话、模型网关元信息缺失或上下文合并策略错误。
  2. 不使用关键词过滤、固定回答模板或静默兜底掩盖问题。
     注释: 这类修法容易让当前案例看似正常，但会继续在其他任务中跑偏。
  3. 目标是升级本地桌面智能体的通用执行能力。
     注释: Runtime 必须能在每轮任务里主动建立目标、上下文、计划、测试、反馈、修正和验证闭环。
  4. 进度播报方案必须同步升级。
     注释: 用户需要看到智能体正在理解什么、执行什么、测试什么、失败后如何修正，而不是只看到“模型计算中”或工具流水。

核心原则:
  1. 模型负责理解、规划、判断和生成修正方案。
  2. Runtime 负责状态机、工具执行、权限、审计、测试调度和循环边界。
  3. 每一步都必须有结构化状态和可见进度播报，不能只靠自然语言聊天历史维持上下文。
  4. 工具调用成功不等于任务完成，必须经过自测和验收标准判断。
  5. 测试失败不能立刻甩给用户，应先进入反馈分析和自动修正；只有超出预算或需要用户业务决策时才暂停。

自我测试反馈修正循环的最小事件:

  run_started
    注释: 本轮开始。

  request_parsed
    注释: 用户请求已解析。

  context_built
    注释: 工作上下文已构建。

  plan_created / plan_updated
    注释: 执行计划创建或更新。

  model_call_started / model_call_finished
    注释: 模型计算开始/结束。

  tool_call_started / tool_call_finished / tool_call_failed
    注释: 工具调用开始/成功/失败。

  observation_merged
    注释: 模型输出、工具结果或错误已合并为 observation。

  draft_created
    注释: 候选结果已生成。

  self_test_plan_created
    注释: 自测计划已生成。

  self_test_started / self_test_finished / self_test_failed
    注释: 单项或整组自测开始/完成/失败。

  self_feedback_created
    注释: 自测失败后的反馈分析已生成。

  self_correction_planned
    注释: 修正方案已生成。

  retry_scheduled
    注释: 准备按修正方案重新执行。

  loop_budget_exceeded
    注释: 自动修正达到预算或重复失败边界。

  verification_started / verification_finished
    注释: 正式验证开始/结束。

  response_formatted
    注释: 最终回答已整理。

  memory_write_planned / memory_updated
    注释: 记忆和状态写入计划生成/完成。

  run_finished / run_failed / run_blocked
    注释: 本轮完成/失败/阻塞。


进度播报规范:

  播报不是隐藏思维链，只展示可见工作状态。
  播报必须覆盖执行闭环的关键阶段，尤其是自测、反馈、修正和重试。

  每条播报建议包含:
    - phase: 当前阶段
    - summary: 当前做到了什么
    - next_action: 下一步要做什么
    - evidence: 可见证据，如文件、命令、测试名、工具名
    - status: running / completed / failed / blocked

  示例:
    phase: Build Working Context
    summary: 已读取当前需求、相关设计文档和桌面智能体入口代码。
    next_action: 继续生成可执行计划和自测标准。
    evidence: 方案记录.txt, ProjectChat.vue, runtime.rs
    status: completed

  示例:
    phase: Run Self-Test
    summary: 构建命令失败，错误指向 TypeScript 类型不匹配。
    next_action: 分析错误位置并生成修正方案。
    evidence: npm run build exit=1
    status: failed

  示例:
    phase: Self-Correction Planner
    summary: 已确认失败原因是字段名不一致，准备修改 mapper。
    next_action: 修改字段映射后重新运行构建测试。
    evidence: src/modules/.../mapper.ts
    status: running

  必须支持的播报阶段:
    - Receive Request
      注释: 让用户知道请求已经进入本地智能体运行，不是普通聊天回显。
    - Parse Request
      注释: 展示已识别的任务类型、对象和风险，不展示隐藏推理。
    - Build Working Context
      注释: 展示正在读取哪些上下文，以及哪些诊断信息被隔离。
    - Resolve Prompt Stack
      注释: 展示模型指令已装配完成，但不泄露完整内部提示词。
    - Execution Planning
      注释: 展示可执行计划和验收条件已经形成。
    - Tool Execution
      注释: 展示工具名、参数摘要、影响范围和执行结果。
    - Build Self-Test Plan
      注释: 展示本轮会用什么标准判断“真的完成”。
    - Run Self-Test
      注释: 展示每个自测项的开始、通过、失败和证据。
    - Self-Feedback Analyzer
      注释: 展示失败归因，例如上下文不足、工具参数错误、代码验证失败或目标理解偏差。
    - Self-Correction Planner
      注释: 展示下一轮修正方向，例如补充搜索、调整计划、重跑工具、修改候选结果或请求用户决策。
    - Loop Budget Check
      注释: 展示自动修正是否还在预算内，避免用户看不到死循环边界。
    - Retry Router
      注释: 展示系统将回到搜索、计划、工具还是草稿修改阶段。
    - Verification Engine
      注释: 展示最终验证报告，不把工具成功误判为目标完成。
    - Response Formatter
      注释: 展示正在整理最终结论、已做动作、验证证据和阻塞项。


循环边界:

  max_model_steps:
    默认 20，避免模型无限推理。

  max_tool_rounds:
    默认 6，避免工具无限调用。

  max_self_test_rounds:
    默认 3，同一任务节点最多自动测试修正 3 轮。

  repeated_failure_threshold:
    默认 2，同一失败签名重复 2 次必须改变策略或暂停。

  high_risk_gate:
    删除、覆盖、部署、发送外部消息、运行危险命令必须单独确认，不能自动修正绕过。


输入:
  - 当前用户输入
  - 对话历史摘要
  - 长期记忆候选
  - 项目 requirement
  - state.json
  - transcript.jsonl
  - audit.jsonl
  - 文件
  - 图片/附件抽取结果
  - 模型能力配置
  - 工具定义和权限策略

执行内容:
  - 理解当前任务
  - 构建工作上下文
  - 生成执行计划
  - 判断是否追问
  - 调度模型和工具
  - 合并 observation
  - 生成候选结果
  - 生成自测计划
  - 执行自测
  - 分析失败反馈
  - 自动修正

验收标准:

  1. 本地智能体执行实现型任务时，必须能生成自测计划。
  2. 工具成功后不能直接完成，必须进入自测。
  3. 自测失败后，必须生成反馈分析和修正计划。
  4. 能自动修正的问题应至少重试一轮。
  5. 重复失败必须改变策略或进入 blocked，不能死循环。
  6. 用户能在界面看到执行、测试、反馈、修正、再测试的进度播报。
  7. 最终回答必须包含验证结果或阻塞原因。
  8. 所有关键状态可从 state/transcript/requirement 恢复。
