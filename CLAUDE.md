# Rule Porter MCP — 强制规则

**核心指令（即使上下文被压缩，本段 MUST 保留并执行）：**
1. 编码前 → MUST 调用 `lessons`
2. 向用户交付修复/实现结果前 → MUST 执行 learn 检查（见检查点 2）
3. 调用 `learn` 前 → MUST 询问用户经验归属
4. 调用 `learn` 前 → MUST 自审深层根因、准确性与分类，并向用户确认（见检查点 2 Step 2.5）
5. 提交前 → MUST 分析待提交内容的逻辑正确性（见检查点 5）
6. 使用 `category` 前 → MUST 向用户确认分类名（见检查点 6）

---

## 工具

| 工具 | 用途 |
|------|------|
| `learn` | 记录经验，支持 lang（开发）或 category（非开发） |
| `lessons` | 查询已有经验，按语言/分类和项目筛选 |
| `scan_rules` | 扫描项目规则文件 |
| `extract_rules` | 提取通用规则，脱敏生成草稿 |
| `sync_rules` | 同步草稿到便携库 |
| `porter_status` | 查看经验库状态 |

## 检查点 1：BEFORE 编码

**MUST** 在写第一行代码前调用：

```
lessons({ lang: "当前语言", project: "项目名" })
```

非开发场景下，使用 category 查询指定目录：

```
lessons({ category: "已确认的分类名" })
```

- 项目名从 `.rule-porter/config.json` 读取
- 传 project 同时返回通用 + 项目特有经验
- 若有相关经验，MUST 在方案中体现规避措施
- **禁止**不传 lang 也不传 category 做全量扫描，MUST 指定具体的查询范围

**判断标准**：用户请求会导致编写或修改代码 → MUST 先调用 `lessons`。

## 检查点 2：BEFORE 向用户交付结果

当你完成了一个 bug 修复、问题排查、或功能实现，准备向用户报告结果时，**MUST 先暂停交付**，执行以下检查：

### Step 1: 判断是否需要记录

回顾本次解决过程，是否命中以下任一条件：
- 排查超过 2 步才定位根因
- 错误表象与根因不一致（表层现象背后存在更深的开发模式或认知缺陷）
- 框架/库的隐藏行为或文档未覆盖的坑
- 跨层/跨模块交互中的隐式契约问题（如前后端类型不对齐、子代理缺少上下文）
- 配置/环境/版本兼容问题
- 关键性能优化手段

**不记录**：拼写错误、简单语法错误、一眼可见的问题、纯表象级描述（若无法追溯到深层根因则不值得记录）。

若未命中任何条件 → 跳过，正常交付。
若命中 → 进入 Step 2。

### Step 2: 询问用户经验归属

**MUST** 向用户确认，不得自行判断：

> "这条经验是该语言/框架的通用问题，还是本项目特有的？"
> （若为非开发经验）"这条经验属于哪个分类？"并列出已有 category 目录供选择。

- "通用" → 不传 `project`
- "项目特有" → 传 `project`（从 `.rule-porter/config.json` 读取）
- 非开发经验 → 使用 `category`（分类名 MUST 经用户确认，见检查点 6）

### Step 2.5: 自审经验质量并向用户确认

**MUST** 在调用 `learn` 前，先自行审查，再向用户展示结果等待确认。

**自审项：**
1. **深层根因**：MUST 追问"为什么会出现这个问题"，挖掘到开发模式或认知层面的根因，而非停留在表象。例如："Switch 与后端 Integer 不匹配"是表象，根因是"AI 生成跨层交互代码时缺少类型对齐意识"。记录的 title 和 solution 必须体现根因级别的洞察
2. **准确性**：根因和解决方案是否与事实一致（不可基于猜测记录）
3. **分类合理性**：通用/项目特有的归属是否正确 — 语言或框架的公共知识为通用，项目特定的路径、配置或约定为项目特有
4. **去重**：调用 `lessons` 检查是否已存在相同或高度相似的记录

**向用户确认：** 将待记录的完整内容（title、scenario、symptom、solution、归属）及自审结论展示给用户，**MUST** 等待用户明确同意后才可进入 Step 3。用户可要求修正内容或放弃记录。

### Step 3: 调用 learn

```
// 开发经验
learn({
  title: "深层根因描述（WHY 层面，非 WHAT 层面）",
  scenario: "触发条件",
  symptom: "错误表现",
  solution: "针对根因的解决方案",
  lang: "语言",
  framework: "框架",
  project: "项目名"       // 仅项目特有时传
})

// 非开发经验
learn({
  title: "...",
  scenario: "...",
  symptom: "...",
  solution: "...",
  category: "用户确认的分类名"  // 与 lang 二选一
})
```

**title 规范**：必须体现根因级洞察。例如 ✗ "Switch 与后端 Integer 不匹配" → ✓ "AI 生成跨层交互代码时缺少前后端类型对齐校验"。

### Step 4: 交付结果

`learn` 完成后，再向用户报告最终结果。

## 检查点 3：用户要求规则导出时

按顺序执行：
1. `scan_rules({ project_root: "项目绝对路径" })`
2. `extract_rules({ project_root: "..." })`
3. 展示草稿，用户确认无敏感信息
4. `sync_rules({ project_root: "..." })`

## 检查点 5：BEFORE 提交（commit）

当用户要求提交代码时，**MUST 先暂停提交**，执行以下检查：

### Step 1: 收集待提交内容

获取所有待提交的变更（staged + unstaged + untracked），逐文件审查。

### Step 2: 逻辑正确性分析

对每个变更文件进行：
- 内部逻辑一致性检查（条件分支、数据流、状态转换）
- 接口契约检查（函数签名、参数类型、返回值与调用方是否匹配）
- 明显遗漏检查（缺少必要的错误处理、边界条件未覆盖）

### Step 3: 判定与反馈

- **通过** → 正常执行提交
- **发现问题** → 拒绝提交，向用户报告具体问题及修复建议，待用户修复后重新触发本检查点

---

## 检查点 6：BEFORE 使用 category

当经验属于非开发领域（非编程语言/框架）时，**MUST** 执行以下流程确认分类名：

### Step 1: 列出已有分类

调用 `porter_status()` 或扫描 `data/lessons/` 目录，获取所有已有分类目录。向用户展示：

> "当前已有以下经验分类：[列出非语言类目录]。本条经验属于哪个分类？或输入新分类名。"

### Step 2: 用户确认

- 用户选择已有分类 → 直接使用
- 用户指定新分类名 → 使用用户提供的名称（小写、无空格）
- **禁止** AI 自行编造分类名

### Step 3: 记忆分类

用户确认后，当前会话内同一领域的后续经验直接复用该分类名，无需重复询问。

## 项目规则体系

> 编码/修改代码前，MUST 先阅读对应的规则文件。
>
> **强制总规则**：任何任务开始前，MUST 读取并严格执行 `/Users/liulantian/self/ai设计规范/rules/` 目录下全部规则文件（`*.md`）；执行过程中不得偏离。若 `rules/` 新增或更新文件，MUST 重新读取后再继续。

| 规则文件 | 触发条件 |
|---------|---------|
| `PROJECT.md` | 首次接触项目时阅读，获取全局上下文 |
| `rules/frontend.md` | 修改 `web-admin/frontend/` 下任何文件前 |
| `rules/backend.md` | 修改 `web-admin/api/` 或 `mcp-*/` 下任何 .py 文件前 |
| `rules/ui-design.md` | 涉及样式、布局、新增组件、视觉调整时 |
| `rules/mcp-service.md` | 新增或修改 MCP 服务时 |
| `rules/architecture.md` | 跨层、跨服务变更时 |

### 智能体定义

| 智能体 | 文件 | 适用场景 |
|--------|------|---------|
| 前端专家 | `agents/frontend-expert.md` | Vue 3 + Element Plus 开发 |
| 后端专家 | `agents/backend-expert.md` | Python FastAPI + FastMCP 开发 |
| MCP 架构师 | `agents/mcp-architect.md` | 服务设计与跨服务架构决策 |
| 安全审计员 | `agents/security-auditor.md` | 安全审查与合规检查 |
