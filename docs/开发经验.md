# 开发经验

## /ai/chat 项目切换与多轮对话状态问题复盘

日期：2026-04-25

### 问题现象

- 进入 `/ai/chat?project_id=...` 后，主内容区能显示项目名，但侧边栏项目下拉仍显示“未选择”。
- 切换项目后，项目数据没有完整更新，最近对话列表和历史消息可能为空或保留旧状态。
- 对话发送一次后，第二次发送按钮或输入框状态异常，表现为“对话框没有变化 / 发送按钮状态没变化 / 不能第二次对话”。

### 根因总结

1. **初始化只同步了项目 ID，没有强制水合会话数据**
   - `syncProjectFromRoute()` 会把 route/localStorage 中的项目 ID 写入 `selectedProjectId`。
   - 但初始进入页面时，如果依赖 `watch(selectedProjectId)` 去加载 providers、sessions、history，可能出现 watcher 未覆盖初始化赋值或同值不触发的情况。
   - 结果是：顶部项目名来自 `selectedProjectId` 能显示，但侧栏最近对话、历史消息、任务树没有完整加载。

2. **切换项目时复用了旧项目的会话上下文**
   - 原逻辑里项目切换只是改 `selectedProjectId`，URL query 里的旧 `chat_session_id` / `message_id` 可能还在。
   - 新项目加载过程中如果拿旧项目的 `currentChatSessionId` 去请求 task-tree/history，容易触发 403/404 或空数据。
   - 外层错误处理又可能清空 `messages`，于是用户看到“历史对话内容没了”。

3. **UI 下拉组件存在旧触发器文本残留**
   - Element Plus 的 dropdown trigger 在某些嵌入式/desktop shell 场景下可能保留旧渲染结果。
   - 即使 Vue 状态里 `selectedProjectId` 已经是正确项目，侧栏下拉仍可能显示“未选择”。

4. **运行时快照恢复函数缺失会中断历史恢复**
   - 页面里调用了 `schedulePersistChatRuntime()`、`applyPersistedChatRuntimeRows()`、`restoreInteractiveChatRuntime()`。
   - 如果这些函数缺失，会在 watcher 或历史恢复链路里抛 `ReferenceError`，阻断后续 UI 状态更新。
   - 这种错误会间接导致发送状态、历史消息、终端状态等恢复不完整。

5. **“不能第二次对话”通常是状态未收尾的外在表现**
   - 发送按钮依赖 `chatLoading`、`pendingRequests`、`currentChatSessionId`、输入框内容等状态。
   - 当项目/会话错配或运行时错误打断收尾逻辑时，`chatLoading` 可能没有及时回到可发送状态。
   - 修复项目水合、会话隔离和 runtime 恢复后，多轮发送状态恢复正常。

### 本次修复策略

- **切换项目先清理旧上下文**
  - 清空旧 `currentChatSessionId`、`messages`、历史计数、任务树和终端面板状态。
  - 取消旧 pending request 和 WebSocket，避免旧请求继续写入新项目 UI。

- **项目切换同步 URL query**
  - 选择新项目时写入 `project_id`。
  - 同时删除旧的 `chat_session_id` 和 `message_id`，避免跨项目复用旧会话。

- **初始化后显式水合项目会话**
  - `syncProjectFromRoute()` 返回解析到的初始项目 ID。
  - `onMounted()` 在项目列表、模型配置、全局 provider 加载后，主动调用统一的项目会话加载函数。

- **统一项目会话加载入口**
  - 通过 `loadSelectedProjectConversation(projectId)` 顺序加载：providers → ongoing task → sessions → history → task-tree。
  - 每个异步阶段后检查当前项目是否仍然匹配，避免快速切换时旧请求覆盖新项目状态。

- **下拉组件按项目强制重渲染**
  - 给项目 dropdown 增加 `:key="selectedProjectId || 'no-project-selected'"`，避免 trigger 文本残留。

- **补齐 runtime 快照函数**
  - 增加消息/终端运行态快照构建、合并、恢复和延迟持久化。
  - 历史消息加载时合并 runtime payload，恢复终端输出、操作卡片、状态备注等交互态。

### 回归验证清单

每次改 `/ai/chat` 的项目切换、会话、历史或发送状态，至少验证：

1. **直达 URL 初始化**
   - 打开 `/ai/chat?project_id=<A>`。
   - 侧栏项目名、顶部项目名、最近对话、历史消息都应属于项目 A。
   - Network 应出现项目 A 的 `/chat/providers`、`/chat/sessions`、`/chat/history`、`/chat/task-tree`。

2. **项目切换**
   - 从项目 A 切到项目 B。
   - URL query 里的 `project_id` 应变为 B。
   - 旧的 `chat_session_id` / `message_id` 不应继续挂在新项目 URL 上。
   - 侧栏项目名、最近对话、历史消息应全部切到 B。

3. **带旧会话 query 的切换**
   - 从带 `chat_session_id=<A-session>` 的 URL 切到项目 B。
   - 不应拿 A 的 session 请求 B 的 history/task-tree。
   - 不应因为 403/404 清空 B 的正常历史。

4. **多轮发送**
   - 发送第一条消息时，输入框/按钮进入处理中状态。
   - 回答完成或失败收口后，输入框恢复可输入。
   - 再输入第二条消息，发送按钮应可启用并能正常发送。

5. **Console 检查**
   - 不应出现 `schedulePersistChatRuntime is not defined` 等 runtime 恢复相关错误。
   - watcher 中的错误不能被忽略，因为它们可能会阻断历史恢复和按钮状态收尾。

### 后续开发原则

- 不要只改 `selectedProjectId`，项目上下文切换必须同时处理 route、session、history、task-tree、WebSocket 和 pending request。
- 不要在新项目 session 未解析前，用旧 `currentChatSessionId` 请求新项目接口。
- 不要完全依赖 watcher 完成初始数据加载；URL 初始化场景应有显式水合入口。
- 对嵌入式页面或 desktop shell 场景，遇到状态正确但 UI 文本旧的情况，优先考虑组件缓存/trigger 重渲染问题。
- 修改历史恢复或发送状态时，必须同时做浏览器实测，不要只依赖构建通过。

## /ai/chat 群上下文对话设计经验

日期：2026-04-25

### 目标

- 新建 AI 对话时，允许用户只填写平台和群名称，系统自动补齐可推断的上下文字段。
- 不同平台、不同机器人、不同群、不同主题必须用结构化字段隔离，避免历史对话和群消息串线。

### 字段分层

- **会话级字段**：`platform`、`connector_id`、`external_chat_id`、`external_chat_name`、`source_type`、`project_id`、`chat_session_id`、`thread_key`，用于区分这个 AI 对话绑定的是哪个外部群或主题。
- **消息级字段**：除继承会话级字段外，真实群消息才补 `external_message_id`、`sender_id`、`sender_name`。
- **source_type 约定**：手动在 AI 页面创建的是 `manual_ai_chat`；机器人收到群消息是 `group_message`；私聊是 `private_message`；针对群做离线分析可用 `group_analysis`。

### 实现原则

- 只有在用户提供了平台、群名、群 ID、连接器或真实消息字段时，才写入群上下文；普通新对话不能被误标为 `manual_ai_chat`。
- `connector_id` 可以根据 `platform + project_id` 从机器人连接器配置自动匹配；如果没有匹配到，先保留群名和平台，后续由平台事件或 API 补齐。
- `external_chat_id` 不应仅凭群名伪造；群名不是稳定 ID。没有平台稳定 ID 时，`thread_key` 使用平台、连接器和群名 hash 生成临时隔离键。
- AI Prompt 中要显式注入当前群上下文，用户说“这个群/群里/该群上下文”时默认指向当前 `thread_key`，不要混入其他群。

### 回归清单

1. 普通“新对话”不应带任何 `source_context`。
2. “群对话”创建后，会话列表和顶部 meta 应显示平台、群名和“待解析 ID/已解析”。
3. 发送消息时，请求 payload 应携带当前会话的 `source_context`，后端消息记录应继承会话级上下文。
4. 飞书机器人真实消息应写入 `group_message/private_message`、`external_chat_id`、`external_message_id`、`sender_id` 和 `thread_key`。
5. 一旦后续平台 API 能解析稳定群 ID，应更新会话的 `external_chat_id` 和 `thread_key`，不要只按群名匹配。
