# AI 对话中心类 macOS 交互升级方案

> 版本：v1.0  
> 日期：2026-04-21  
> 状态：方案设计完成，待进入开发  
> 目标页面：[web-admin/frontend/src/views/projects/ProjectChat.vue](/Volumes/苹果1_5T/self/ai-employee/web-admin/frontend/src/views/projects/ProjectChat.vue)

## 1. 背景

当前 AI 对话中心已经具备较完整的信息密度和功能入口，但主交互仍然是“左侧列表 + 中间工作区 + 顶部操作条”的 Web 工作台范式，而不是“像操作系统一样管理多个小应用”的范式。

现状入口主要集中在以下结构：

- 会话侧栏：项目切换、新对话、最近对话
- 主工作区：上下文条、恢复 banner、消息区、输入区
- 浮层/辅助面板：任务树、MCP 接入、技能资源、设置中心、项目详情、素材库

对应现有代码区块：

- 左侧会话栏：[ProjectChat.vue](/Volumes/苹果1_5T/self/ai-employee/web-admin/frontend/src/views/projects/ProjectChat.vue)
- 上下文条与主操作区：[ProjectChat.vue](/Volumes/苹果1_5T/self/ai-employee/web-admin/frontend/src/views/projects/ProjectChat.vue)
- 消息区与输入区：[ProjectChat.vue](/Volumes/苹果1_5T/self/ai-employee/web-admin/frontend/src/views/projects/ProjectChat.vue)
- 设置中心路由容器：[index.js](/Volumes/苹果1_5T/self/ai-employee/web-admin/frontend/src/router/index.js)

这次升级不只是换视觉皮肤，而是把“页面式切换”升级为“桌面式工作环境”。

## 2. 需求清单

- [ ] #1 将 AI 对话中心重定义为“AI Workspace Desktop”，形成类 macOS / PC OS 的交互模型
- [ ] #2 将现有功能入口重组为“Dock + 小应用窗口”，而不是顶部零散按钮
- [ ] #3 将消息区升级为主应用窗口，支持更强的系统感、专注感和任务连续性
- [ ] #4 将项目详情、素材库、MCP、技能资源、设置中心设计为可切换的小应用
- [ ] #5 输出可直接指导后续开发的文档：信息架构、交互流、视觉规则、分阶段实施、验收标准

## 3. 设计目标

基于项目 UI 规则 `rule-134beefd`，这次升级必须同时满足：

- 保留 `/intro` 的浅色空气感、轻玻璃、低噪音系统感
- 强化“AI Operating System”而不是传统后台页面
- 菜单不再只是导航，而是应用启动器
- 每个核心能力都像一个独立小应用，但运行在同一个项目桌面里
- 项目上下文、任务树、会话恢复、消息流必须保持连续，不能被页面切换打断

一句话定义：

把当前 `/ai/chat` 做成“AI 工作桌面”，而不是“聊天页面”。

## 4. 现状问题

### 4.1 交互模型问题

- 左侧会话栏承担了导航、项目切换、历史管理三种职责，语义混杂
- 顶部 `项目详情 / 素材库 / MCP 接入 / 技能资源` 是动作按钮，不像应用入口
- 设置中心已经有较强的“独立空间”特征，但与聊天主界面仍然割裂
- 任务树、恢复提示、审计、消息、输入都堆在同一纵向流里，缺少桌面式层级

### 4.2 感知问题

- 用户进入后感知仍然是“一个聊天页”，不是“一个 AI 操作系统”
- 项目上下文、会话上下文、任务上下文分散在不同模块
- 现有空状态能表达能力，但缺少“应用桌面已就绪”的启动感

### 4.3 扩展问题

- 后续如果继续增加更多入口，顶部按钮会越来越像工具栏堆砌
- 设置中心路由继续扩展后，主聊天和设置中心会形成两套交互语言
- 项目详情、素材库、技能资源等模块无法自然形成统一工作流

## 5. 核心方案

## 5.1 总体交互模型

将现有 AI 对话中心升级为三层结构：

1. `Desktop`
   承载整个项目工作环境、背景氛围、当前项目上下文。
2. `Dock`
   固定在底部或侧下方，作为应用启动器。
3. `App Windows`
   每个核心入口都是一个“窗口化小应用”，但默认只保持单主窗口聚焦，避免真多窗混乱。

### 目标映射

- `AI 对话`：主窗口，默认聚焦应用
- `项目详情`：项目信息应用
- `素材库`：资产应用
- `任务树`：任务推进应用
- `MCP`：连接与运行时应用
- `技能资源`：能力目录应用
- `设置中心`：系统设置应用

## 5.2 信息架构重组

### 现有结构

- 左侧：项目 + 会话 + 用户
- 中部：上下文条 + 恢复 banner + 消息
- 底部：输入区
- 顶部按钮：多个跳转/弹窗入口

### 新结构

- 顶部保留轻量系统栏
  - 左：当前项目、工作状态、时间/在线状态
  - 中：当前聚焦应用名称
  - 右：搜索、通知、账号、快捷命令入口
- 中部为桌面工作区
  - 主窗口默认是 AI 对话应用
  - 次级应用按“切换视图”进入，不建议第一期就做自由拖拽真窗口
- 底部为 Dock
  - 显示 5 到 7 个核心应用图标
  - 当前应用高亮
  - hover 显示应用名和状态点

## 5.3 应用定义

### A. AI 对话应用

替代当前整页聊天视图，成为桌面主窗口。

窗口内部拆成四层：

- `window-header`
  显示项目名、模式、模型、会话状态、恢复状态
- `window-sidebar`
  左侧保留“最近会话”与项目范围切换，但弱化为窗口内部导航
- `window-canvas`
  消息流、任务推进提示、执行审计、AI 输出
- `window-composer`
  输入区、附件、快速动作、模式切换

关键变化：

- 顶部 `chat-context-bar` 改造成窗口标题栏，而不是独立大横幅
- “项目详情 / 素材库 / MCP / 技能资源”从按钮改成 Dock app，不继续堆在标题栏
- 恢复 banner 升级为系统通知条，可折叠进入窗口标题区

### B. 项目详情应用

由当前 `openCurrentProjectDetail()` 触达的详情页面，改造为桌面中的“项目总览应用”。

承担内容：

- 项目基础信息
- 当前项目成员
- 当前项目规则摘要
- 最近任务与最近变更

定位：

- 用于“看项目状态”
- 不承载聊天输入
- 与 AI 对话应用形成主从关系

### C. 素材库应用

由当前 `openCurrentMaterialLibrary()` 触达。

定位：

- 独立资产工作台
- 支持从 Dock 直接进入
- 后续可和短片、声音、作品等子路由形成一个应用内 tab

### D. 任务树应用

由当前 `openTaskTreePanel()` 触达的弹层，升级为独立小应用。

原因：

- 任务树是持续运行中的“任务管理器”
- 不应只是消息流中的附属面板
- 更接近操作系统里的“活动监视器 / 工作流控制台”

### E. MCP 应用

由当前 `openUnifiedMcpDialog()` 触达。

定位：

- 连接管理
- 当前会话绑定状态
- tool 调用健康度
- 连接器、工作区、风险提示

### F. 技能资源应用

由当前 `openSkillResourceCenter()` 触达。

定位：

- 技能目录
- 技能说明
- 与当前任务的适配建议

### G. 设置中心应用

当前设置中心已经接近独立应用，应保留并统一进应用系统。

建议：

- `/ai/chat/settings/*` 不再被感知为“另一个页面”
- 在视觉与交互上被定义为桌面中的“System Settings”

## 6. 关键交互设计

## 6.1 Dock 交互

Dock 是这次升级的核心，不只是导航。

规则：

- 固定 5 到 7 个核心应用
- 图标统一圆角方形，不做拟物
- hover 显示应用名、最近状态、未读提示
- click 切换当前聚焦应用
- 当前应用有轻量高亮底座

建议应用顺序：

1. AI 对话
2. 项目详情
3. 素材库
4. 任务树
5. MCP
6. 技能资源
7. 设置中心

第一期不做：

- 应用拖拽排序
- 真正多窗口重叠
- 桌面文件系统

## 6.2 窗口交互

窗口采用“类窗口”而不是“浏览器弹窗”。

窗口头部包含：

- 左侧三点式状态灯
  只做语义装饰，不承载关闭浏览器行为
- 中间标题
  当前应用名称 + 当前项目
- 右侧快捷动作
  搜索、刷新、切换会话、打开命令面板

窗口行为：

- 第一阶段以单窗口主切换为主
- 次级面板可用抽屉、分栏、半屏 overlay 承载
- 第二阶段再考虑双栏并行，例如“对话 + 任务树”

## 6.3 Command Palette

新增全局命令面板，承担系统级快速导航。

触发方式：

- `Cmd/Ctrl + K`
- 顶部系统栏搜索入口

支持动作：

- 切换应用
- 切换项目
- 打开指定会话
- 打开任务树
- 打开 MCP / 技能资源 / 设置
- 新建对话

这样可以降低用户对左侧固定导航的依赖。

## 6.4 会话与任务连续性

这是桌面化最重要的非视觉部分。

必须保留：

- 当前项目不丢失
- 当前 chat_session_id / work_session 状态可感知
- 恢复中的任务有明确系统提示
- 任务树应用与 AI 对话应用可以互跳

具体要求：

- 在系统栏显示当前会话状态和任务状态
- 恢复通知不要只出现在消息区顶部，应同时进入系统通知中心
- 当任务树校验出现异常时，Dock 上“任务树应用”显示警示点

## 7. 视觉方向

基于 `rule-134beefd`，本次视觉采用：

- 浅色桌面背景
- 稀薄 mesh 光场
- 窗口式玻璃容器
- 冷静的蓝青高光
- 低噪音排版

### 7.1 Desktop 背景

延续当前 `chat-layout__ambient` 和 `chat-layout__mesh` 方向，但加强“桌面”而不是“整页背景”的感受。

建议：

- 背景更平整
- 环境光更靠边缘
- 中间工作区对比度更稳定

### 7.2 Window 材质

- 圆角：`28px ~ 34px`
- 背景：`rgba(255,255,255,0.76) ~ 0.84`
- 边框：细白边 + 轻浅灰
- 阴影：弱，避免厚重

### 7.3 Dock 样式

- 悬浮胶囊底座
- 半透明浅底
- 图标统一网格尺寸
- hover 有轻微上浮，不做夸张放大

### 7.4 系统栏

- 高度控制更薄
- 像系统状态栏，而不是网页导航栏
- 文字小，信息准

## 8. 页面级改造映射

## 8.1 `chat-layout`

当前作用：

- 页面根容器
- 背景氛围层

改造后：

- `workspace-desktop`
- 挂载系统栏、桌面区、Dock、通知层

## 8.2 `chat-shell`

当前作用：

- 左侧会话栏 + 右侧主区

改造后：

- `workspace-window-host`
- 负责渲染当前聚焦应用窗口

## 8.3 `chat-conversation-sidebar`

当前作用：

- 历史会话、项目选择、账号区

改造后：

- 降级为 AI 对话应用的内部导航栏
- 不再承担整个系统的全局导航

## 8.4 `chat-context-bar`

当前作用：

- 页面顶部说明 + 一排按钮

改造后：

- 并入 AI 对话应用窗口标题栏
- 只保留上下文、模型、状态
- 按钮型入口下沉为 Dock app 或命令面板动作

## 8.5 `chat-messages-shell` / `chat-messages`

改造目标：

- 成为窗口画布区
- 增强对消息、任务提示、审计块的层级组织
- 支持更明确的系统输出语义

建议：

- AI 消息保留大卡片流
- 用户消息更接近 command / request bubble
- 任务树审计、执行审计从普通消息附属块升级为“系统卡片”

## 8.6 `chat-composer`

改造目标：

- 更像操作系统的 command bar
- 强化输入框的“命令入口”感
- 把上传、模式、建议 prompt、发送动作压缩进一套统一底座

## 9. 分阶段实施计划

- 把项目详情、素材库、任务树、MCP、技能资源、设置中心纳入统一应用系统

## 10. 技术实施建议

## 10.1 组件拆分

建议从巨型单文件中拆出：

- `components/chat-workspace/WorkspaceDesktop.vue`
- `components/chat-workspace/WorkspaceSystemBar.vue`
- `components/chat-workspace/WorkspaceDock.vue`
- `components/chat-workspace/apps/ChatAppWindow.vue`
- `components/chat-workspace/apps/TaskTreeAppWindow.vue`
- `components/chat-workspace/apps/SettingsAppWindow.vue`

## 10.2 状态模型

建议新增：

```ts
type WorkspaceAppId =
  | "chat"
  | "project"
  | "materials"
  | "task-tree"
  | "mcp"
  | "skills"
  | "settings";
```

关键状态：

- `activeAppId`
- `appBadgeState`
- `workspaceNotifications`
- `appLaunchHistory`

## 10.3 路由策略

保持现有路由能力不变，但增加“应用层映射”：

- `/ai/chat` -> `chat app`
- `/ai/chat/settings/*` -> `settings app`
- `/projects/:id` -> `project app`
- `/materials/*` -> `materials app`

原则：

- 路由是底层寻址
- 应用是上层交互语义

## 11. 风险与取舍

## 风险 1：过度模仿桌面导致 Web 可用性下降

控制方式：

- 第一阶段只做“桌面感”和“应用感”
- 不做难以维护的真拖拽自由窗口

## 风险 2：现有功能入口迁移导致用户迷失

控制方式：

- Dock 和命令面板双入口并存
- 旧按钮短期可保留一版，但逐步下线

## 风险 3：`ProjectChat.vue` 过大，直接硬改风险高

控制方式：

- 先拆组件再改布局
- 不在第一阶段同时改消息逻辑和布局逻辑

## 风险 4：任务树与工作会话状态已经复杂

控制方式：

- 第一阶段只改展示与入口
- 不改底层 MCP / work session 绑定逻辑

## 12. 验收标准

- 用户进入 `/ai/chat` 后，第一感知必须是“AI 工作桌面”而不是“聊天页”
- 顶部零散按钮被收敛为 Dock 应用入口或命令面板动作
- AI 对话成为桌面里的主应用窗口
- 项目详情、素材库、任务树、MCP、技能资源、设置中心都能被理解为小应用
- 现有项目选择、会话切换、任务恢复、消息输入能力不回归
- 视觉结果符合 `rule-134beefd`：浅色、轻玻璃、低噪音、系统感
- 移动端和平板端可以自然退化，不出现桌面式布局硬塞到小屏的问题

## 13. 开发顺序建议

1. 先做骨架和 app registry
2. 再做 Dock 与系统栏
3. 再把 AI 对话窗口化
4. 再接入任务树 / 设置中心 / 项目详情 / 素材库
5. 最后补命令面板和通知中心

## 14. 本轮结论

本轮先完成文档，不进入开发。

后续真正开发时，建议把本方案拆成以下两个开发任务：

- 任务 A：AI 对话中心桌面化骨架重构
- 任务 B：Dock 应用系统与各业务入口接入

这样可以控制风险，并保持每一轮都有可验证结果。
