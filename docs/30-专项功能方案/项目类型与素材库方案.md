# 项目类型与素材库方案

> 目标：让同一个项目对话入口同时承载文字、图片、视频/分镜规划能力，并把真正需要沉淀的视觉资产收敛到项目素材库。

---

## 1. 背景

当前项目对话页已经承担项目主入口职责，但随着图片生成、视频生成、分镜规划需求进入同一工作流，原有模型需要补充两件事：

1. 项目需要一个更明确的“创作倾向”定义，避免每个项目都像同一种对话容器。
2. 产物沉淀需要区分“聊天过程”与“素材资产”，不能把所有文本都塞进素材库。

如果不做这层约束，会出现三个问题：

- 项目创建后缺少明确创作预期，用户需要每次重新说明自己是在做图片还是视频。
- 对话历史和素材资产混在一起，后续检索成本越来越高。
- 文字规划、临时讨论、提示词草稿被误当成正式资产沉淀，素材库会迅速失真。

---

## 2. 目标与边界

### 2.1 目标

- 为项目增加 `type` 字段，用于表达项目当前的主创作方向。
- 当前对话框继续支持文字、图片、视频/分镜规划三类能力。
- 项目类型影响默认界面重心、推荐动作、素材组织方式。
- 素材库只沉淀正式视觉资产。
- 普通文本回复只保留在聊天历史，不自动进入素材库。

### 2.2 非目标

- 不把项目类型做成绝对能力开关。
- 不为文字聊天单独建立素材库目录。
- 不在本期直接做完整视频剪辑工作台。

---

## 3. 核心定义

### 3.1 项目类型

建议首期支持以下类型：

| 类型值 | 名称 | 默认关注点 | 主要产物 |
|------|------|------------|----------|
| `image` | 图片项目 | 图片生成、改图、风格探索 | 图片素材 |
| `storyboard_video` | 分镜视频项目 | 分镜规划、镜头脚本、视频生成 | 分镜方案、视频素材 |
| `mixed` | 综合项目 | 同时存在图片与视频产出 | 图片 + 分镜方案/视频 |

说明：

- `type` 是项目默认工作流标记，不是能力白名单。
- 只要当前选择的模型具备能力，对话框仍可执行文字、图片、视频相关任务。
- 项目类型主要决定“打开项目时优先展示什么”和“推荐什么动作”，不直接决定最终能力上限。

### 3.2 聊天能力

当前对话框支持：

- 文字聊天
- 图片生成
- 视频生成
- 分镜规划

具体可用能力由“当前模型能力 + 当前项目类型”共同决定：

- 模型不支持某能力时，不展示对应快捷动作或直接禁用。
- 项目类型支持做默认引导，例如图片项目默认突出参考图、尺寸、风格；视频项目默认突出分镜、时长、节奏、镜头语言。

### 3.3 分镜的定义

本方案里的“分镜”不是最终视频文件，而是面向视频生成和后续编辑的镜头规划资产。

常见表现形式：

- 镜头列表
- 镜头脚本
- 时间轴规划
- 结构化 shot JSON

因此需要明确区分：

- `storyboard`：可编辑、可复用、可再次驱动视频生成的方案资产
- `video`：真正可播放的最终媒体文件

### 3.4 素材库范围

素材库只保留正式视觉资产：

- 图片素材
- 分镜方案
- 视频素材

默认不自动进入素材库的内容：

- 普通问答文本
- 提示词草稿
- 讨论过程
- 临时分析结论
- 纯文字解释回复

这些内容仍然保留在聊天历史中，用于上下文追溯。

### 3.5 正式资产判定

不再用“是不是文字”判断是否入素材库，而改用“是否产生结构化资产”判断。

只有满足以下条件之一的结果，才自动进入素材库：

- 生成了图片文件
- 生成了视频文件
- 生成了结构化分镜对象

同时补充一个手动通道：

- 用户可把某条聊天结果手动提升为素材资产

这意味着：

- 普通 assistant 文本不自动入库
- 分镜只有在结构化输出时才自动入库
- 视频规划讨论默认只保留在聊天历史

## 4. 产品规则

### 4.1 项目类型影响什么

项目类型影响：

- 新建项目时的默认说明文案
- 对话空状态的推荐动作
- 输入区快捷能力顺序
- 素材页默认聚焦分组
- 项目详情页中的素材统计维度

项目类型不影响：

- 聊天主链路的统一性
- 文本聊天能力本身
- 模型支持能力的真实上限

### 4.2 统一对话入口

对用户来说，项目内仍然只有一个主对话入口：

```text
项目对话框
├── 文字聊天
├── 图片生成
├── 分镜规划
└── 视频生成
```

这样做的理由：

- 用户不需要先判断自己该进入哪个页面。
- 文本、图片、分镜规划、视频生成往往属于同一轮创作链路，拆页面会打断上下文。
- 模型能力切换比页面跳转更轻，适合探索型工作流。

### 4.3 素材沉淀规则

统一沉淀策略：

| 内容类型 | 是否进入素材库 | 归档方式 |
|---------|----------------|----------|
| 普通文字回复 | 否 | 保留在聊天记录 |
| 图片生成结果 | 是 | 写入项目素材库图片分组 |
| 分镜结构化结果 | 是 | 写入项目素材库分镜分组 |
| 视频生成结果 | 是 | 写入项目素材库视频分组 |
| 失败结果 / 中间推理 | 否 | 仅保留在会话上下文 |
| 用户手动收藏的中间结果 | 是 | 写入对应素材分组 |

当前聊天链路约定：

- 后端流式输出新增 `artifact` 事件；当工具返回图片资产时，事件体统一携带 `artifacts[]` 与 `images[]`
- `done` 事件会回带本轮已收集的图片结果，前端 assistant 消息据此自动回显图片
- 自动入库仅处理结构化图片资产，普通文本 `delta` / `done` 不自动入素材库
- 自动入库写入 `source_chat_session_id`、`source_message_id`、`source_username`，并在 `metadata` 中补 `artifact_source=project-chat-auto-artifact`
- assistant 聊天记录与素材库共用同一个 assistant `message_id`，素材库“查看来源”可稳定回跳到原消息

### 4.4 后续工作台策略

正式设计或剪辑阶段再打开独立工作页：

- 图片项目可进入图片设计/定稿页
- 视频项目可进入分镜编排/视频剪辑页

对话页负责“产出内容”和“形成资产”，工作页负责“编辑资产”和“完成交付”。

### 4.5 素材库页面定位

素材库本身也应是项目内独立页面，而不是对话页里的长期驻留大面板。

推荐信息架构：

```text
项目
├── 对话
├── 素材库
│   ├── 图片
│   └── 分镜 / 视频
└── 制作页（后续）
```

这样设计的原因：

- 对话页强调连续创作，不适合承担重资产管理操作。
- 素材库需要独立的浏览、筛选、分组、重命名、删除、引用能力。
- 文字聊天不进入素材库后，独立页面更容易保持资产边界清晰。

关键约束：

- 素材库是独立页面。
- 素材库不是独立生成入口。
- 正式生成仍优先在项目对话内完成，成功后再进入素材库沉淀与管理。

## 5. 页面结构调整建议

### 5.1 项目创建与编辑

新增 `项目类型` 字段：

- 支持选择现有类型
- 支持未来扩展新的类型枚举
- 默认值建议为 `mixed`

文案建议：

- 图片项目：更适合海报、KV、插画、商品图探索
- 分镜视频项目：更适合镜头脚本、分镜规划、短视频生成
- 综合项目：适合图文混合和前期创意探索

### 5.2 对话页

对话页保留统一输入区，但根据项目类型做差异化：

- 图片项目：
  - 默认推荐“生成图片”“参考图改写”“延展同风格”
  - 素材侧重点放图片
- 分镜视频项目：
  - 默认推荐“生成分镜”“拆镜头”“输出视频提示”
  - 素材侧重点放分镜和视频
- 综合项目：
  - 同时展示图片与视频能力入口

### 5.3 素材库

素材库建议固定两大分组：

```text
项目素材库
├── 图片
└── 分镜 / 视频
```

不再增加“文字素材”栏目。

并且素材库以项目独立页面承载，首期建议能力包括：

- 资产列表浏览
- 图片 / 分镜 / 视频分组切换
- 缩略图或卡片预览
- 关联来源消息查看
  - 支持从素材库直接打开项目聊天
  - 若已记录 `source_chat_session_id` / `source_message_id`，优先定位到原始会话和消息
- 删除、重命名、再次引用

## 6. 数据结构建议

### 6.1 Project

```json
{
  "id": "proj-xxxx",
  "name": "春季活动创意",
  "description": "品牌春季传播项目",
  "type": "storyboard_video"
}
```

### 6.2 MaterialAsset

```json
{
  "id": "asset-xxxx",
  "project_id": "proj-xxxx",
  "asset_type": "image",
  "group_type": "image",
  "source_message_id": "chat-xxxx",
  "source_chat_session_id": "chat-session-xxxx",
  "source_username": "alice",
  "created_by": "alice",
  "created_at": "2026-03-19T10:00:00Z",
  "updated_at": "2026-03-19T10:00:00Z",
  "title": "主视觉海报 v2",
  "preview_url": "...",
  "content_url": "...",
  "mime_type": "image/png",
  "status": "ready",
  "metadata": {
    "model": "xxx",
    "prompt": "..."
  }
}
```

`asset_type` 首期建议值：

- `image`
- `storyboard`
- `video`

`group_type` 首期建议值：

- `image`
- `storyboard_video`

说明：

- `asset_type` 用于数据层精确区分资产类型
- `group_type` 用于前端素材库一级分组展示
- 分镜类资产应支持 `structured_content`

## 7. 交互流程

### 7.1 图片项目

1. 用户进入图片项目
2. 对话页默认展示图片创作推荐动作
3. 用户发起文字描述或图片生成请求
4. 生成图片成功后，图片先在当前 assistant 消息内自动回显
5. 同步写入素材库图片分组，并记录来源会话/消息
6. 刷新聊天历史后，assistant 图片结果仍可回显
7. 文本解释与讨论仅保留在当前会话

### 7.2 分镜视频项目

1. 用户进入分镜视频项目
2. 对话页默认展示分镜生成、镜头拆解、视频提示词推荐动作
3. AI 输出结构化分镜或视频结果
4. 分镜方案或视频文件写入素材库分镜/视频分组
5. 镜头说明与策略讨论仅保留在聊天历史

### 7.3 手动入库

1. 用户在聊天历史中选中某条结果
2. 点击“加入素材库”
3. 系统弹出轻量确认表单，允许调整资产类型、标题、摘要、预览链接、结构化内容和元数据
4. 默认规则：
   - 带图片的消息默认按 `image` 预填
   - 纯文本结果默认按 `storyboard` 预填
   - 自动写入来源会话、来源消息、创建人
5. 用户确认后创建素材资产，并保留回溯字段用于后续跳转聊天上下文

## 8. 实施步骤

### Step 1：项目类型

1. 为项目模型增加 `type`
2. 为项目创建/更新接口增加 `type`
3. 历史项目未携带 `type` 时默认按 `mixed` 处理

### Step 2：资产模型

1. 新增 `MaterialAsset`
2. 为素材模型补充 `asset_type`、`group_type`
3. 增加来源追踪字段：`source_message_id`、`source_chat_session_id`、`source_username`
4. 分镜资产增加 `structured_content`

### Step 3：项目页

1. 项目列表新增类型展示
2. 项目创建/编辑弹窗新增类型字段
3. 项目详情页新增类型说明

### Step 4：对话页

1. 根据项目类型调整空状态与推荐动作
2. 根据模型能力控制文字/图片/视频动作显隐
3. 生成完成后由 `artifact payload` 驱动自动回显并自动入库
4. 支持手动将聊天结果加入素材库
5. 手动入库弹窗独立组件化，不继续堆叠在项目聊天页主体模板中
6. assistant 消息 ID 需要透传到后端，保证聊天记录、素材来源跳转和历史回放一致

### Step 5：素材库

1. 只保留图片、分镜/视频两个一级分组
2. 普通文本不自动沉淀
3. 支持从聊天消息快速打开对应素材
4. 支持删除、重命名、再次引用

---

## 9. 风险与约束

- 如果把项目类型误做成能力开关，会导致用户误以为图片项目不能做视频尝试。
- 如果把所有文字都当成“分镜”，素材检索会迅速退化成聊天记录检索。
- 如果分镜结果没有结构化归档，后续剪辑页无法稳定消费。
- 如果素材没有来源会话和来源消息，后续无法稳定回跳原始上下文。

因此本方案的关键约束是：

- 项目类型只控制默认工作流，不控制最终能力上限。
- 素材库只接收正式资产，不自动接收普通文本。
- 正式资产由 `artifact payload` 判定，而不是由“是不是文字”判定。
- 分镜是视频生成方案，不等于最终视频文件。

---

## 10. 结论

推荐采用“统一对话入口 + 类型化默认工作流 + 结构化资产入库”的方案：

- 一个项目只有一个主对话入口
- 对话同时承载文字、图片、视频/分镜规划能力
- 项目类型用于调整默认界面重心
- 图片、结构化分镜、视频结果进入素材库
- 普通文本只留在聊天记录
- 素材必须保留来源会话与来源消息

这样既保留创作流程连续性，也能保证素材库长期可维护。
