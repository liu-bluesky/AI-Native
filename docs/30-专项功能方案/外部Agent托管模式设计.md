# 外部 Agent 托管模式设计

> **版本**: v1.0  
> **日期**: 2026-03-07  
> **状态**: 方案确认 / 待实施

---

## 1. 背景

当前系统已经具备以下基础能力：

- 以项目为中心管理 `workspace_path`、成员、聊天配置。
- 在 AI 对话中按配置聚合项目技能与外部 MCP 工具。
- 对工具进行运行时过滤，例如 Shell 类工具、写文件类工具。
- 通过项目运行时统一调用内部技能工具与外部 MCP 工具。

但如果目标是获得类似 `Codex CLI`、`Claude Code` 这类开发型 Agent 的真实开发能力，仅依赖当前系统自带对话链路还不够。

核心原因：

- 当前系统更适合做 **对话层 / 控制层 / 权限层**。
- 外部开发型 Agent 更适合做 **推理决策层 / 执行层**。
- 与其在当前系统中重复实现一整套开发型 Agent，不如直接托管外部 Agent，并让其使用当前系统提供的项目上下文、规则、技能与审批能力。

---

## 2. 目标

将当前系统演进为统一入口，同时支持两种运行模式：

### 2.1 普通对话模式

继续沿用现有 `/ai/chat` 或项目聊天链路：

- 由当前系统自己调用配置好的 LLM Provider。
- 适合日常问答、项目咨询、工具辅助查询。

### 2.2 Agent 开发模式

由当前系统托管外部开发型 Agent：

- 前端仍使用当前系统的对话窗口。
- 后端不一定再调用本系统的 LLM Provider。
- 而是直接启动并托管本机已安装好的 Agent，例如：
  - `Codex CLI`
  - `Claude Code`
  - 未来其他支持 CLI / MCP / API 的开发型 Agent

在该模式下：

- 当前系统负责：项目、权限、规则、技能、审批、审计、会话管理。
- 外部 Agent 负责：理解需求、决策步骤、调用工具、开发实现、运行验证。

---

## 3. 核心设计结论

### 3.1 当前系统不必强制承担“AI 大脑”角色

未来产品形态不应限定为：

- “当前系统必须自己接 LLM API 才能工作”

更合理的定位是：

- “当前系统既可以自己对话，也可以托管外部开发型 Agent”

即：

- 当前系统是 **Agent 控制台**。
- 外部平台是 **真实执行者**。

### 3.2 不是所有平台都能直接接入

适合接入的对象：

- 本地 CLI 型 Agent
- 提供 MCP / API / SDK 的 Agent 服务

不适合直接接入的对象：

- 仅存在于网页产品内部、未开放控制接口的私有命令窗口

因此目标不应表述为“使用任意平台的工具”，而应表述为：

- **使用任意可被当前系统托管或可被当前系统调用的 Agent 平台能力**

### 3.3 当前系统只做控制，不做重复造轮子

当前系统的核心价值在于：

- 工作区绑定
- 权限控制
- 审批机制
- 项目规则
- 技能注册与暴露
- 会话审计
- 多项目统一入口

而不是重新实现一套与 `Codex` / `Claude Code` 等价的完整开发型 Agent。

## 4. 外部 Agent 如何使用当前系统能力

仅仅把外部 Agent 的命令窗口嵌入当前系统，并不意味着它天然知道当前系统有什么能力。

外部 Agent 默认通常只知道：

- 当前工作目录
- 自身默认能力
- 启动时注入的上下文文本

它并不知道当前系统内部存在：

- 项目规则
- 技能
- 智能体定义
- 项目审批策略
- 项目上下文查询能力

因此必须把这些能力主动暴露给外部 Agent。

### 4.1 规则与约束：通过上下文注入

适合通过以下形式告知：

- `AGENTS.md`
- 启动 Prompt / System Prompt
- 环境变量
- 工作区初始化文件

这类信息解决的是：

- 你是谁
- 当前项目是什么
- 需要遵守哪些规则
- 哪些路径可写
- 哪些操作需要确认

### 4.2 技能与工具：通过工具协议暴露

如果希望外部 Agent 真正使用当前系统能力，最佳方式不是“文字告诉它”，而是“把能力变成工具”。

推荐优先级：

1. **MCP Server**
2. 本地 CLI Wrapper
3. HTTP API + Agent 适配层

其中最推荐的是 MCP 化，因为其具备：

- 可发现
- 可枚举
- 可调用
- 便于跨 Agent 复用

### 4.3 权限与审批：必须后端硬控制

安全能力不能只靠模型自觉遵守。

必须由当前系统后端硬性控制：

- 工作区范围
- 命令白名单 / 黑名单
- 写文件限制
- 高风险操作二次确认
- 全量审计日志

## 6. 模式划分

### 6.1 普通对话模式

适用场景：

- 查询项目状态
- 业务问答
- 规则说明
- 轻量工具辅助

特点：

- 继续复用当前已有 LLM Provider 体系。
- 继续复用当前项目聊天配置、工具过滤与 MCP 聚合能力。

### 6.2 Agent 开发模式

适用场景：

- 浏览真实目录结构
- 读取与修改项目文件
- 运行命令与测试
- 迭代开发与修复

特点：

- 当前系统只负责托管外部 Agent 会话。
- 外部 Agent 负责开发过程中的决策与执行。

## 7. 最小可用版本（MVP）

建议第一阶段只做 `Codex CLI` 接入，目标是尽快验证模式成立。

### 7.1 必做能力

- 选择项目并启动 Agent 会话
- 将工作目录切换到 `workspace_path`
- 前端聊天框向 PTY 会话发送输入
- 实时回显终端输出
- 支持停止会话
- 保存会话日志

### 7.2 暂缓能力

- 多 Agent 并发
- 复杂审批流
- 自动 diff 展示
- 大规模工具编排
- 跨机器远程托管

## 8. 外部 Agent 与当前系统技能的关系

### 8.1 结论

外部 Agent **可以**使用当前系统技能，但前提是这些技能需要被转换为 Agent 可接入的能力形式。

### 8.2 推荐方式

- 规则：通过 `AGENTS.md`、Prompt、环境变量注入
- 技能：通过 MCP 暴露
- 安全：通过后端强控

### 8.3 不推荐方式

- 只在对话里告诉外部 Agent “你可以使用某某技能”
- 只靠提示词约束文件改动范围或危险命令

原因：

- 不可发现
- 不可验证
- 不可审计
- 难以长期维护

## 9. 分阶段实施建议

### 阶段一：外部 Agent 会话托管

目标：先跑通 `当前系统 UI → PTY → Codex CLI`

交付物：

- Agent 会话后端接口
- PTY 会话管理器
- 前端 Agent 模式聊天窗口
- 项目工作目录注入

### 阶段二：上下文注入标准化

目标：让外部 Agent 明确知道项目背景与约束。

交付物：

- 启动上下文模板
- 项目规则注入策略
- `AGENTS.md` 生成或映射策略

### 阶段三：项目技能 MCP 化

目标：让外部 Agent 真正可调用当前系统能力。

交付物：

- 项目技能 MCP Server / MCP Bridge
- 项目规则查询工具
- 项目上下文查询工具
- 项目审批工具

### 阶段四：安全与审计增强

目标：形成生产可用能力。

交付物：

- 高风险命令审批
- 文件写入拦截
- diff 审查
- 操作审计日志

### 阶段五：多 Agent 适配

目标：在相同控制层下接入多个外部平台。

交付物：

- `Codex CLI Adapter`
- `Claude Code Adapter`
- 统一 AgentAdapter 抽象

## 10. 本次结论摘要

今天讨论后确认以下方向：

- 当前系统未来应同时支持“普通对话模式”和“Agent 开发模式”。
- 当前系统可以不必始终承担模型推理职责。
- 对于真实项目开发，更推荐托管 `Codex CLI` / `Claude Code` 等外部开发型 Agent。
- 当前系统应聚焦“控制层”价值，而不是重复造一个新的开发型 Agent。
- 若希望外部 Agent 使用当前系统技能，最佳做法是将技能 MCP 化并进行标准化暴露。
- 安全能力必须由当前系统后端硬控制，不能只依赖提示词约束。
