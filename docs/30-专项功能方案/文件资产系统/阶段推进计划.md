## 2. 阶段边界

### Phase A：冻结现状并补文档

目标：

- 明确当前已交付能力
- 明确哪些是预览级方案，哪些是正式方案
- 固定下一阶段实施顺序

完成标准：

- 本目录文档建立完成
- 当前能力与缺口可追溯

### Phase B：补齐导出预览链路的可用性

目标：

- 确认短片工作台导出结果稳定进入素材库
- 确认素材卡片能正确回显封面、标题、元数据
- 确认时间线可回用已导出视频素材

完成标准：

- 导出预览后可在素材库看到结果
- 导出预览后可重新加入时间线
- 前端构建与关键回归验证通过

### Phase C：设计正式导出任务模型

目标：

- 为正式成片导出建立独立任务模型
- 前后端约定任务状态与结果回写策略

建议新增实体：

- `studio_export_jobs`
- `studio_export_job_items`
- 或在现有项目域下新增导出任务存储

最少字段：

- `id`
- `project_id`
- `timeline_payload`
- `export_format`
- `export_resolution`
- `aspect_ratio`
- `status`
- `progress`
- `attempt_count`
- `result_asset_id`
- `error_message`
- `created_at`
- `updated_at`

完成标准：

- 接口契约定稿
- 存储模型定稿
- 状态机定稿

### Phase D：接入后端 `FFmpeg` 正式渲染

目标：

- 用后端渲染替换当前仅浏览器侧预览导出承担“正式导出”的角色
- 让导出分辨率、编码格式、音视频合成由服务端统一保证

主方案：

- 前端提交时间线和导出参数
- 后端创建导出任务
- Worker 调用 `FFmpeg` 执行渲染
- 渲染结果回写为项目视频素材
- 结果可进一步沉淀到作品页

不建议方案：

- 继续依赖纯前端浏览器渲染承担正式成片

原因：

- 编码兼容不稳定
- 长视频容易卡死
- 浏览器资源占用不可控
- 无法稳定支持任务重试和后台运行

完成标准：

- 后端可生成正式视频文件
- 支持至少 `720p` 和 `1080p`
- 结果自动回写素材库
- 首期允许使用“时间线占位渲染”承接任务闭环，再逐步替换为真实素材合成

### Phase E：导出任务状态与失败重试

目标：

- 在短片工作台中看到导出任务状态
- 失败后支持重试
- 渲染成功后能回跳素材或作品

最少状态：

- `queued`
- `processing`
- `succeeded`
- `failed`
- `canceled`

完成标准：

- 前端能查看任务状态
- 失败原因可见
- 至少支持一次重试

### Phase F：作品页沉淀与生命周期治理

目标：

- 导出成功的视频同时进入作品页
- 文件删除从“直接删”升级为“受控删除”

后续范围：

- 作品记录
- 回收站
- 恢复
- 延迟物理删除

## 3. 执行顺序

按以下顺序推进，不跳步：

1. `Phase A` 文档与边界冻结
2. `Phase B` 导出预览链路复核
3. `Phase C` 正式导出任务模型
4. `Phase D` 后端 `FFmpeg` 渲染
5. `Phase E` 状态、重试、回写
6. `Phase F` 作品页与生命周期治理
