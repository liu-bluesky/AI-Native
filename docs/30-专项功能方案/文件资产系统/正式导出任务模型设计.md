# 正式导出任务模型设计

## 1. 目标

为短片工作台建立正式导出任务模型，承接后续：

- 后端 `FFmpeg` 渲染
- 导出状态查询
- 失败重试
- 素材库结果回写
- 作品页沉淀

- [ProjectShortFilmStudio.vue](/Users/liulantian/self/ai设计规范/web-admin/frontend/src/views/projects/ProjectShortFilmStudio.vue) 已切到“提交正式导出任务”
- [ProjectWorksGallery.vue](/Users/liulantian/self/ai设计规范/web-admin/frontend/src/views/projects/ProjectWorksGallery.vue) 仍是占位页
- 当前仓库已落地 `POST /projects/:id/studio/exports`、`GET /projects/:id/studio/exports` 及后台 worker

## 2. 模型边界

正式导出任务不是素材本身，也不是作品本身。

三者关系：

1. `studio_export_job`
   - 负责描述“这次导出任务”
2. `project_material_asset`
   - 负责描述“导出后生成的视频素材”
3. `project_work`
   - 负责描述“对外可回看的交付结果”

首期可以先落：

- `studio_export_job`
- `project_material_asset` 回写

`project_work` 可以放到下一阶段补。

## 3. 状态机

建议首期状态：

- `queued`
- `processing`
- `succeeded`
- `failed`
- `canceled`

状态流转：

```text
queued -> processing -> succeeded
queued -> processing -> failed
queued -> canceled
processing -> canceled
failed -> queued   (retry)
```

约束：

- `succeeded` 之后不能直接重试，重试应创建新任务
- `failed` 可基于原参数创建重试任务
- `canceled` 只表示任务终止，不删除已生成中间文件

## 4. 数据结构

建议新增 `studio_export_jobs`。

### 4.1 主表字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 导出任务 ID |
| `project_id` | text | 所属项目 |
| `title` | text | 任务标题，默认来自剧本名或章节摘要 |
| `status` | text | `queued / processing / succeeded / failed / canceled` |
| `progress` | int | 0-100 |
| `export_format` | text | `mp4-h264` / `mp4-h265` |
| `export_resolution` | text | `720p` / `1080p` / `4K` |
| `aspect_ratio` | text | `16:9` / `9:16` / `1:1` 等 |
| `timeline_duration_seconds` | int | 时间线总时长 |
| `clip_count` | int | 片段数量 |
| `timeline_payload` | json | 导出用时间线快照 |
| `audio_payload` | json | 音轨快照 |
| `source_type` | text | 固定 `studio_export` |
| `result_asset_id` | text | 成功后写回的视频素材 ID |
| `result_work_id` | text | 成功后写回的作品 ID，可为空 |
| `cover_asset_id` | text | 若单独沉淀封面资产可记录 |
| `attempt_count` | int | 已尝试次数 |
| `error_code` | text | 错误码 |
| `error_message` | text | 错误信息 |
| `created_by` | text | 发起人 |
| `created_at` | text/timestamptz | 创建时间 |
| `updated_at` | text/timestamptz | 更新时间 |
| `started_at` | text/timestamptz | 开始渲染时间 |
| `finished_at` | text/timestamptz | 完成时间 |

### 4.2 timeline_payload 最小结构

```json
{
  "clips": [
    {
      "id": "clip-1",
      "sourceType": "storyboard",
      "sourceId": "storyboard-1",
      "title": "镜头一",
      "startSeconds": 0,
      "durationSeconds": 5,
      "endSeconds": 5
    }
  ],
  "audioTracks": [],
  "summary": {
    "timelineDurationSeconds": 12,
    "clipCount": 3
  }
}
```

要求：

- 存快照，不依赖渲染时再回查前端临时状态
- 允许后端按任务快照重放渲染

## 5. API 契约

### 5.1 创建正式导出任务

`POST /api/projects/{project_id}/studio/exports`

请求体建议：

```json
{
  "title": "春季短片正式导出",
  "export_format": "mp4-h264",
  "export_resolution": "1080p",
  "aspect_ratio": "16:9",
  "timeline_payload": {
    "clips": [],
    "audioTracks": [],
    "summary": {
      "timelineDurationSeconds": 18,
      "clipCount": 4
    }
  }
}
```

返回：

```json
{
  "status": "created",
  "job": {
    "id": "studio-export-001",
    "status": "queued",
    "progress": 0
  }
}
```

### 5.2 查询导出任务列表

`GET /api/projects/{project_id}/studio/exports`

支持参数：

- `status`
- `limit`

### 5.3 查询单个导出任务

`GET /api/projects/{project_id}/studio/exports/{job_id}`

用途：

- 轮询状态
- 获取失败原因
- 获取结果素材 ID

### 5.4 重试导出任务

`POST /api/projects/{project_id}/studio/exports/{job_id}/retry`

建议行为：

- 不直接复用原任务状态
- 基于原任务参数创建一个新任务
- 原任务保留为失败记录

### 5.5 取消导出任务

`POST /api/projects/{project_id}/studio/exports/{job_id}/cancel`

## 6. 成功回写策略

当任务 `succeeded` 后，后端需要：

1. 写入项目视频素材
2. 写入封面
3. 回填导出元数据
4. 更新任务结果字段

素材建议写入：

- `asset_type=video`
- `source_type=manual_upload` 不合适
- 建议新增或约定 `source_type=studio_export`

若暂时不改枚举，至少要在 `metadata` 补齐：

- `artifact_source=studio-export-final`
- `export_job_id`
- `requested_export_format`
- `requested_export_resolution`
- `aspect_ratio`
- `timeline_duration_seconds`
- `clip_count`

当前第一版已按此策略回写到项目素材库。

## 7. 前端接入方式

短片工作台正式导出建议改成两段：

1. `提交正式导出`
   - 创建任务
2. `查看导出状态`
   - 列表或侧栏展示任务进度

当前第一版已落地为：

1. 前端只创建正式导出任务
2. 后端 worker 自动拉取 `queued` 任务执行
3. 前端通过任务列表轮询查看结果

前端最少展示：

- 任务标题
- 状态
- 进度
- 分辨率
- 编码
- 发起时间
- 失败原因
- 成功后“查看素材”
- 成功后“查看作品”

## 8. 存储建议

考虑当前项目同时存在 JSON 和 PostgreSQL 两套 store，建议首期做法：

- 先在项目 store 体系下新增 `project_studio_export_store`
- JSON 和 PostgreSQL 两端都补对应实现
- 路由仍挂在 [projects.py](/Users/liulantian/self/ai设计规范/web-admin/api/routers/projects.py) 下，避免现在就拆成独立服务

当前阶段不建议：

- 单独拆独立导出 API 服务

原因：

- 当前项目规模还没到必须拆服务
- 先把任务模型和回写链路做稳定更重要

## 10. 实施顺序

1. 新增导出任务 store 和序列化
2. 新增创建、查询、取消、重试接口
3. 前端导出按钮切到“创建正式导出任务”
4. 后端接 `FFmpeg` worker
5. 成功后回写素材库
6. 把占位渲染升级为真实素材合成
7. 最后再接作品页

## 11. 当前结论

正式导出任务模型应作为“任务层”单独存在，不能继续复用当前前端预览导出直接上传素材的方式承担正式成片。

下一步代码实现时，应优先落：

- store
- API
- 状态机
- `FFmpeg` worker
- 素材回写

再把占位渲染替换为真实素材拼接。
