# 短片导出与正式渲染方案

## 1. 当前实现

当前短片工作台已经具备“导出预览并自动入素材库”的能力。

现阶段技术路线：

- 前端使用 `canvas`
- 通过 `captureStream()`
- 使用 `MediaRecorder`
- 生成短时长预览视频和封面
- 上传到项目素材库视频分组

这条链路的定位是：

- 先打通导出结果自动沉淀
- 让用户能立刻看到导出结果
- 为后端正式渲染留接口与交互位置

它不是最终正式成片方案。

## 2. 分辨率是怎么处理的

当前用户在导出面板选择：

- `720p`
- `1080p`
- `4K`

前端会结合：

- `aspectRatio`
- `resolution`

映射出一个用于预览导出的画布尺寸，再生成预览视频。

这意味着当前的“分辨率选择”已经生效，但生效范围是：

- 预览导出的画布尺寸
- 预览封面的画布尺寸
- 素材元数据中的目标导出配置

它还不是后端 `FFmpeg` 意义上的正式成片分辨率。

## 3. 为什么当前不用前端插件做正式渲染

不把正式视频渲染押在纯前端插件上的原因：

- 浏览器编码支持不稳定
- 不同设备和浏览器产物差异大
- 长视频、批量任务、后台运行都不可靠
- 无法很好承接失败重试和任务队列
- 分辨率、码率、音视频合成难以稳定保证

结论：

- 预览导出可以继续使用浏览器原生能力
- 正式导出必须转到后端渲染

## 4. 正式方案

正式成片导出采用后端 `FFmpeg`。

### 4.1 前端职责

- 提交时间线结构
- 提交导出格式与分辨率
- 轮询或订阅任务状态
- 展示成功、失败、重试入口

### 4.2 后端职责

- 创建导出任务
- 拉起渲染 worker
- 用 `FFmpeg` 完成画面与音轨合成
- 生成封面
- 回写项目素材库
- 视需要同步写入作品页

### 4.3 当前已落地的第一版

当前仓库已经补上第一版正式导出执行链路：

- API 启动时拉起后台导出 worker
- worker 轮询 `queued` 状态任务
- 使用 `FFmpeg` 生成正式导出视频与封面
- 成功后自动写入项目素材库
- 前端导出面板改为“提交正式导出任务”

这一版的定位是：

- 先把正式导出从“前端伪任务”切成“后端真实任务”
- 先打通刷新不丢、可重试、可回写素材的主链路
- 先让分辨率、编码、封面生成统一由服务端负责

当前还不是最终形态，但当前版本已经支持：

- 本地素材参与正式渲染
- `http(s)` / `data:` 素材临时下载后参与渲染
- 时间线片段级 `content_url / preview_url / storage_path` 直接参与正式渲染
- 导出阶段上传背景音乐并在正式导出时优先使用真实音频文件
- 分镜上传旁白音频并在正式导出时优先使用真实音频文件

兼容策略仍然保留：

- 若旧草稿没有真实旁白或背景音乐文件，后端仍会回退到程序化音频，避免正式导出直接失败

### 4.4 当前占位渲染方式

首版正式导出当前使用“本地素材优先 + 远端素材补充 + 占位回退”的 `FFmpeg` 渲染策略：

- 时间线片段若引用的是本地已上传图片或视频素材，优先使用真实文件生成片段
- 若时间线片段自身已经带有 `storage_path`、`content_url` 或 `preview_url`，后端也会优先使用这些真实来源
- 若素材没有本地文件，但 `content_url` / `preview_url` 是 `http(s)` 或 `data:`，会先下载/解码到临时目录再参与渲染
- 暂时无法定位到本地文件的片段，回退为占位色块片段
- 若时间线存在 `voice / bgm` 音轨，会优先使用真实音频文件混音；缺失真实文件时再回退为程序化音频
- 按导出分辨率和画幅输出正式视频文件
- 生成一张封面图
- 在素材 `metadata` 中标记 `artifact_source=studio-export-final`

这样做的意义是：

- 保留后端正式导出任务模型
- 保留真正的视频文件产物
- 已经开始消费真实本地素材文件
- 已经开始消费一部分 AI 生成/远端素材
- 已经把 `audio_payload` 和真实旁白/BGM 文件真正接入正式导出链路
- 为下一步切到真实音频文件合成和更完整时间线拼接留稳定接口

## 5. 正式渲染的建议输出

首期建议只做：

- `720p`
- `1080p`

编码优先级建议：

- 主输出：`MP4 (H.264)`
- 预留扩展：`MP4 (H.265)`

`4K` 不建议在第一版正式渲染就放开，原因是：

- 资源成本更高
- 渲染时间更长
- 容易把任务链路复杂度拉高

## 6. 后续实施顺序

1. 先固定导出任务模型
2. 再接 `FFmpeg` 渲染执行器
3. 再接状态、失败重试、素材回写
4. 再把占位渲染升级为真实素材拼接
5. 最后再决定是否进入作品页与 `4K`

## 7. 当前结论

当前短片导出分成两条链路：

- 预览导出：仍可保留浏览器预览级生成能力
- 正式导出：已经切到后端 `FFmpeg` 任务链路

下一阶段的关键不是换前端插件，而是继续把后端正式导出从“真实素材优先 + 回退兼容”升级到“完整真实素材合成”。
