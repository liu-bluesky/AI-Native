# 文件资产系统开发计划

> 目标：在当前项目素材库基础上，补齐“AI 生成结果自动沉淀 + 用户手动上传入库”的统一文件资产系统，并先完成计划文档，再进入开发实现。

---

## 1. 背景

当前项目已经具备项目素材库、项目聊天、图片自动入库的局部能力，但仍存在明显缺口：

- AI 自动入库目前只覆盖图片，未完整覆盖视频。
- 手动新增素材当前更接近“录入一条素材记录”，依赖手填地址，不是完整的文件上传链路。
- 素材模型以 `preview_url` / `content_url` 为中心，尚未形成统一的文件对象层。
- 删除、回收站、引用保护、视频后处理仍未形成闭环。

结合当前产品方向，文件资产系统必须同时满足两条链路：

1. AI 生成图片/视频后自动入库，默认保留，除非用户主动删除。
2. 用户可手动上传图片、视频、文档，并进入同一素材库统一管理。

---

## 2. 现状盘点

### 2.1 已有能力

- 项目素材库已支持 `image`、`storyboard`、`video` 三类资产。
- 素材库已有列表、筛选、手动新增、编辑、删除基础能力。
- 项目聊天已支持图片类资产自动沉淀，并记录来源会话、来源消息、来源用户。
- 素材库前端页面和后端路由已存在，可作为本轮迭代承接面。

### 2.2 关键缺口

- 自动沉淀逻辑当前写死为图片资产，视频生成结果未统一入库。
- 手动文件上传链路缺失，无法安全处理真实二进制文件。
- 当前素材模型缺少“文件对象”层，导致文件来源、大小、哈希、存储位置、删除状态难以统一管理。
- 视频封面、时长、分辨率、转码状态、可播放地址未标准化。
- 删除现在更接近直接删记录，缺少软删除、回收站和物理清理策略。

---

## 3. 目标与非目标

### 3.1 本期目标

- 建立统一文件资产模型，覆盖 AI 自动生成与手动上传两条来源。
- 保持项目素材库作为统一资产承接页，不再分裂出第二套文件管理入口。
- 支持图片、视频、文档三类文件对象；素材资产首期仍聚焦图片、分镜、视频。
- 实现默认保留、用户主动删除、软删除回收站的生命周期管理。
- 为后续视频封面、转码、作品导出、引用复用留出稳定扩展位。

### 3.2 本期非目标

- 不在本期做完整视频剪辑工作台。
- 不在本期做复杂的多级文件夹体系。
- 不在本期做公网分享、外链协作、公开相册。
- 不在本期做重度内容审核平台，只预留状态和任务扩展位。

## 4. 核心原则

### 4.1 统一资产，不拆两套系统

- AI 自动生成结果和手动上传文件必须进入同一套资产体系。
- 前端统一在项目素材库查看、筛选、编辑、删除和回跳来源。

### 4.2 文件与素材分层

- 文件层负责真实文件对象、存储位置、哈希、大小、生命周期。
- 素材层负责项目内展示、业务标签、来源回跳、资产分组。

### 4.3 AI 自动沉淀优先

- AI 生成结果自动入库是主链路。
- 手动上传是补充链路，用于导入参考图、已有视频、脚本文档等外部资料。

### 4.4 默认保留，删除受控

- AI 生成结果默认长期保留。
- 手动上传文件默认也保留，避免双重策略增加理解成本。
- 用户删除先进入回收站，再按策略执行物理清理。

### 4.5 大文件直传，处理异步化

- 手动上传文件不经 FastAPI 主服务中转。
- 视频封面、转码、元数据抽取走异步任务，不阻塞主请求。

### 4.6 新增素材必须通过上传，不走手填地址

- 项目素材库中的“新增素材”入口必须改为真实上传操作。
- 用户不再手动填写 `preview_url`、`content_url` 作为主录入方式。
- 地址字段仅用于系统内部回填、旧数据兼容或 AI 自动入库链路，不作为面向用户的主要交互。
- 对于本地参考图、本地视频、本地文档，统一通过上传或文件选择器进入系统。

## 5. 需求清单

### 5.1 业务需求

- `R1` AI 生成图片自动入库。
- `R2` AI 生成视频自动入库。
- `R3` 用户手动上传图片、视频、文档入库。
- `R4` 素材库“新增素材”不允许通过手填链接创建文件型素材。
- `R5` 素材支持来源追溯到项目、会话、消息、生成任务。
- `R6` 素材支持重命名、编辑摘要、查看状态、删除。
- `R7` 删除默认走软删除，支持回收站恢复。
- `R8` 视频支持封面、时长、分辨率、状态展示。
- `R9` 后续支持转码、审核、引用保护扩展。

### 5.2 用户侧场景

- 在项目聊天中生成图片后，系统自动沉淀到素材库图片分组。
- 在项目聊天中生成视频后，系统自动沉淀到素材库视频分组。
- 在素材库页面上传一张参考图，后续在项目聊天或工作台继续复用。
- 在素材库页面上传一个本地视频作为参考素材。
- 删除某个素材时，先进入回收站，避免误删。

## 6. 方案总览

建议采用“两层模型 + 两条入库链路 + 一套生命周期管理”。

### 6.1 两层模型

1. `file_objects`
   - 负责文件对象
   - 管理对象存储、文件大小、哈希、MIME、状态、删除标记
2. `project_material_assets`
   - 负责项目内资产展示
   - 管理 `asset_type`、`group_type`、标题、摘要、来源、预览、业务元数据

### 6.2 两条入库链路

1. AI 自动生成入库
   - 生成服务产出结果
   - 写入对象存储
   - 回调 API 创建文件对象
   - 自动创建素材记录
2. 手动上传入库
   - 前端发起上传会话
   - 直传对象存储
   - 完成上传后创建文件对象
   - 再创建素材记录

### 6.3 生命周期管理

- 新建：`uploading` / `processing` / `ready`
- 删除：`soft_deleted`
- 清理：延迟物理删除

## 7. 数据模型设计

### 7.1 file_objects

建议新增统一文件表：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | text | 文件 ID |
| `project_id` | text | 所属项目 |
| `source_type` | text | `ai_generated` / `manual_upload` |
| `source_task_id` | text | 生成任务 ID，可空 |
| `bucket` | text | 对象存储桶 |
| `object_key` | text | 对象键 |
| `original_filename` | text | 原始文件名 |
| `mime_type` | text | MIME 类型 |
| `size_bytes` | bigint | 文件大小 |
| `sha256` | text | 去重与校验 |
| `status` | text | `uploading` / `processing` / `ready` / `failed` / `soft_deleted` |
| `storage_provider` | text | `minio` / `s3` / `oss` 等 |
| `created_by` | text | 创建人 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |
| `deleted_at` | timestamptz | 软删除时间 |

### 7.2 file_variants

建议新增派生文件表：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | text | 派生文件 ID |
| `file_id` | text | 主文件 ID |
| `variant_type` | text | `thumbnail` / `poster` / `preview` / `transcoded_mp4` |
| `bucket` | text | 对象存储桶 |
| `object_key` | text | 对象键 |
| `mime_type` | text | MIME 类型 |
| `width` | int | 宽 |
| `height` | int | 高 |
| `duration_ms` | int | 时长 |
| `status` | text | 状态 |

### 7.3 project_material_assets

现有表继续保留，但建议扩字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `file_id` | text | 对应文件对象 ID |
| `source_type` | text | `ai_generated` / `manual_upload` / `manual_collect` |
| `source_task_id` | text | 来源生成任务 ID |
| `deleted_at` | timestamptz | 软删除时间 |

保留现有核心字段：

- `asset_type`
- `group_type`
- `title`
- `summary`
- `source_message_id`
- `source_chat_session_id`
- `source_username`
- `preview_url`
- `content_url`
- `mime_type`
- `status`
- `structured_content`
- `metadata`

### 7.4 upload_sessions

为手动上传链路预留会话表：

- `id`
- `project_id`
- `uploader_id`
- `upload_mode`
- `status`
- `target_mime_type`
- `target_size_bytes`
- `storage_provider`
- `bucket`
- `object_key`
- `expires_at`

### 7.5 media_jobs

为视频后处理预留任务表：

- `id`
- `project_id`
- `file_id`
- `job_type`
- `status`
- `attempt_count`
- `payload`
- `result`
- `error_message`

## 8. 存储与访问策略

### 8.1 存储策略

- 开发环境使用 MinIO。
- 生产环境使用 S3 / OSS / COS 任一对象存储。
- PostgreSQL 仅存元数据，不存二进制文件。

### 8.2 访问策略

- 默认私有存储。
- 素材预览和下载通过签名 URL 获取。
- 公开播放能力不在本期交付范围内。

### 8.3 删除策略

- 业务删除先改状态为 `soft_deleted`。
- 回收站恢复时恢复文件与素材关联。
- 定时任务对过期回收站数据做物理删除。

## 9. API 设计

### 9.1 AI 自动入库

- `POST /api/projects/{project_id}/materials/auto-ingest`
  - 用于 AI 生成结果回调入库
  - 支持 `image`、`video`、`storyboard`

请求重点字段：

- `asset_type`
- `title`
- `source_task_id`
- `source_message_id`
- `source_chat_session_id`
- `file` 或 `storage_ref`
- `metadata`

### 9.2 手动上传

- `POST /api/files/init-upload`
- `POST /api/files/complete-upload`
- `POST /api/projects/{project_id}/materials/from-file`

说明：

- 初始化上传返回预签名地址、会话 ID、对象键。
- 上传完成后创建 `file_object`。
- 再由素材接口将文件对象提升为素材资产。
- 面向用户的“新增素材”交互统一走这条链路，不再直接提交 `preview_url` / `content_url` 创建文件型素材。

### 9.3 素材管理

在现有素材接口基础上增强：

- `GET /api/projects/{project_id}/materials`
- `POST /api/projects/{project_id}/materials`
- `PATCH /api/projects/{project_id}/materials/{asset_id}`
- `DELETE /api/projects/{project_id}/materials/{asset_id}`
- `POST /api/projects/{project_id}/materials/{asset_id}/restore`

### 9.4 文件访问

- `GET /api/files/{file_id}`
- `GET /api/files/{file_id}/url`
- `GET /api/files/{file_id}/variants`

## 10. 前端页面与交互规划

### 10.1 项目聊天页

目标：

- 继续承接 AI 生成主链路。
- 生成成功后自动展示“已加入素材库”反馈。
- 视频生成中展示处理中状态，完成后可回跳素材库。

本期改造点：

- 图片自动沉淀逻辑泛化到图片/视频/分镜。
- 消息级素材状态补充图标、文案、失败重试提示。

### 10.2 项目素材库页

目标：

- 成为项目内统一资产页。
- 同时承接自动沉淀结果与手动上传结果。

本期改造点：

- 增加“上传文件”入口。
- 将当前“新增素材”表单从地址录入改为上传驱动。
- 支持图片、视频、文档卡片差异化展示。
- 支持状态筛选：处理中、可用、失败、回收站。
- 增加来源标签：AI 生成 / 手动上传 / 手动收藏。

### 10.3 上传交互

建议：

- 在素材库页提供主上传入口。
- 聊天页保留轻量附件上传，不承担完整文件管理职责。
- 视频上传展示进度条、大小限制、失败重试。
- 图片、视频、文档上传完成后，再补标题、摘要、分类等业务信息。
- 旧版 `preview_url` / `content_url` 输入框不再作为新增素材默认表单字段。

## 11. 开发拆分

### Phase 0：计划与契约确认

- 完成开发计划文档
- 确认字段、接口、存储方案
- 确认对象存储选型与环境变量

### Phase 1：统一文件模型落地

- 新增 `file_objects`
- 扩展 `project_material_assets`
- 完成存储层与序列化改造

- 将当前自动图片入库逻辑扩展到视频/分镜
- 新增 `source_type=ai_generated`
- 补充来源任务、状态、预览策略

### Phase 3：手动上传链路

- 新增上传初始化与完成接口
- 前端素材库页增加上传能力

- 新增视频封面抽帧
- 新增时长/分辨率探测
- 接入异步任务与状态回写

- 素材软删除
- 文件软删除
- 回收站恢复
- 延迟物理清理任务

## 12. 兼容与迁移策略

### 12.1 旧数据兼容

- 现有只包含 `preview_url` / `content_url` 的素材记录必须继续可读。
- 迁移前不强制补齐 `file_id`。
- 对旧素材采用“惰性补齐”策略：
  - 编辑或重新保存时逐步补入 `file_id`
  - 新生成/新上传数据全部走新模型

### 12.2 API 兼容

- 保持现有素材列表与编辑接口可用。
- 新能力尽量通过新增字段和新增接口实现，避免一次性破坏前端。
- 旧地址字段保留兼容读取，但文件型素材新增入口不再继续暴露纯地址创建模式。

## 13. 风险与边界

### 13.1 技术风险

- 视频自动入库如果仍依赖外部 URL，稳定性受第三方文件时效影响。
- 对象存储直传需要前后端同时调整，存在联调成本。
- 视频后处理引入异步任务后，状态一致性需要额外关注。

### 13.2 产品风险

- 如果“聊天附件上传”和“素材库文件上传”边界不清，用户会混淆两个入口。
- 如果不做回收站，误删成本过高。
- 如果文档类文件直接混入视觉素材默认看板，页面噪音会上升。
- 如果继续保留手填地址作为新增素材主入口，用户会误以为系统支持外链托管，和统一文件资产目标冲突。

### 13.3 约束边界

- 本期素材库一级分组仍保持 `图片` 与 `分镜/视频` 主结构不变。
- 文档文件进入统一文件层，但在素材页应以次级展示策略呈现，不抢主视觉层级。

## 14. 测试与验收

### 14.1 核心测试清单

- AI 生成图片后自动入库成功。
- AI 生成视频后自动入库成功。
- 手动上传图片成功并可预览。
- 手动上传视频成功并可看到处理中状态。
- 手动上传文档成功并可查看文件信息。
- 删除素材后进入回收站。
- 回收站恢复后素材重新可见。
- 素材来源回跳会话与消息成功。

### 14.2 验收标准

- 用户在单个项目内可以统一管理自动生成和手动上传的资产。
- 视频素材具备可识别的状态与基础元数据。
- 删除行为可恢复，不出现直接不可逆误删。
- 旧素材数据不因新模型改造而失效。

## 15. 发布与回滚

### 15.1 发布步骤

1. 先发布数据库变更与存储配置。
2. 再发布后端接口与存储层。
3. 最后发布前端素材库和聊天页适配。
4. 验证图片自动入库、视频自动入库、手动上传三条链路。

### 15.2 回滚策略

- 前端回滚到旧版后，现有素材列表与基本查看仍应可用。
- 后端新表保留，不做破坏性回退。
- 如对象存储链路异常，可临时关闭手动上传入口，仅保留已有素材展示。
