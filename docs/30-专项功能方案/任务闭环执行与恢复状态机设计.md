# 任务闭环执行与恢复状态机设计

> 日期：2026-04-15  
> 项目：`proj-d16591a6`  
> 适用范围：统一查询 MCP、项目任务树、工作轨迹、CLI/Chat 多轮续跑  
> 约束来源：`rule-ef629f1c`、`rule-00fc59e6`、`rule-e8dd737e`

## 1. 目标

把当前“提问题 -> 记问题 -> 出方案 -> 执行 -> 失败重试 -> 中断恢复 -> 完成收口”的口语化流程，收敛成一套可执行的工程模型。

本设计不新造一套独立系统，直接落到现有能力上：

- 任务树：负责目标拆解、节点推进、父子完成关系
- 工作轨迹：负责检查点、恢复、失败原因、下一步
- 项目记忆：负责沉淀稳定结论，不负责承载完整执行轨迹
- Query MCP：负责绑定、分析、规划、恢复、交付

一句话原则：

> 顶层目标先拆解，底层叶子先执行；每步必须验证，每次异常必须留痕，每次恢复必须先判定是否已经完成。

## 2. 当前结论

用户提出的原工作流方向是对的，但需要补四层约束：

1. 计划必须是“顶层拆解，叶子执行”，不是“从底层往上规划”
2. 状态不能只有完成/失败/中断，必须区分 `failed`、`blocked`、`interrupted`、`verifying`
3. 失败不能覆盖旧方案，必须保留旧尝试并派生新版本
4. 中断恢复前必须先做完成态校验，避免重复执行

## 3. 总体流程

```text
用户问题
-> 问题登记
-> 需求确认
-> 方案设计
-> 生成任务树
-> 执行叶子节点
-> 节点验证
-> 回写状态与检查点
-> 父节点收口
-> 顶层验收
-> 交付报告 / 归档
```

异常分支：

```text
执行失败 -> 记录失败原因 -> 基于当前节点派生 plan_v2 -> 重新执行
外部阻塞 -> 标记 blocked -> 等待依赖解除后恢复
人工/系统中断 -> 写 checkpoint -> 下次恢复前先检查是否已完成 -> 未完成则续跑
```

## 4. 核心对象模型

建议把一轮任务闭环拆成 6 个对象。

### 4.1 issue_record

表示问题登记，不等于任务树。

建议字段：

```json
{
  "issue_id": "iss-20260415-001",
  "project_id": "proj-d16591a6",
  "chat_session_id": "cli.proj-d16591a6.20260415T101446.liudeM.52459.r788nv",
  "session_id": "ws_proj-d16591a6_team_20260415T021512Z_1b1a",
  "raw_question": "用户原始问题",
  "summary": "一句话摘要",
  "keywords": ["任务闭环", "状态机", "恢复"],
  "source": "cli",
  "created_at": "2026-04-15T10:15:12+08:00"
}
```

用途：

- 对应 `rule-ef629f1c` 的需求登记
- 作为后续检索、恢复、归档的统一入口

### 4.2 solution_record

表示某一轮方案，不和执行记录混在一起。

建议字段：

```json
{
  "solution_id": "sol-20260415-001",
  "issue_id": "iss-20260415-001",
  "version": 1,
  "status": "active",
  "goal": "把口语化工作流整理成可执行模型",
  "scope": ["问题登记", "状态机", "恢复"],
  "non_goals": ["重做聊天 UI"],
  "acceptance": [
    "定义状态机",
    "定义对象模型",
    "定义恢复流程"
  ],
  "risks": [
    "若失败覆盖旧方案，将丢失历史决策"
  ],
  "created_at": "2026-04-15T10:18:00+08:00"
}
```

约束：

- `failed` 后不能覆盖原记录
- 重设计必须生成 `version + 1`

### 4.3 plan_version

表示与某个方案绑定的一版任务树拆解结果。

建议字段：

```json
{
  "plan_id": "plan-20260415-001",
  "issue_id": "iss-20260415-001",
  "solution_id": "sol-20260415-001",
  "version": 1,
  "planning_mode": "top_down",
  "execution_mode": "leaf_first",
  "root_goal": "完成闭环工作流设计",
  "status": "active",
  "task_tree_session_id": "tts-eb765ff5b6"
}
```

约束：

- 计划由顶层目标向下拆，不由底层实现向上拼
- 执行时始终优先最小可验证单元 `MVU`

### 4.4 task_node

这是执行主对象，直接对接现有任务树节点。

建议每个叶子节点至少补齐：

```json
{
  "node_id": "ttn-001",
  "title": "定义状态机",
  "objective": "产出状态流转和约束",
  "dependencies": [],
  "input": ["issue_record", "solution_record"],
  "output": ["状态机定义"],
  "done_definition": "文档已写入且人工核对通过",
  "verification_method": [
    "规则对照",
    "人工检查",
    "必要时测试"
  ],
  "evidence": [],
  "status": "pending"
}
```

### 4.5 execution_attempt

表示某个节点的一次执行尝试，解决“失败重试覆盖历史”的问题。

建议字段：

```json
{
  "attempt_id": "att-001",
  "node_id": "ttn-001",
  "attempt_no": 1,
  "status": "failed",
  "reason": "状态定义缺少 blocked/interrupted 区分",
  "changed_files": [
    "docs/30-专项功能方案/任务闭环执行与恢复状态机设计.md"
  ],
  "verification": [
    "人工复核发现状态语义不完整"
  ],
  "next_action": "派生 plan_v2 修订状态机"
}
```

约束：

- 同一节点允许多次 attempt
- `failed` 不等于节点永久失败
- 节点只有在某个 attempt 验证通过后才可 `done`

### 4.6 checkpoint_snapshot

表示中断恢复的最小现场。

建议字段：

```json
{
  "checkpoint_id": "ckp-001",
  "issue_id": "iss-20260415-001",
  "session_id": "ws_proj-d16591a6_team_20260415T021512Z_1b1a",
  "task_tree_session_id": "tts-eb765ff5b6",
  "current_node_id": "ttn-001",
  "phase": "implementation",
  "step": "design_workflow_spec",
  "status": "interrupted",
  "changed_files": [
    "docs/30-专项功能方案/任务闭环执行与恢复状态机设计.md"
  ],
  "verification": [],
  "risks": [],
  "next_steps": [
    "恢复后先检查该节点是否已完成，再决定是否继续写文档"
  ]
}
```

## 5. 状态机设计

### 5.1 任务级状态

推荐状态：

- `pending`
- `in_progress`
- `verifying`
- `done`
- `failed`
- `blocked`
- `interrupted`
- `canceled`
- `superseded`

含义：

- `failed`：本次尝试失败，需要重设计或重试
- `blocked`：外部依赖未满足，当前不应继续执行
- `interrupted`：人为暂停、CLI 中断、上下文切换，需要恢复
- `superseded`：旧计划或旧方案已被新版本替代

### 5.2 合法流转

```text
pending -> in_progress
in_progress -> verifying
verifying -> done

in_progress -> failed
verifying -> failed

in_progress -> blocked
blocked -> in_progress

in_progress -> interrupted
verifying -> interrupted
interrupted -> in_progress

pending -> canceled
in_progress -> canceled

failed -> superseded
done -> superseded   (仅当被更高版本替代，不代表原结果无效)
```

### 5.3 父子节点约束

- 子节点未全部 `done`，父节点不得 `done`
- 子节点存在 `failed / blocked / interrupted` 时，父节点只能 `in_progress`
- 父节点进入 `verifying` 前，必须有整体验证结论

## 6. 计划拆解规则

### 6.1 拆解方向

必须遵守：

```text
顶层目标
-> 一级目标节点
-> 二级可执行节点
-> 叶子 MVU 节点
```

禁止：

- 先按代码细节拆，再反推顶层目标
- 把 `search_ids`、`get_manual_content`、`generate_execution_plan` 这类内部工具名当任务节点

### 6.2 MVU 定义

MVU = 最小可验证单元。

判断标准：

- 能独立验证
- 能独立失败
- 能独立恢复
- 能独立产出证据

不以文件数量划分，而以验证边界划分。

### 6.3 每个叶子节点的固定子流程

```text
分析
-> 开发/修改
-> 自检
-> 测试/验证
-> 状态回写
-> 检查点
```

这一步直接对应 `rule-e8dd737e` 的“分片 -> 验证 -> 检查点”。

## 7. 失败、阻塞、中断的处理语义

### 7.1 failed

触发条件：

- 方案本身错误
- 实现结果不满足验收
- 验证不通过且短时间内无法直接修正

处理方式：

1. 记录 attempt
2. 写入失败原因
3. 判断是否可在原方案内修正
4. 若不可，派生 `solution_record.version + 1` 和 `plan_version.version + 1`

### 7.2 blocked

触发条件：

- 缺少权限
- 缺少外部依赖
- 缺少用户输入
- 环境或资源不可用

处理方式：

1. 不进入重设计
2. 保留当前计划
3. 记录解除阻塞条件
4. 依赖满足后继续原节点

### 7.3 interrupted

触发条件：

- CLI 中断
- 人工暂停
- 上下文压缩切断
- 切换到别的需求

处理方式：

1. 立即落 checkpoint
2. 记录当前节点、文件、验证、下一步
3. 下次恢复时先执行完成态校验

## 8. 恢复流程

恢复顺序必须固定：

```text
1. 恢复 chat_session_id / session_id
2. bind_project_context(...)
3. resume_work_session(...)
4. summarize_checkpoint(...)
5. 读取当前任务树
6. 检查当前节点是否已完成
7. 若任务已完成，直接补完成结论
8. 若未完成，回到中断节点继续
```

### 8.1 恢复前完成态校验

恢复时先检查三层：

1. 当前叶子节点是否已 `done`
2. 当前父节点是否已 `done`
3. 根任务是否已 `done`

处理规则：

- 根任务已完成：直接标记本次恢复请求为完成，不再执行
- 叶子节点已完成但父节点未完成：进入父节点收口
- 叶子节点未完成：继续执行该叶子节点

### 8.2 恢复摘要最小输出

恢复后至少输出：

- 已完成
- 当前节点
- 未完成原因
- 下一个动作
- 最近验证结果
- 最近修改文件

## 11. 执行门禁

对应 `rule-00fc59e6`，每一阶段都要有门禁。

### 11.1 需求门禁

未确认以下内容不得进入实现：

- 目标
- 范围
- 验收标准
- 非目标

### 11.2 实现门禁

未完成以下内容不得进入验证：

- 节点输出已形成
- 自检完成
- 影响面已记录

### 11.3 验证门禁

未形成以下内容不得标记 `done`：

- 验证方式
- 验证结果
- 证据

### 11.4 交付门禁

未完成以下内容不得关闭根任务：

- 完成项
- 未完成项
- 风险
- 回滚/补救说明

## 12. 推荐落地顺序

第一步：

- 先让文档/方案型任务也走完整状态机，而不是只写结论

第二步：

- 为任务树节点补 `attempt` 与 `checkpoint` 语义

第三步：

- 给前端任务树增加 `failed / blocked / interrupted / verifying` 可视状态

第四步：

- 增加失败派生新计划版本的后端能力

第五步：

- 把恢复前完成态校验做成统一逻辑，避免重复执行

## 13. 本轮建议定稿

最终推荐工作流：

```text
问题登记
-> 需求确认
-> 方案版本 v1
-> 顶层拆解任务树
-> 叶子节点执行
-> 每步验证 + 检查点
-> 子节点全部完成后父节点收口
-> 根任务完成后生成交付报告

若失败：
-> 保留失败 attempt
-> 视情况派生方案 v2 / 计划 v2
-> 重新执行

若中断：
-> 保存 checkpoint
-> 恢复时先检查是否已完成
-> 未完成则续跑当前节点
```

一句话定稿：

> 方案要版本化，执行要叶子化，验证要强制化，恢复要先判完成，再决定是否继续。
