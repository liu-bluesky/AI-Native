# 飞书机器人自动回复需求与测试用例

> 日期：2026-04-25  
> 关联问题：机器人自动回复功能应该结合“系统状态助手”还是“项目聊天 AI”  
> 结论：以“项目聊天 AI”为主入口，系统状态助手作为系统状态/页面检查类子能力。

## 1. 背景与目标

当前系统已有两类 AI 能力：

- **系统状态助手**：全局悬浮助手，面向当前页面、系统快照、接口请求、功能可用性和页面操作。
- **项目聊天 AI**：项目会话主线，面向项目需求、任务树、工作会话、员工协作、文档沉淀、代码/方案执行。

飞书机器人自动回复的核心诉求不是单纯“回答一句话”，而是把外部沟通转成系统内可追踪任务：识别需求、追问澄清、沉淀需求文档、生成测试用例，并按项目上下文继续推进。因此应优先接入项目聊天 AI，而不是直接接系统状态助手。

## 2. 推荐集成方案

### 2.1 总体结论

| 场景 | 推荐结合对象 | 原因 |
| --- | --- | --- |
| 用户在飞书里提需求、让 AI 整理 PRD、测试用例、开发计划 | 项目聊天 AI | 需要项目上下文、任务树、工作会话、文档产出和长期记忆 |
| 用户问“系统现在正常吗”“页面是否拿到数据”“接口有没有报错” | 系统状态助手 | 它已有页面快照、浏览器请求、DOM 操作等工具 |
| 用户问“这个系统怎么用”“哪个功能在哪里配置” | 系统状态助手优先，可回落项目聊天 AI | 系统状态助手内置 system guide 工具 |
| 用户说“帮我改代码/继续开发/生成需求文档” | 项目聊天 AI | 它能绑定项目、推进任务树、维护 requirement 和交付记录 |
| 飞书自动回复入口本身 | 独立 Lark Bot Connector | 飞书事件、鉴权、幂等、限流、安全审计应独立于两类 AI |

### 2.2 推荐架构

```text
飞书用户/群聊
  ↓ im.message.receive_v1
Lark Bot Connector
  ↓ 解析事件、去重、鉴权、路由
Message Router
  ↓ 通用 AI 工作流（读取机器人提示词 + 项目上下文 + 飞书来源上下文 + lark-cli 技能说明）
  ├─ 项目需求/文档/测试/开发 → Project Chat AI / 项目工具
  ├─ 飞书资源操作 → 通过 project_host_run_command 调用 lark-cli，并按 skills/lark-* 说明执行
  ├─ 系统状态/页面检查/接口问题 → Global Assistant 子能力
  └─ 无法判断/低置信度 → 澄清回复或人工确认
  ↓
Reply Dispatcher
  ↓ lark-cli im +messages-reply（按配置选择 bot 或 user 身份）
飞书自动回复
```

## 3. 边界与前置约束

### 3.1 飞书侧边界

- `lark-cli event +subscribe` 的事件订阅是 **bot-only**，需要飞书开放平台应用机器人能力。
- 自动回复应基于 `im.message.receive_v1` 消息事件，适合监听发给机器人、群内 @ 机器人或机器人所在群的消息。
- 这不是“监听别人发给真人账号的所有私聊”。如果要用真人身份回复，只能在发送环节使用 user identity，并需要额外授权，不建议作为 MVP。
- 回复发送统一通过 `lark-cli im +messages-reply`；默认使用 `--as bot`，需要以当前登录人身份回复时配置 `reply_identity=user`，并确保已完成 user 授权。
- 机器人必须具备接收消息和发送/回复消息所需权限；群聊场景下机器人需要在群内。

### 3.2 系统侧边界

- 不把飞书 Connector 直接写进系统状态助手组件；系统状态助手只保留“系统状态查询能力”。
- 不让飞书消息绕过项目聊天直接写代码或执行高风险命令；需要经过项目任务树、安全策略和权限确认。
- 自动回复需要幂等键，避免飞书重试或多实例监听导致重复回复。
- 自动回复默认只处理白名单群、白名单用户或明确 @ 机器人的消息。

## 4. 功能需求

### FR-01 飞书机器人连接配置

- 支持配置飞书应用凭据、事件订阅方式和机器人开关。
- 支持配置白名单群、白名单用户、是否仅处理 @ 机器人消息。
- 支持配置默认生成内容类型、生成模板、输出格式和保存位置。
- 支持配置机器人提示词和回复身份；机器人提示词驱动大模型理解与处理，回复身份决定最终 lark-cli 使用 `--as bot` 还是 `--as user`。
- 支持测试连接：校验 app、事件订阅、消息发送权限是否可用。

### FR-02 消息接收与标准化

- 接收 `im.message.receive_v1` 事件。
- 标准化字段：`event_id`、`message_id`、`chat_id`、`chat_type`、`sender_id`、`message_type`、`content`、`create_time`。
- 过滤机器人自己发送的消息，避免循环回复。
- 按 `event_id` 或 `message_id` 做幂等去重。

### FR-03 意图识别与路由

- 不再为每一种任务开发一套固定代码分支。长连接收到标准化消息后，统一交给大模型结合机器人提示词判断下一步。
- 大模型可输出或触发的处理类型包括：
  - `project_task`：需求、文档、测试用例、开发、项目问题。
  - `lark_cli_action`：需要查询、创建或更新飞书资源时，按本地 `skills/lark-*` 说明通过 `lark-cli` 执行。
  - `system_status`：页面状态、接口报错、系统配置、功能可用性。
  - `clarification`：缺少项目、上下文或目标，需要追问。
- 默认路由仍偏向 `project_task`，因为飞书沟通多数需要形成项目资产；但具体动作由提示词和模型判断。
- 支持显式指令：
  - `@机器人 项目：xxx ...` 强制进入项目聊天 AI。
  - `@机器人 系统状态 ...` 强制进入系统状态助手。

### FR-04 项目聊天 AI 集成

- 支持将飞书会话映射到系统内 `project_id + chat_session_id + session_id`。
- 支持从飞书消息创建或续接项目聊天任务树。
- 支持输出：澄清问题、需求摘要、PRD、测试用例、开发计划、交付结论。
- 支持将文档保存到项目文档目录，或返回可查看的文档链接/路径。

### FR-05 系统状态助手集成

- 仅在系统状态类问题中调用。
- 支持回答系统功能说明、当前页面/接口状态、最近请求、页面元素检查等。
- 如果问题需要项目上下文或文档产出，应转交项目聊天 AI。

### FR-06 自动回复策略

- 支持三种回复模式：
  - `auto`：低风险问题自动回复。
  - `draft`：生成草稿，需要人工确认后发送。
  - `silent`：只记录，不回复。
- 自动回复由系统统一调 `lark-cli im +messages-reply` 发送，带 `--idempotency-key` 防重复；大模型不得在生成最终回复时自行重复调用发送命令。
- 回复身份可配置：`bot` 表示以机器人身份回复，`user` 表示以当前已授权登录人身份回复；写操作或 user 身份缺 scope 时进入失败记录或待授权状态。
- 高风险操作、执行命令、发送外部通知、修改代码、删除数据必须进入 `draft` 或系统内确认。
- 回复内容应带状态，例如“已收到，正在整理需求”“已生成需求文档：xxx”。

### FR-07 文档产出

- 支持从飞书沟通自动整理：
  - 需求背景
  - 用户故事
  - 功能范围
  - 非功能需求
  - 风险与开放问题
  - 验收标准
  - 测试用例
- 文档默认落到 `docs/30-专项功能方案/` 或项目指定目录。
- 文档应关联飞书来源：`chat_id`、`message_id`、发起人、时间。

### FR-08 审计与可观测性

- 记录每条消息的接收、路由、AI 响应、发送结果。
- 支持查看失败原因：权限不足、AI 超时、飞书发送失败、路由低置信度。
- 支持重试，但必须保持幂等。

### FR-09 生成内容自定义配置

- 支持管理员配置“生成内容方案”，每个方案定义触发条件、输出内容、章节结构、格式和保存位置。
- 支持常见内置方案：`需求文档`、`测试用例`、`开发文档`、`会议纪要`、`待办清单`、`风险清单`、`接口联调说明`、`图片生成提示词`、`视频脚本`、`分镜脚本`、`行业营销方案`。
- 支持在飞书消息中显式指定生成类型，例如 `@机器人 根据刚才讨论生成测试用例`、`@机器人 输出开发文档`。
- 支持多个生成产物组合输出，例如同一次对话同时生成 `PRD + 测试用例 + 开发任务拆解`。
- 支持配置输出格式：Markdown、本系统项目文档、飞书文档、纯文本摘要、图片/视频生成任务；MVP 文档类先支持 Markdown，多模态类先生成任务草稿。
- 支持配置章节模板、必填字段、可选字段、示例提示词和验收规则。
- 支持按项目覆盖默认模板；没有项目级配置时使用全局默认配置。
- 当用户未指定生成内容时，系统根据意图选择默认方案；低置信度时先追问确认。


### FR-10 多模态与行业生成方案

- 支持将飞书对话记录转成多模态生成任务，而不只生成文档。
- 支持输出类型：`document`、`image_prompt`、`image_task`、`video_script`、`storyboard`、`shot_list`、`material_list`、`video_task`、`industry_report`。
- 图片类需求先生成图片提示词、尺寸、风格、参考素材和生成任务草稿；接入图片服务后再自动创建任务。
- 视频类需求先生成短视频脚本、分镜、镜头表、旁白、字幕、素材清单和视频生成任务草稿；接入视频生成服务后再返回任务进度。
- 支持行业模板：电商、本地生活、教育培训、企业软件、医美、房产、招聘、短视频营销、售后客服等。
- 支持按项目配置行业默认模板，例如某项目默认按“短视频营销”生成脚本、分镜、封面图提示词和发布文案。
- 多模态任务默认异步执行：飞书先回复“已创建生成任务/草稿”，生成完成后再通过链接、任务卡片或后续消息通知。
- 多模态生成必须保存来源对话、生成参数、素材引用、任务状态和结果链接，确保可追溯。


## 5. 非功能需求

- **安全**：不在日志输出 appSecret、accessToken、用户敏感内容全文。
- **可靠性**：长连接断开后自动重连；重复事件不重复处理。
- **可控性**：支持全局暂停、群/人白名单、关键词触发。
- **可追溯**：飞书消息、项目会话、任务树、文档路径之间可互相追踪。
- **扩展性**：后续可支持飞书审批、任务、文档等更多触发源。

## 6. MVP 范围

### 包含

1. 飞书 bot 事件监听。
2. 仅处理白名单群内 @ 机器人文本消息。
3. 默认路由到项目聊天 AI。
4. 支持“系统状态”关键词路由到系统状态助手。
5. 支持自动生成需求文档和测试用例草稿。
6. 回复飞书消息时带文档路径或处理结果摘要。
7. 记录处理日志和幂等状态。
8. 支持配置默认生成内容方案，MVP 至少支持 PRD、测试用例、开发文档三类。
9. 支持多模态生成任务草稿，MVP 至少支持图片提示词、视频脚本和分镜脚本。

### 暂不包含

1. 监听真人账号所有私聊。
2. 自动执行高风险命令。
3. 不经确认自动改代码并部署。
4. 多租户复杂权限模型。
5. 富文本卡片和交互按钮，MVP 先用文本回复。

## 8. 测试用例

| 编号 | 场景 | 前置条件 | 操作 | 期望结果 |
| --- | --- | --- | --- | --- |
| TC-01 | 白名单群 @ 机器人提需求 | 机器人在白名单群内 | 发送“@机器人 帮我整理机器人自动回复 PRD 和测试用例” | 系统路由到项目聊天 AI，创建/续接项目会话，回复已收到 |
| TC-02 | 系统状态问题路由 | 机器人在白名单群内 | 发送“@机器人 系统状态：当前页面接口有报错吗” | 系统路由到系统状态助手，返回页面/接口状态或说明缺少快照 |
| TC-03 | 未 @ 机器人消息 | `mention_required=true` | 群内普通聊天 | 系统忽略，不回复，不创建项目任务 |
| TC-04 | 非白名单群消息 | 群不在白名单 | @ 机器人提需求 | 系统拒绝或静默忽略，记录安全日志 |
| TC-05 | 重复事件 | 同一 `event_id` 投递两次 | 模拟飞书重试 | 只处理一次，只发送一条回复 |
| TC-06 | 机器人自发消息 | 收到 sender 为机器人自身 | 自动回复消息再次触发事件 | 系统过滤，避免循环回复 |
| TC-07 | 需求信息不足 | 用户只说“帮我写文档” | @ 机器人发送短消息 | 系统追问项目、目标和输出格式，不直接生成错误文档 |
| TC-08 | 生成需求文档 | 用户提供背景、功能范围、期望产物 | @ 机器人提交完整需求 | 系统生成 PRD 文档，飞书回复文档路径/摘要 |
| TC-09 | 生成测试用例 | 用户要求输出测试用例 | 完成需求文档后继续要求“补测试用例” | 系统在同一项目会话中补充测试用例章节 |
| TC-10 | 高风险执行请求 | 用户要求“直接部署/删除数据” | @ 机器人发送高风险操作 | 系统不自动执行，回复需要系统内确认 |
| TC-11 | AI 超时 | 下游 AI 服务超时 | 提交需求 | 回复“已收到但处理超时”，记录可重试状态 |
| TC-12 | 飞书发送失败 | IM 回复权限缺失 | 提交需求 | 处理记录为 reply_failed，后台可见错误原因 |
| TC-13 | 路由显式指定项目聊天 | 消息包含“项目：xxx” | @ 机器人发送项目需求 | 使用指定项目上下文处理，不走系统状态助手 |
| TC-14 | 路由显式指定系统状态 | 消息包含“系统状态” | @ 机器人询问配置入口 | 调用系统状态助手/system guide 能力回答 |
| TC-15 | 文档可追踪 | 已生成文档 | 打开文档 | 文档中包含来源 chat_id/message_id、发起人和时间 |
| TC-16 | 自定义生成类型 | 已配置生成内容方案 | 发送“@机器人 根据刚才讨论生成开发文档和测试用例” | 系统按指定类型生成开发文档和测试用例，不强制生成 PRD |
| TC-17 | 项目级模板覆盖 | 项目配置了自定义章节模板 | 发送“@机器人 生成需求文档” | 系统优先使用项目级模板，文档包含项目配置的必填章节 |
| TC-18 | 未指定生成类型 | 配置了默认生成方案 | 用户只说“整理一下刚才讨论” | 系统使用默认生成方案；如果意图低置信度则先追问确认 |
| TC-19 | 图片提示词生成 | 已配置图片类生成方案 | 发送“@机器人 根据刚才讨论生成产品主图提示词” | 系统生成图片 prompt、风格、尺寸、参考素材和任务草稿 |
| TC-20 | 视频脚本生成 | 已配置短视频行业方案 | 发送“@机器人 把刚才讨论生成短视频脚本和分镜” | 系统生成视频脚本、分镜脚本、镜头表和素材清单 |
| TC-21 | 多模态异步任务 | 已接入图片/视频生成服务 | 发送“@机器人 生成封面图和短视频任务” | 系统先回复任务已创建，后台生成完成后返回结果链接或任务状态 |
| TC-22 | 行业模板匹配 | 项目默认行业为电商 | 发送“@机器人 生成商品详情页素材方案” | 系统使用电商模板输出卖点、主图提示词、详情页结构和投放文案 |
| TC-23 | 行业模板低置信度 | 用户只说“做一套素材” | 无明确行业和输出类型 | 系统追问行业、渠道、目标人群和期望输出，不直接生成泛化结果 |
| TC-24 | 通用提示词驱动处理 | 连接器配置了机器人提示词 | @ 机器人发送新类型任务 | 不新增专用代码分支，项目聊天 AI 根据提示词和上下文理解后回复或调用工具 |
| TC-25 | lark-cli bot 身份回复 | `reply_identity=bot` 且机器人在群内 | @ 机器人发送消息 | 系统通过 `lark-cli im +messages-reply --as bot` 回复，并带幂等键 |
| TC-26 | lark-cli user 身份回复 | `reply_identity=user` 且已完成 user 授权 | @ 机器人发送消息 | 系统通过 `lark-cli im +messages-reply --as user` 回复；缺权限时记录待授权/失败原因 |
| TC-27 | 飞书技能目录可用 | 本地存在 `skills/lark-*` | 用户要求查/写飞书文档或表格 | AI 先参考对应 lark 技能说明，再通过 `project_host_run_command` 调用 lark-cli |

## 9. 验收标准

- 白名单群内 @ 机器人后，系统能稳定收到消息并回复。
- 默认项目需求会进入项目聊天 AI，而不是系统状态助手。
- 系统状态类问题能路由到系统状态助手，并在缺少页面快照时明确说明。
- 需求文档、测试用例、开发文档、图片提示词、视频脚本、行业方案等生成内容可通过全局或项目级方案配置。
- 图片/视频类需求能生成可追溯的任务草稿或异步生成任务，并返回任务状态或结果链接。
- 需求文档和测试用例能沉淀到项目文档路径。
- 每次自动回复都有幂等记录和审计日志。
- 自动回复发送链路使用 lark-cli，并能按配置切换 bot/user 身份。
- 新任务类型优先通过机器人提示词、大模型和 lark-cli 技能处理，不需要每个任务新增专用后端代码。
- 高风险操作不会被飞书消息直接触发执行。
