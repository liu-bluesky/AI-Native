# 系统统一用户反馈中心产品需求文档

> 文档版本：V2.0
>
> 更新日期：2026-07-20
>
> 原始范围：项目 AI 对话回答级 Bug 反馈
>
> 当前范围：系统产品体验 Bug、AI 问题、功能建议及其他用户反馈的统一闭环
>
> 目标端：桌面端、Web 管理后台
>
> 优先级：P0

## 1. 文档目的

建设一个系统级“用户反馈中心”，让用户可以针对整个产品提交问题和建议，而不是只能针对某个智能体或某条 AI 回答提交反馈。

统一反馈中心覆盖：

- 产品功能 Bug。
- 页面与交互体验问题。
- 性能、卡顿、闪退和稳定性问题。
- AI 回答质量问题。
- AI 执行、工具调用和任务产物问题。
- 功能建议与产品改进建议。
- 安全、隐私和权限问题。
- 无法归类的其他反馈。

AI 回答 ID和执行监管不是所有反馈的必填项，只在反馈与 AI 回答或执行过程有关时，作为可选或自动关联的排查证据。

统一反馈中心与当前“智能体反馈升级/规则反思”功能不是同一个产品：

- **统一用户反馈中心**面向所有用户，收集系统产品问题并管理处理流程。
- **智能体反馈升级**面向内部 AI 治理，用于对特定智能体进行反思、生成规则候选、审核和发布。
- 用户反馈可以在后台分析后，选择性地触发智能体反馈升级，但不能直接等同或强制绑定。

## 2. 背景与现状

### 2.1 当前系统实际能力

当前系统已经存在名为“反馈”的页面和接口，但它们主要服务于智能体反馈升级：

- 反馈对象依赖 `employee_id`。
- 入口位于智能体详情页。
- 只有智能体开启 `feedback_upgrade_enabled` 时才展示入口。
- 反馈后续重点是 AI 反思、规则候选、审核、发布和回滚。
- 路由存在，但桌面启动器和全局菜单没有独立的“用户反馈”入口。

因此，当前能力不能被视为完整的系统产品反馈功能。

### 2.2 当前用户问题

1. 用户不知道从哪里反馈系统问题，因为全局菜单没有“用户反馈”。
2. 页面报错、按钮失效、交互不合理等产品问题无法直接提交。
3. 反馈入口隐藏在智能体详情中，且名称容易被理解成普通客户反馈。
4. 非 AI 场景没有 `employee_id`，无法自然进入现有反馈模型。
5. AI 回答反馈、产品 Bug、功能建议和安全问题混在同一概念下，不利于处理和统计。
6. 用户无法集中查看自己提交的反馈和处理进度。
7. 后台缺少跨项目、跨页面、跨反馈类型的统一处理中心。

### 2.3 产品机会

建立统一入口和统一工单模型后，系统可以形成以下闭环：

```text
用户在任意页面发现问题或产生建议
  → 打开统一反馈入口
  → 选择反馈类型
  → 系统自动采集页面和运行环境
  → 用户补充描述与附件
  → 生成系统反馈工单
  → 后台分类、分派、分析和处理
  → 用户在“我的反馈”查看进度和结果
  → 必要时关联产品版本、AI 执行监管或智能体规则改进
```

## 3. 产品定位

### 3.1 产品名称

用户端统一使用：**用户反馈**。

后台统一使用：**反馈处理中心**。

不建议继续把面向用户的入口命名为“反馈升级”“反思反馈”或“规则反馈”。

### 3.2 产品定义

用户反馈中心是系统级产品能力，负责：

- 提供全局可发现的反馈入口。
- 根据反馈类型展示合适的动态表单。
- 自动采集可用于排查的产品上下文。
- 管理用户反馈的生命周期。
- 为后台提供分类、分派、分析、回复、解决和统计能力。
- 将特定反馈选择性关联到 AI 执行监管、产品发布记录或智能体治理流程。

### 3.3 核心原则

1. **全局可见**：入口不能依赖某个智能体或项目开关。
2. **类型优先**：用户先选问题类型，系统再决定需要哪些字段和证据。
3. **少填多带**：系统自动采集页面、版本、设备和运行上下文。
4. **证据按需关联**：回答 ID、日志和执行监管只在相关场景出现。
5. **用户可追踪**：每条反馈都有工单 ID、状态、处理记录和公开回复。
6. **后台可运营**：支持负责人、SLA、标签、合并、关联版本和统计。
7. **隐私默认保护**：敏感数据默认不采集，必要采集必须告知并脱敏。

## 4. 产品目标与指标

### 4.1 核心目标

1. 用户在任何核心页面都能在 10 秒内找到反馈入口。
2. 普通反馈在 60 秒内完成提交。
3. 系统自动携带足够的页面和版本上下文，减少后台追问。
4. AI 相关反馈可选关联回答 ID和执行监管，实现可追溯排查。
5. 用户能查看自己反馈的处理状态和结果。
6. 后台能统一处理产品 Bug、建议、AI 问题和安全问题。

### 4.2 成功指标

| 指标 | 首版目标 |
|---|---:|
| 核心页面反馈入口可发现率 | 100% |
| 反馈提交成功率 | ≥ 99% |
| 反馈提交时长中位数 | ≤ 60 秒 |
| 自动携带页面与版本上下文比例 | 100% |
| AI 回答类反馈回答 ID有效关联率 | ≥ 98% |
| 因缺少基础环境信息而追问用户的比例 | ≤ 10% |
| 用户可查看处理进度的反馈比例 | 100% |
| 高危安全反馈进入专属队列的比例 | 100% |

## 5. 用户角色

| 角色 | 主要诉求 | 权限范围 |
|---|---|---|
| 普通用户 | 快速反馈问题、建议并查看进度 | 创建反馈；查看和补充自己的反馈 |
| 项目成员 | 反馈项目使用问题并查看项目相关处理 | 创建反馈；查看本人反馈和授权项目公开反馈 |
| 客服/运营 | 分类、沟通、合并和分派反馈 | 查看业务反馈；更新分类、标签和用户回复 |
| 产品经理 | 分析产品问题与建议、关联版本 | 查看产品反馈；设置优先级、版本和产品结论 |
| 研发/测试 | 排查 Bug、更新修复信息和验证结果 | 查看授权证据；更新技术分析和验证结果 |
| AI 治理人员 | 处理 AI 回答和执行质量问题 | 查看 AI 证据；选择性创建智能体改进任务 |
| 安全管理员 | 处理安全、隐私和权限问题 | 访问安全专属队列和审计数据 |
| 系统管理员 | 配置分类、权限、SLA 和数据保留 | 系统级管理与审计 |

## 6. 反馈分类体系

### 6.1 一级分类

| 分类值 | 用户文案 | 适用范围 | 回答 ID |
|---|---|---|---|
| `product_bug` | 产品功能有问题 | 功能不可用、流程错误、数据异常 | 不需要 |
| `ui_experience` | 页面或交互体验问题 | 布局、文案、按钮、可用性、适配 | 不需要 |
| `performance_stability` | 卡顿、闪退或稳定性问题 | 卡顿、无响应、闪退、超时、资源占用 | 不需要 |
| `ai_answer` | AI 回答有问题 | 回答错误、遗漏、格式或理解偏差 | 可选，推荐关联 |
| `ai_execution` | AI 执行过程有问题 | 工具失败、任务中断、产物缺失、状态错误 | 可选，推荐关联 |
| `feature_request` | 功能建议 | 新功能、能力增强、流程优化 | 不需要 |
| `security_privacy` | 安全、隐私或权限问题 | 越权、敏感数据、凭据暴露、权限异常 | 按场景可选 |
| `other` | 其他反馈 | 无法归类的问题或意见 | 可选 |

### 6.2 二级分类建议

#### 产品功能有问题

- 功能无法使用。
- 操作结果错误。
- 数据未保存或丢失。
- 状态显示不正确。
- 权限与预期不一致。
- 文件上传、下载或导出异常。
- 通知或同步异常。

#### 页面或交互体验问题

- 页面布局错乱。
- 按钮、菜单或入口难以发现。
- 文案不清晰。
- 操作步骤过多。
- 空状态或错误提示不明确。
- 深色模式、分辨率或缩放适配问题。
- 键盘和无障碍操作问题。

#### 性能稳定性问题

- 页面加载慢。
- 操作卡顿。
- 应用闪退。
- 长时间无响应。
- 网络重试异常。
- CPU、内存或磁盘占用异常。
- 长任务进度不更新。

#### AI 回答有问题

- 内容不正确。
- 没有理解要求。
- 结果不完整。
- 格式不符合要求。
- 使用了错误上下文。
- 出现安全或隐私风险。

#### AI 执行过程有问题

- 工具调用失败。
- 任务执行中断。
- 任务声称完成但未产生产物。
- 文件修改不正确。
- 重复执行或死循环。
- 暂停、继续、取消状态异常。
- 执行监管记录不完整。

## 7. 产品范围

### 7.1 首版范围

- 桌面启动器增加“用户反馈”应用菜单。
- 用户头像或系统菜单增加“提交反馈”和“我的反馈”。
- 核心错误页面增加“反馈此问题”。
- 项目 AI 回答保留回答级“反馈问题”快捷入口。
- 反馈表单支持一级分类、动态字段、截图和附件。
- 系统自动采集页面、路由、版本、设备和错误上下文。
- AI 相关反馈支持关联回答 ID和执行监管。
- 用户可以查看自己的反馈列表、详情、状态和公开回复。
- 后台提供统一反馈处理中心。
- 支持状态流转、负责人、优先级、标签、内部备注和用户回复。
- 安全隐私类反馈进入独立权限队列。
- 明确现有智能体反馈升级功能的保留和调用边界。

### 7.2 暂不纳入首版

- 匿名外部网站反馈。
- Jira、Linear、禅道等外部工单系统双向同步。
- 自动执行代码修复并发布。
- 自动根据反馈修改智能体规则并直接生效。
- 面向全体用户公开展示全部反馈。
- 录屏和语音反馈。
- 基于 AI 的自动客服回复。

## 8. 全局入口设计

### 8.1 桌面启动器菜单

在桌面启动器中新增独立应用：

```text
应用名称：用户反馈
应用 ID：user-feedback
默认路径：/feedback
图标：MessageSquareWarning 或 ChatDotRound
适用角色：全部已登录用户
```

该入口必须加入应用启动器，不依赖当前项目和智能体。

Dock 默认不强制固定，但用户可自行固定。

### 8.2 用户菜单

用户头像菜单增加：

- 提交反馈。
- 我的反馈。

建议顺序：

```text
个人设置
我的反馈
提交反馈
帮助与文档
退出登录
```

### 8.3 全局快捷入口

可在系统右上角或帮助菜单中提供常驻“反馈”按钮。

交互要求：

- 图标必须配文字或 Tooltip。
- 键盘可聚焦。
- 点击区域不小于 36px，触控模式不小于 44px。
- 打开反馈抽屉时自动携带当前页面信息。

### 8.4 页面错误态入口

以下场景提供“反馈此问题”：

- 全局异常页。
- 接口请求失败且重试无效。
- 桌面应用启动失败。
- 文件上传、下载或保存失败。
- 长任务失败或超时。
- 权限拒绝但用户认为权限异常。

错误态反馈自动携带：

- 错误码。
- 请求 ID。
- 当前路由。
- 失败操作。
- 错误时间。
- 脱敏后的错误摘要。

### 8.5 AI 回答快捷入口

每条已完成的 AI 回答保留“反馈问题”按钮。

与全局反馈的区别：

- 默认分类为“AI 回答有问题”或“AI 执行过程有问题”。
- 自动带入回答 ID、项目、会话和回答摘要。
- 用户可以取消回答关联，并切换成其他反馈类型。
- 回答 ID不是系统反馈的全局必填字段。

## 9. 用户端反馈流程

### 9.1 主流程

```text
打开反馈入口
  → 选择反馈类型
  → 填写问题或建议
  → 查看系统自动采集的信息
  → 可选添加截图、附件和关联对象
  → 确认隐私授权
  → 提交
  → 获得反馈工单 ID
  → 在“我的反馈”查看进度
```

### 9.2 第一步：选择反馈类型

反馈抽屉首先展示 7 个清晰的分类卡片，不直接展示长表单。

```text
你想反馈什么？

[产品功能有问题]   [页面或交互体验问题]
[卡顿、闪退或稳定性问题]
[AI 回答有问题]    [AI 执行过程有问题]
[功能建议]          [安全、隐私或其他问题]
```

分类卡片应包含图标、一句话说明和典型例子。

### 9.3 第二步：动态表单

#### 通用字段

| 字段 | 必填 | 规则 |
|---|---:|---|
| 反馈类型 | 是 | 一级分类，允许返回修改 |
| 标题 | 是 | 5–100 字，可根据描述自动生成草稿但用户可编辑 |
| 详细描述 | 是 | 10–3000 字 |
| 期望结果 | 条件必填 | Bug 类必填，建议类可选 |
| 影响程度 | 是 | 轻微、一般、严重、阻塞 |
| 发生频率 | Bug 类必填 | 首次、偶尔、经常、每次 |
| 截图/附件 | 否 | 最多 5 个，支持图片、文本和日志压缩包 |
| 联系与通知 | 否 | 是否接收处理通知 |

#### 产品 Bug 动态字段

- 操作步骤。
- 实际结果。
- 期望结果。
- 是否可稳定复现。
- 是否影响数据安全或业务继续进行。

#### 页面体验动态字段

- 问题所在区域。
- 是否影响完成任务。
- 用户认为更合理的交互。
- 当前缩放比例和窗口尺寸自动采集。

#### 性能稳定性动态字段

- 发生时间。
- 持续时长。
- 是否导致闪退或数据丢失。
- 是否允许上传脱敏诊断日志。
- 系统自动采集性能摘要。

#### AI 回答动态字段

- 回答 ID，可为空。
- AI 实际回答摘要。
- 正确答案或期望内容。
- 问题类型：事实错误、理解错误、遗漏、格式、安全等。
- 是否允许处理人员查看脱敏后的对话和执行上下文。

#### AI 执行动态字段

- 回答 ID或运行 ID，可为空。
- 失败步骤或现象。
- 预期产物。
- 是否已经尝试重试、暂停或继续。
- 可选关联执行监管。

#### 功能建议动态字段

- 使用场景。
- 当前解决方式。
- 希望达到的效果。
- 使用频率。
- 影响用户范围。

#### 安全隐私动态字段

- 风险类型。
- 是否仍在发生。
- 是否涉及账号、权限或敏感数据。
- 是否需要紧急联系。
- 禁止要求用户在描述中粘贴密钥或密码。

### 9.4 自动采集信息

系统默认自动采集以下非敏感信息：

| 信息 | 示例 |
|---|---|
| 当前页面 | `/projects/proj_xxx/chat` |
| 页面标题 | 项目 AI 对话 |
| 入口来源 | 全局菜单、错误页、AI 回答等 |
| 客户端类型 | desktop/web |
| 客户端版本 | `2.4.0` |
| 操作系统 | macOS/Windows/Linux |
| 屏幕和窗口尺寸 | `1728x1117`、`1280x800` |
| 主题和缩放 | dark、125% |
| 当前项目 | 可为空 |
| 当前用户 | 服务端根据登录态确定 |
| 错误码和请求 ID | 存在错误上下文时采集 |
| 提交时间 | UTC 保存，本地时区展示 |

自动采集区域必须向用户可见，并允许用户展开查看。

### 9.5 可选诊断数据

以下数据必须由用户主动授权：

- 最近一段时间的前端错误日志。
- 脱敏后的网络请求摘要。
- 性能指标。
- 桌面端诊断日志。
- AI 对话上下文。
- AI 执行监管信息。
- 本地文件路径和文件元数据。

授权文案必须说明用途、范围和脱敏规则。

### 9.6 提交结果

成功后显示：

```text
反馈已提交
工单 ID：ufb_xxx
我们会在“我的反馈”中更新处理进度。

[查看反馈详情] [继续使用]
```

提交按钮必须支持 loading 和幂等，网络失败时保留已填写内容。

## 10. AI 回答与执行监管关联

### 10.1 适用范围

只有以下类型默认展示回答关联区域：

- AI 回答有问题。
- AI 执行过程有问题。
- 安全隐私问题且来源是 AI 对话。
- 其他反馈且用户主动选择关联 AI 回答。

### 10.2 回答 ID规则

使用 `assistant_message_id` 作为回答的 canonical ID。

用户可以：

- 从回答旁打开反馈，系统自动带入回答 ID。
- 在反馈表单中粘贴完整回答 ID。
- 从最近回答列表中选择。
- 移除错误的回答关联。

桌面端从本地聊天历史读取回答并自动带入回答 ID、会话 ID和回答快照。服务端校验用户项目权限、回答 ID与本地消息 ID的一致性以及快照格式，不查询服务端聊天消息库。

### 10.3 回答 ID展示

回答 ID在用户反馈详情和后台详情中可点击：

- 点击打开执行监管。
- 提供复制按钮。
- 无权限时显示原因。
- 监管数据不存在时显示“未采集”，不得伪造。

监管路径建议：

```text
/ai/supervision?project_id=<project_id>&answer_id=<assistant_message_id>&source=user_feedback
```

### 10.4 执行监管证据范围

用户授权且处理人员有权限时，可查看：

- 回答和用户问题。
- 项目、会话、请求 ID和运行 ID。
- 模型调用、模型用量和 Token。
- 工具调用、状态、耗时和错误。
- 任务树、执行步骤和验证结果。
- 上下文来源和压缩摘要。
- 暂停、继续、取消、恢复和超时事件。
- 产物引用和文件变更摘要。

敏感提示词、密钥、凭据和工具参数必须在服务端脱敏。

### 10.5 快照与实时数据

AI 相关反馈保存：

- 提交时回答与监管摘要快照。
- 最新监管数据引用。

提交时快照不可被覆盖。后续恢复运行产生的新数据展示为“最新执行数据”。

## 13. 工单状态机

### 13.1 状态定义

| 状态 | 含义 |
|---|---|
| `new` | 新提交，尚未处理 |
| `triaged` | 已完成初步分类和优先级判断 |
| `assigned` | 已分派负责人 |
| `in_progress` | 正在分析或处理 |
| `waiting_user` | 等待用户补充信息 |
| `pending_verification` | 已处理，等待验证 |
| `resolved` | 已确认解决 |
| `closed` | 工单关闭 |
| `reopened` | 问题仍存在，重新处理 |
| `duplicate` | 与其他反馈重复 |
| `rejected` | 无效、非产品问题或无法处理 |

### 13.2 状态流转

```text
new → triaged → assigned → in_progress
in_progress → waiting_user → in_progress
in_progress → pending_verification → resolved → closed
resolved/closed → reopened → assigned/in_progress
new/triaged/in_progress → duplicate
new/triaged → rejected
```

所有状态变更记录操作者、时间、原因和用户可见性。

### 13.3 用户影响程度与后台优先级

用户提交的是影响程度：

- 轻微。
- 一般。
- 严重。
- 阻塞。

后台维护独立优先级：

- P0 紧急。
- P1 高。
- P2 中。
- P3 低。

系统可根据类型、影响范围、安全风险和重复数量给出优先级建议，但最终由后台人员确认。

## 14. 与智能体反馈升级功能的边界

### 14.1 两套能力的区别

| 对比项 | 统一用户反馈中心 | 智能体反馈升级 |
|---|---|---|
| 面向对象 | 全部产品用户 | AI 治理和内部人员 |
| 反馈范围 | 产品、界面、性能、AI、建议、安全等 | 特定智能体输出和规则问题 |
| 是否依赖员工 | 否 | 是，依赖 `employee_id` |
| 用户入口 | 全局菜单和页面快捷入口 | 智能体详情或内部工具 |
| 核心结果 | 产品工单及处理闭环 | 反思分析和规则候选 |
| 是否可公开进度 | 用户可查看自己的工单 | 通常仅内部可见 |
| 状态机 | 产品工单状态 | 分析、审核、发布状态 |

### 14.2 关联方式

统一用户反馈中心可以将符合条件的 AI 类反馈发送到智能体反馈升级：

```text
用户反馈工单
  → 后台确认属于智能体能力或规则问题
  → 选择“创建智能体改进分析”
  → 生成内部分析记录或规则候选
  → 分析结果回写用户反馈工单
```

该动作必须由有权限的后台人员触发，不能因为用户选择“AI 回答有问题”就自动发布规则。

### 14.3 数据关系

建议关系字段：

- `user_feedback_id`：统一用户反馈 ID。
- `agent_feedback_id`：智能体反馈升级记录 ID。
- `relation_type`：根因分析、规则改进、重复关联等。
- `created_by`：创建关联的后台人员。

两边状态独立：智能体分析失败不能把用户反馈工单直接标记为失败。

### 14.4 现有页面处理建议

- 现有 `/feedback/:employee_id` 页面更名为“智能体反馈升级”或“智能体改进”。
- 不再把该页面作为系统“用户反馈”菜单。
- 新增 `/feedback` 用于用户端“我的反馈”。
- 新增 `/admin/feedback` 或等价后台路径用于反馈处理中心。
- 路由命名必须避免用户反馈和智能体反馈概念混淆。

## 17. 权限、隐私与安全

### 17.1 权限原则

- 所有已登录用户都能创建反馈和查看自己的反馈。
- 普通用户不能查看其他用户的反馈正文。
- 产品、研发和运营人员按职责查看相应队列。
- 安全隐私类反馈默认进入受限队列。
- 查看 AI 完整上下文需要额外权限和用户授权。
- 导出诊断数据必须记录审计日志。

### 17.2 禁止自动采集的数据

- 密码和验证码。
- API Key、Access Token、Cookie 和 Authorization Header。
- 私钥、证书和完整环境变量。
- 未经授权的文件正文。
- 完整剪贴板内容。
- 与当前反馈无关的聊天和浏览历史。

### 17.3 脱敏要求

脱敏必须由服务端执行：

- 凭据替换为固定掩码。
- 本地路径可隐藏用户名部分。
- 邮箱、手机号和个人标识按角色脱敏。
- 工具参数和错误堆栈按白名单展示。
- AI 提示词只展示必要摘要或受控原文。

### 17.4 数据保留

建议默认策略：

- 普通反馈正文：跟随产品合规策略长期保存或匿名化。
- 诊断日志：默认 30–90 天。
- AI 执行监管快照：跟随反馈有效期，敏感内容可单独提前过期。
- 已撤回反馈：保留最小审计记录。
- 安全反馈：按安全事件管理制度保存。

## 18. 通知与沟通

### 18.1 用户通知

- 提交成功。
- 处理人员请求补充信息。
- 状态变为处理中。
- 已有解决方案。
- 已解决或关闭。
- 被合并为重复反馈。
- 重开申请结果。

### 18.2 后台通知

- 新的严重或阻塞反馈。
- 新的安全隐私反馈。
- 工单被分派。
- 用户补充信息。
- SLA 即将超时或已经超时。
- 已解决工单被重开。

首版使用站内通知；飞书、邮件和其他渠道作为可配置扩展。

## 19. 异常场景

| 场景 | 产品处理 |
|---|---|
| 当前没有项目 | 允许提交系统级反馈，`project_id` 为空 |
| 用户从 AI 回答进入但回答已删除 | 允许提交，标记回答证据不可用 |
| 回答 ID无权限 | 不展示回答内容，提示无法关联 |
| 诊断日志采集失败 | 允许提交，标记诊断数据部分缺失 |
| 网络中断 | 本地保留草稿，恢复后重试 |
| 重复点击提交 | 幂等返回同一工单 |
| 附件上传失败 | 保留表单，允许移除附件后提交 |
| 用户退出抽屉 | 有内容时确认是否保存草稿 |
| 安全问题包含敏感信息 | 提醒用户删除明文凭据并进入受限队列 |
| 后台无法复现 | 请求用户补充，不直接关闭 |
| 智能体反馈升级关闭 | 不影响系统用户反馈提交 |
| 用户反馈功能维护 | 保留离线草稿并明确展示服务状态 |

## 20. 视觉与交互规范

### 20.1 信息层级

- 首屏先选择反馈类型，不展示所有字段。
- 主按钮始终只有一个：“提交反馈”。
- 自动采集信息放在可展开的“诊断信息”区域。

- 安全风险使用图标、文字和语义色共同表达。

### 20.2 状态设计

所有关键组件覆盖：

- 默认。
- Hover。
- Focus。
- Loading。
- Disabled。
- Error。
- Success。

### 20.3 无障碍与适配

- 键盘可完成全部提交流程。
- Focus 状态清晰。
- 文本对比度符合可访问性要求。
- 不只使用颜色表达状态。
- 小窗口使用全屏 Dialog 或分步表单。
- 深色模式重新设计背景层级和语义色。

埋点禁止包含用户反馈正文、AI 完整回答、提示词、工具参数和敏感信息。

### 21.2 后台看板

- 页面和模块分布。
- 高影响问题数量。
- 平均首次响应时间。
- 平均解决时间。
- SLA 达成率。
- 重开率。
- 重复反馈聚类。

## 25. 产品决策结论

1. 系统必须建设独立的“用户反馈”产品，而不是复用名称相同但目标不同的智能体反馈升级页面。
2. 系统反馈不依赖项目、智能体或 `employee_id`。
3. 产品 Bug、页面体验、性能稳定性、功能建议和安全问题是一等反馈类型。
4. AI 回答 ID和执行监管是 AI 相关反馈的证据，不是所有反馈的前置条件。
5. 用户必须有全局入口和“我的反馈”。
6. 后台必须有统一反馈处理中心，不能只按单个智能体查看反馈。
7. 现有智能体反馈升级保留为内部治理能力，并通过显式关联与用户反馈中心连接。

## 26. 最终产品主路径

```text
用户在任意页面发现产品问题或提出建议
  → 从全局菜单、错误态或页面快捷入口打开“用户反馈”
  → 选择反馈类型并填写动态表单
  → 系统自动携带页面、版本和设备上下文
  → AI 相关场景可选关联回答 ID与执行监管
  → 创建系统用户反馈工单
  → 用户在“我的反馈”查看状态和回复
  → 后台反馈处理中心完成分类、分派、分析、修复和关闭
  → AI 能力问题可选择性关联智能体反馈升级
```

统一用户反馈中心的核心价值，是让所有产品问题都有明确入口、统一工单和可追踪处理结果；回答 ID与执行监管负责增强 AI 类反馈的证据能力，而不是限制反馈范围。
