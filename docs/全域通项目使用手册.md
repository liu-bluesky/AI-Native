# 全域通项目使用手册

> **项目 ID**：`proj-550d976c`
> **项目名称**：全域通
> **项目定位**：直播电商项目
> **反馈升级**：已启用
> **版本**：v2.0
> **更新日期**：2026-03-16

> **每次对话流程（强制执行）：**
> 1. 收到用户提问后，先检索相关记忆（`recall_project_memory`）
> 2. 解决问题过程中的关键信息会自动保存到记忆系统
> 3. 问题解决后，系统会自动记录本次对话的要点
>
> **记忆自动保存的内容包括：**
> - 用户提出的问题
> - 使用的解决方案
> - 调用的工具和参数
> - 遇到的问题和解决方法
> - 重要的技术决策
>
> **注意：**
> - 记忆系统会自动工作，无需手动调用保存
> - 如果遇到重要问题或发现 Bug，可手动提交反馈工单（`submit_project_feedback_bug`）用于规则进化

## 第一部分：项目总览

### 1. 项目简介

#### 定位

全域通是面向直播电商场景的 AI 项目模块，核心作用是把项目成员能力、项目规则、项目记忆和技能工具编排到一个统一入口中，帮助 AI 在同一项目上下文里完成方案分析、界面设计、Vue 开发、跨端协作和数据库核验。

它重点解决四类问题：

- 项目知识分散，历史决策难复用
- 团队规则分散，执行标准不一致
- 工具调用分散，AI 难以选择合适成员和技能
- 多角色协作时，上下文不统一

#### 适用场景

- 直播电商后台页面、运营页面、企业站、移动端页面设计与实现
- Vue 组件开发、Element Plus 页面落地、前端工程化治理
- 跨端适配、性能优化、兼容性排查
- 数据结构核验、表结构查看、联调前字段确认
- 需求理解、规则检索、历史问题复盘

#### 能力边界

**项目能做什么**

- 检索项目成员、规则、记忆和运行时上下文
- 调用成员挂载的技能工具，如 `db-query`
- 对复杂前端与跨端问题给出结构化方案
- 对设计任务输出可交付的规格与交互说明
- 记录并沉淀反馈问题，推动规则进化

**项目不能做什么**

- 不能替代真实业务系统做最终审批和业务决策
- 不能默认修改数据库，`db-query` 仅适用于查询与核验
- 不能跳过项目规则直接输出高风险架构变更
- 不能用手册替代真实工具定义、真实权限和真实运行结果

### 2. 核心工具说明

以下说明以当前项目运行时代码为准，示例字段做了适度归纳。接入时应以实际返回值为最终标准。

#### `list_project_members`

**用途**

列出项目成员详情，用于判断当前项目有哪些 AI 成员、他们的技能、规则绑定和默认风格。

**参数**

无。

**返回值**

返回 `list[dict]`，每个成员对象常见字段包括：

```json
[
  {
    "project_id": "proj-550d976c",
    "employee_id": "emp-18b9cdfa",
    "name": "前端架构与跨端攻坚专家",
    "role": "member",
    "enabled": true,
    "skills": ["db-query"],
    "skill_names": ["db-query"],
    "rule_bindings": [
      {
        "id": "rule-xxx",
        "title": "前端开发经验",
        "domain": "前端开发经验"
      }
    ],
    "tone": "professional",
    "verbosity": "concise",
    "language": "zh-CN",
    "default_workflow": [],
    "tool_usage_policy": "",
    "employee_exists": true
  }
]
```

**使用场景**

- 任务开始前确认可用成员
- 需要按成员定向检索记忆或规则
- 调用技能前确定 `employee_id`
- 多角色协作时确认职责分工

#### `get_project_profile`

**用途**

获取项目基础配置，用于确认项目身份、描述、反馈能力和当前项目模型中的其他配置字段。

**参数**

无。

**返回值**

返回 `dict`。当前至少应包含以下信息，实际可能包含更多项目模型字段：

```json
{
  "id": "proj-550d976c",
  "name": "全域通",
  "description": "直播电商项目",
  "feedback_upgrade_enabled": true
}
```

**使用场景**

- 校验当前是不是“全域通”项目
- 检查反馈升级是否启用
- 给 AI 注入项目身份信息

#### `get_project_runtime_context`

**用途**

获取项目运行时摘要，适合在一次任务开始时快速建立上下文。

**参数**

无。

**返回值**

返回 `dict`，当前运行时摘要字段如下：

```json
{
  "project_id": "proj-550d976c",
  "project_name": "全域通",
  "member_count": 3,

    "emp-18b9cdfa",

    "emp-8cd60aec"

  "scoped_proxy_tool_count": 3,
  "rule_count": 7
}
```

**使用场景**

- 新会话开始时快速建立项目上下文
- 判断成员规模、规则数量、工具数量
- 在进一步检索前做影响面分析

#### `recall_project_memory`

**用途**

检索项目成员沉淀的历史记忆。支持按关键词检索，也支持按成员过滤。

**参数**

- `query`：可选，检索关键词。为空时返回最近记忆。
- `employee_id`：可选，指定成员 ID，仅检索该成员记忆。
- `project_name`：可选。运行时默认回退到当前项目名。为了避免跨项目误用，推荐显式传 `"全域通"`。
- `limit`：可选，默认 `100`，最大 `200`。

**返回值**

返回 `list[dict]`，每条记忆为一个对象，常见字段示例如下：

```json
[
  {
    "id": "mem-xxx",
    "employee_id": "emp-18b9cdfa",
    "project_name": "全域通",
    "content": "跨端列表页优先统一字段模型，再分端处理展示差异。",
    "created_at": "2026-03-10T09:30:00Z",
    "importance": 0.8,
    "type": "project-context"
  }
]
```

**使用场景**

- 开始实现前检索历史方案和坑点
- 复盘技术决策背景
- 排查线上问题时查找类似案例

**使用建议**

- 本项目统一规范：显式传 `project_name="全域通"`
- 先用精确关键词，再退化到更宽泛关键词
- 不确定归属成员时，先不传 `employee_id` 做全局检索

#### `query_project_rules`

**用途**

检索项目规则内容，支持按关键词和成员过滤。

**参数**

- `keyword`：可选，规则检索关键词，如 `"UI 设计"`、`"数据库设计"`、`"跨端"`。
- `employee_id`：可选，指定成员 ID，仅返回该成员相关规则。

**返回值**

返回 `list[dict]`，每条规则为一个对象，常见字段如下：

```json
[
  {
    "id": "rule-xxx",
    "domain": "UI 设计规范",
    "title": "表格页状态规范",
    "content": "必须补齐加载态、空态、错误态、权限态。",
    "updated_at": "2026-03-01T10:00:00Z"
  }
]
```

**使用场景**

- 设计前检索 UI 规范
- 开发前检索工程规范和组件约束
- 技术方案前检索架构与兼容性规则

#### `list_project_proxy_tools`

**用途**

列出项目成员当前可执行的技能代理工具。它只列技能代理，不替代内建工具清单。

**参数**

无。

**返回值**

返回 `list[dict]`，每个工具对象常见字段如下：

```json
[
  {
    "tool_name": "emp_18b9cdfa__db_query",
    "employee_id": "emp-18b9cdfa",
    "base_tool_name": "db-query",

    "entry_name": "db-query",
    "script_type": "py",
    "description": "数据库直连查询技能。"
  }

```

**使用场景**

- 确认项目里有哪些可执行技能
- 判断同名工具是否存在多成员冲突
- 在直接调用技能前做工具发现

#### `invoke_project_skill_tool`

**用途**

按工具名调用项目成员技能。适合统一入口调用 `db-query` 这类技能。

**参数**

- `tool_name`：必填，技能名或作用域后的技能名。
- `employee_id`：建议必填。若多个成员都挂载同名工具，不传会出现歧义错误。
- `args`：可选，JSON 对象形式的工具参数。
- `args_json`：可选，字符串形式 JSON；当未传 `args` 时生效。
- `timeout_sec`：可选，默认 `30`，最大 `600`。

**返回值**

返回 `dict`，执行结果示例如下：

```json
{
  "status": "ok",
  "exit_code": 0,
  "stdout": "{\"columns\":[\"id\",\"name\"],\"rows\":[[1,\"商品A\"]]}",
  "stderr": "",
  "command": [
    "python",
    "path/to/script.py",
    "--query",
    "SELECT id, name FROM products LIMIT 1",
    "--employee-id",
    "emp-8cd60aec"
  ]
}
```

失败时通常返回：

```json
{
  "error": "Ambiguous tool_name, provide employee_id. Candidates: ['emp-18b9cdfa', 'emp-8c12941b', 'emp-8cd60aec']"
}
```

**使用场景**

- 查看数据库表结构
- 验证接口字段与数据库字段是否一致
- 排查脏数据和联调数据问题

**本项目常用调用**

```json
{
  "tool_name": "db-query",
  "employee_id": "emp-8cd60aec",
  "args": {
    "query": "SELECT * FROM products LIMIT 10"
  }
}
```

#### `submit_project_feedback_bug`

**用途**

向项目反馈系统提交问题，用于规则进化和经验沉淀。

**参数**

- `employee_id`：必填，问题归属成员
- `title`：必填，问题标题
- `symptom`：必填，问题现象
- `expected`：必填，期望行为
- `category`：可选，默认 `general`
- `severity`：可选，默认 `medium`
- `session_id`：可选，会话 ID
- `rule_id`：可选，关联规则 ID
- `source_context`：可选，上下文对象

**返回值**

成功时返回：

```json
{
  "status": "created",
  "bug": {
    "id": "bug-xxx",
    "employee_id": "emp-18b9cdfa",
    "title": "跨端筛选条件不一致",
    "severity": "medium"
  }
}
```

失败时通常返回：

```json
{
  "error": "Employee emp-xxx is not an active project member"
}
```

**使用场景**

- 发现规则不适用或存在冲突
- 发现工具行为异常
- 发现高价值 Bug、边界问题或通用坑点

## 第三部分：推荐工作流

### 标准开发流程

```text
1. 获取项目上下文 → get_project_runtime_context
2. 记忆检索 → recall_project_memory
3. 规则检索 → query_project_rules
4. 技能调用 → invoke_project_skill_tool
5. 反馈闭环 → submit_project_feedback_bug
```

### 典型场景示例

#### 场景 1：新增数据库表

1. 调用 `get_project_runtime_context`，确认成员与技能规模。
2. 调用 `recall_project_memory`，推荐参数：

```json
{
  "query": "数据库表设计",
  "project_name": "全域通"
}
```

3. 调用 `query_project_rules`，检索关键词 `"数据库设计"`。
4. 调用 `invoke_project_skill_tool` 查看现有表结构：

```json
{
  "tool_name": "db-query",
  "employee_id": "emp-18b9cdfa",
  "args": {
    "query": "SHOW TABLES"
  }
}
```

5. 如发现命名、约束、索引或字段设计存在高价值经验，调用 `submit_project_feedback_bug` 沉淀问题。

#### 场景 2：开发新的 Vue 组件

1. 调用 `get_project_runtime_context`。
2. 调用 `recall_project_memory`，检索 `"Element Plus 表格组件"` 或 `"分页 筛选 排序"`。
3. 调用 `query_project_rules`，检索 `"UI 设计"` 和 `"组件"`。
4. 调用 `invoke_project_skill_tool` 查询数据结构：

```json
{
  "tool_name": "db-query",
  "employee_id": "emp-8cd60aec",
  "args": {
    "query": "SELECT * FROM products LIMIT 1"
  }
}
```

5. 由多端 UI 设计师给出交互与状态规范，由前端开发VUE完成实现，如有跨端兼容问题，再转交前端架构与跨端攻坚专家评估。

#### 场景 3：跨端协作（前后端联调）

当前项目成员列表中未提供独立后端成员，因此联调建议由“前端架构与跨端攻坚专家”承担接口一致性与跨端协调，由“前端开发VUE”承担前端接入。

1. 调用 `list_project_members` 确认当前成员与分工。
2. 调用 `recall_project_memory` 检索 `"API 接口设计"`、`"API 调用"`、`"订单接口"`。
3. 调用 `query_project_rules` 检索 `"接口"`, `"联调"`, `"字段"` 等关键词。
4. 调用 `invoke_project_skill_tool` 核验数据库表结构：

```json
{
  "tool_name": "db-query",
  "employee_id": "emp-18b9cdfa",
  "args": {
    "query": "DESC orders"
  }
}
```

5. 联调完成后，如发现字段命名、状态枚举、分页协议或错误码约定问题，调用 `submit_project_feedback_bug` 形成闭环。

## 第四部分：常见问题与故障排查

### Q1：数据库查询失败

**现象**

调用 `db-query` 时连接失败、查询失败或超时。

**排查**

- 首次使用需提供数据库配置
- 检查连接信息是否正确
- 仅支持 `SELECT` 语句
- 单次查询最多返回 `1000` 行

**建议**

- 优先加 `LIMIT`
- 避免长事务和全表扫描
- 若是字段核验，用 `LIMIT 1` 或 `DESC table_name`

### Q2：记忆检索无结果

**现象**

`recall_project_memory` 返回空数组。

**排查**

- 尝试更换关键词
- 检查 `project_name` 参数，推荐固定传 `"全域通"`
- 确认记忆保留期内是否有记录
- 尝试不指定 `employee_id` 进行全局检索

**建议**

- 先搜技术名词，再搜业务名词
- 先全局搜，再按成员定向搜

### Q3：规则查询返回多条结果

**现象**

`query_project_rules` 返回多条规则，不知道优先采用哪条。

**处理原则**

- 优先使用最近更新的规则
- 根据 `domain` 字段筛选
- 可以同时参考多条规则

**建议**

- UI 类问题优先看 UI 设计规范相关领域
- 工程类问题优先看前端开发经验、框架和项目模板相关领域

### Q4：技能调用参数错误

**现象**

`invoke_project_skill_tool` 返回参数格式错误、工具不存在或工具歧义。

**排查**

- 查看错误信息中的参数提示
- 确认 `employee_id` 是否正确
- 确认技能名称是否正确
- 确认 `args` 参数格式正确，必须是 JSON 对象

**高频原因**

- 当前项目 3 个成员都挂载了 `db-query`，如果只传 `tool_name="db-query"` 而不传 `employee_id`，会出现歧义

### Q5：反馈提交失败

**现象**

`submit_project_feedback_bug` 返回错误。

**排查**

- 检查必填参数是否完整
- 确认项目反馈升级功能已启用
- 检查 `employee_id` 是否属于该项目成员

**建议**

- 标题写成可检索的短句
- `symptom` 写现象，`expected` 写期望，不要混写

## 第五部分：最佳实践

### 1. 参数规范

- 调用记忆时，建议始终传 `project_name="全域通"`
- 调用技能时，必须传 `employee_id`
- 提交反馈时，必须传 `employee_id`、`title`、`symptom`、`expected`

### 2. 员工选择

- 架构、跨端、性能、兼容性问题：优先选择 `emp-18b9cdfa`
- 页面设计、布局规范、交互状态：优先选择 `emp-8c12941b`
- Vue 开发、Element Plus 落地、页面实现：优先选择 `emp-8cd60aec`
- 跨端任务：分别调用相关员工的能力

### 3. 记忆管理

- 定期检索记忆，避免重复踩坑
- 及时提交反馈，积累项目经验
- 使用精确关键词提高检索准确率
- 同一问题可先全局检索，再按成员缩小范围

### 4. 规则遵循

- 开发前先检索相关规则
- 遵循规则中的最佳实践
- 发现规则不适用时及时反馈
- 涉及技术选型时，必须补齐适用场景、不适用场景、迁移成本和维护成本

### 5. 技能使用

- 首次使用技能时注意配置要求
- 数据库查询注意安全限制
- 技能调用失败时查看详细错误信息
- `db-query` 更适合做结构核验、样本查看和联调辅助，不适合代替正式数据治理流程

## 附录：面向 AI 的使用说明

这份手册可以直接让 AI 使用，但应以“项目操作指南”的方式使用，而不是把它当成唯一事实来源。

**推荐用法**

- 作为系统提示词或开发者提示词的一部分
- 作为项目知识库文档供 AI 检索
- 作为工具调用前的操作规范和选择规则

**不建议的用法**

- 只给这份手册，不给真实工具 schema
- 让 AI 直接根据示例猜测所有返回字段
- 用手册覆盖运行时权限、数据库权限和真实错误返回

**落地建议**

- 手册负责说明“先做什么、为什么这样做、常见坑是什么”
- 真实 MCP 工具定义负责说明“参数 schema 和真实返回”
- 两者同时给 AI，效果最好

**给 AI 的最小执行准则**

```text
1. 先调 get_project_runtime_context 建立上下文
2. 再调 recall_project_memory 检索历史经验
3. 再调 query_project_rules 确认规范
4. 需要数据核验时，用 invoke_project_skill_tool 调 db-query，且必须传 employee_id
5. 遇到高价值问题或规则不适用时，调 submit_project_feedback_bug
```
