# 桌面智能体目标驱动执行流升级计划

## 背景

当前桌面智能体问答能力基本可用，但一进入开发、修改、修复类任务，执行质量明显下降。

核心问题不是模型不能执行，而是 Runtime 的控制逻辑把模型执行空间压坏了：

- 用 `action_markers` 关键词判断实现型任务。
- 一旦命中，就把任务推进“必须产生副作用工具”的轨道。
- 连续读取、搜索、列目录后，会强塞“下一轮必须调用 write_file / apply_patch / run_command”。
- 工具失败后，又强塞“必须换 strategy_signature”。
- 完成判定偏向“有没有写入或命令副作用”，而不是“用户目标是否被满足并验证通过”。

这会导致模型在还没理解清楚、还没定位充分、还没形成稳定方案时，被 Runtime 推着写文件或跑命令，最终表现为工具参数变形、重复失败、修改偏离目标、用户看不到真实推进过程。

## 用户期望流程

用户想要的是目标驱动的执行过程：

```text
用户发送需求
  -> 根据上下文理解需求
  -> 创建本轮任务目标
  -> 动态拆分当前任务小节点
  -> 播报当前小节点的工作内容和工作方向
  -> 执行探索 / 修改 / 工具调用
  -> 测试或检测当前节点结果
  -> 发现偏离目标时修正
  -> 当前节点通过后进入下一节点
  -> 最后整体测试验证
  -> 告知用户修改内容、验证结果、风险和细节
```

重点是：生命周期固定，内容动态。

Runtime 应固定“理解、建目标、拆节点、执行、观察、校验、修正、交付”这些阶段，但不能写死每一步展示文案、节点标题、回答格式或工具动作。不同需求应该生成不同的节点和不同的最终说明。

## 设计目标

1. 取消 `action_markers` 对执行控制的决定权。
2. 将任务推进从“关键词 + 副作用工具”改成“TaskGoal + TaskNode + Verification”。
3. 让模型自主选择工具，Runtime 只负责安全、状态、权限、审计和验证闭环。
4. 让用户能看到当前小节点、工作方向、进度、测试结果和修正动作。
5. 防止修改偏离本轮任务目标。
6. 最终回答根据需求类型动态组织，不强制固定格式。

## 当前严重逻辑缺陷

### 1. 关键词分类被当成控制开关

当前 `text_indicates_implementation_task()` 用词表判断实现型任务：

```text
修复 / 解决 / 修改 / 优化 / 实现 / 创建 / write / fix / update ...
```

这个结果直接变成 `implementation_task`，后续影响 agent loop 的退出和强制控制。

问题：

- 关键词只能作为弱信号，不能决定 Runtime 调度策略。
- “优化一下方案”“解释怎么修复”“看看哪里需要改”都可能被误判成必须执行。
- 实现型任务也不等于必须立即写文件。

### 2. 实现型任务被强行要求产生副作用

当前逻辑里，如果实现型任务已经执行过读取、搜索或列目录，但没有写文件、打补丁或跑命令，Runtime 会强塞控制消息：

```text
下一轮必须调用 write_file、apply_patch 或必要的 run_command。
```

问题：

- Runtime 不应该替模型决定下一步工具。
- 有些任务需要更多定位、对比、确认、生成方案，不能按轮数强迫写入。
- 强制副作用会诱导模型生成不成熟修改。

### 3. 完成判定不是目标导向

当前退出条件倾向于：

- 有没有 tool_calls。
- 有没有副作用工具。
- 有没有重复失败。

问题：

- 执行任务是否完成，应该看用户目标是否满足。
- 修改是否正确，应该看验证证据。
- 没有副作用工具不一定失败；有副作用工具也不一定完成。

### 4. 失败恢复太机械

当前失败后会要求换 `strategy_signature`。

问题：

- “换策略”不等于“修正原因”。
- 有些失败只需要补参数、重新读取上下文或询问用户。
- 重复失败应该进入 Retry Router，由 Runtime 判断是 schema 修复、工具重试、重新规划、验证失败，还是需要用户决策。

### 5. 缺少当前小节点和偏离检测

当前执行循环缺少稳定的 `TaskNode` 状态。

问题：

- 用户看不到当前小节点是什么。
- Runtime 无法判断模型是否偏离目标。
- 工具调用结果无法归属到具体节点。
- 最终总结只能拼工具摘要，不能形成可靠交付说明。

## 新执行模型

### 核心对象

#### TaskIntent

只负责语义分类，不直接控制工具。

建议类型：

```text
query
analysis
implementation
bugfix
refactor
documentation
deployment
destructive
ambiguous
```

#### TaskGoal

本轮任务目标，是执行期间的最高约束。

示例：

```json
{
  "goal_id": "goal_xxx",
  "title": "修复桌面智能体执行任务时过早强制副作用工具的问题",
  "user_request": "用户原文",
  "success_criteria": [
    "不再由 action_markers 直接决定执行控制",
    "开发任务按小节点推进",
    "每个节点有验证结果",
    "最终回答包含修改内容和验证结论"
  ],
  "constraints": [
    "不展示隐藏思维链",
    "不做无关文件修改",
    "高风险操作必须授权"
  ]
}
```

#### TaskNode

动态生成的小节点，不固定为“分析 / 实现 / 验证”三步。

示例：

```json
{
  "node_id": "node_runtime_entry",
  "title": "定位桌面智能体执行控制入口",
  "purpose": "确认当前任务为什么会被推向强制写入",
  "status": "running",
  "allowed_actions": ["read_file", "search_text"],
  "done_check": "找到分类、调度和退出判定的代码位置",
  "verification_result": ""
}
```

#### ProgressUpdate

展示给用户的进度，不是隐藏思维链。

```json
{
  "type": "progress_update",
  "node_id": "node_runtime_entry",
  "summary": "正在定位桌面智能体的执行控制入口。",
  "work_direction": "先确认任务分类如何影响工具调度，再判断是否需要调整 Runtime 状态机。",
  "status": "running"
}
```

#### VerificationReport

每个节点和最终交付都需要验证。

```json
{
  "target": "node_runtime_entry",
  "status": "passed",
  "checks": [
    {
      "type": "fact",
      "summary": "已定位 text_indicates_implementation_task 和 run_agent_loop 控制分支。"
    },
    {
      "type": "goal_alignment",
      "summary": "定位结果与用户提出的 action_markers 怀疑一致。"
    }
  ]
}
```

## 新生命周期

```text
ReceiveUserRequest
  -> UnderstandRequest
  -> BuildTaskGoal
  -> BuildOrUpdateTaskNodes
  -> StartNode
  -> EmitProgressUpdate
  -> ModelChoosesNextAction
  -> RuntimePolicyCheck
  -> ExecuteToolOrAnswer
  -> ObserveResult
  -> VerifyNode
  -> DriftCheck
  -> RepairOrContinue
  -> FinalVerification
  -> DeliverSummary
```

## 关键规则

### 1. action_markers 降级为弱信号

保留也可以，但只能进入 TaskIntent 的候选证据。

禁止：

```text
action_markers 命中 -> implementation_task=true -> 强制 write_file
```

允许：

```text
action_markers 命中 -> TaskIntent 候选为 implementation -> 结合上下文、对象、用户确认和目标生成 TaskGoal
```

### 2. Runtime 不强制模型调用具体副作用工具

禁止 Runtime 注入：

```text
下一轮必须调用 write_file / apply_patch / run_command
```

改为注入结构化状态：

```json
{
  "runtime_state": "node_needs_progress",
  "current_node": "...",
  "observations": ["已读取相关文件，但尚未形成可验证修改"],
  "available_next_actions": [
    "继续读取必要文件",
    "提出修改计划",
    "执行低风险修改",
    "说明阻塞原因并询问用户"
  ],
  "constraint": "下一步必须推进当前节点，不能重复无效探索。"
}
```

模型决定下一步，Runtime 只校验安全和状态合法性。

### 3. 完成判定改为目标验证

一个任务是否完成，不看有没有副作用工具，而看：

- TaskGoal 的 success_criteria 是否满足。
- 当前 TaskNode 是否有 verification_result。
- 是否有未解决失败。
- 是否有目标偏离。
- 是否需要用户确认。

### 4. 节点推进必须可见

每个节点开始时发：

```text
当前节点：<节点标题>
工作方向：<本节点要解决的问题>
```

节点完成时发：

```text
节点结果：<做了什么>
验证结果：<怎么确认>
下一步：<进入哪个节点>
```

这些是可见进度，不是思维链。

### 5. 偏离检测

每次工具调用后检查：

- 当前动作是否服务于 TaskGoal。
- 是否修改了目标范围外文件。
- 是否引入用户没有要求的功能。
- 是否因为工具失败开始无关尝试。
- 是否需要回到上一个节点重新规划。

偏离时不继续硬跑，进入修正：

```text
DriftDetected
  -> summarize drift
  -> choose repair action
  -> revert / adjust plan / ask user / continue
```

## 推荐落地阶段

### 阶段 1：移除关键词对执行控制的决定权

目标：

- `action_markers` 不再直接产生 `implementation_task` 控制开关。
- 原有实现型强制副作用逻辑改成结构化状态提醒。

改动方向：

- 新增 `TaskIntentAssessment`。
- `text_indicates_implementation_task()` 改为 `infer_task_intent_evidence()`。
- `implementation_task` 不再直接参与 `run_agent_loop` 的强制退出。

验收：

- 用户问“怎么修复这个问题”不会被强制写文件。
- 用户要求“创建 register.html”仍能进入开发任务，但不会在未定位前被强制写入。

### 阶段 2：引入 TaskGoal

目标：

- 每轮任务先创建目标。
- 目标写入 state、transcript、requirement。
- 后续工具调用和总结都引用同一个目标。

验收：

- 运行记录里能看到本轮目标。
- 最终回答能对照目标说明完成情况。

### 阶段 3：引入动态 TaskNode

目标：

- 模型或 Runtime 根据目标生成小节点。
- 节点不是固定模板，而是按需求动态生成。
- 每个工具调用归属到当前节点。

验收：

- 用户能看到“当前正在做哪一步”。
- 恢复会话后能继续当前节点。

### 阶段 4：进度播报

目标：

- 每个节点开始、关键发现、工具执行、验证完成都产生可见进度。
- 前端按事件展示，而不是拼接模糊日志。

验收：

- 长任务不再只有“正在执行本地工具”。
- 用户能看出当前工作方向。

### 阶段 5：验证与偏离检测

目标：

- 每个节点结束前必须验证。
- 修改偏离目标时自动修正或暂停询问。

验收：

- 修改目标外文件会被标记为偏离。
- 未验证通过不会直接最终完成。

### 阶段 6：动态最终交付

目标：

- 最终回答根据需求类型组织，不设死模板。

示例：

- 查询类：直接回答核心结论。
- 修复类：说明原因、改动、验证、风险。
- 开发类：说明新增能力、涉及文件、测试结果。
- 文档类：说明写入位置、内容结构、后续项。
- 部署类：说明环境、产物、策略、回滚方式。

验收：

- 不同需求有不同回答形态。
- 但都包含用户需要判断交付是否可信的关键信息。

## 需要废弃或收敛的旧逻辑

| 旧逻辑 | 处理 |
|---|---|
| `action_markers -> implementation_task` | 降级为 intent evidence |
| `implementation_exploration_budget_message` 强制写入 | 替换为节点状态提醒 |
| `必须换 strategy_signature` | 替换为 Retry Router |
| 只按副作用工具判断执行有效性 | 替换为 VerificationReport |
| 前端模糊运行摘要 | 替换为 TaskNode + ProgressUpdate |

## 最小验收用例

### 用例 1：纯查询

用户：

```text
这个字段干嘛的？
```

期望：

- 创建 query 类型 TaskGoal。
- 不强制写文件。
- 直接回答。
- 最终状态 completed。

### 用例 2：分析但不修改

用户：

```text
看看当前桌面智能体为什么执行任务很差。
```

期望：

- 创建 analysis 类型 TaskGoal。
- 可以读取代码。
- 不强制 write_file。
- 输出分析结论和证据位置。

### 用例 3：明确开发

用户：

```text
修复桌面智能体 action_markers 导致执行任务偏离的问题。
```

期望：

- 创建 bugfix 类型 TaskGoal。
- 动态节点：定位、设计、修改、测试、总结。
- 修改前先定位原因。
- 每个节点有进度播报。
- 测试失败进入修正，不直接完成。

### 用例 4：探索多轮后仍不写入

用户：

```text
先帮我确认要怎么改，不要动代码。
```

期望：

- 即使命中“改”，也不能强制副作用工具。
- TaskIntent 应是 analysis 或 planning。
- 最终输出方案。

### 用例 5：修改偏离目标

用户：

```text
只修复任务分类逻辑。
```

如果模型试图重构整个 Runtime：

- DriftCheck 标记偏离。
- 暂停或收敛回任务分类逻辑。
- 不继续扩大改动范围。

## 结论

当前桌面智能体执行差的根因，是 Runtime 用关键词和副作用工具控制任务推进。

升级方向应改为目标驱动：

```text
TaskIntent 负责理解
TaskGoal 负责约束
TaskNode 负责推进
ProgressUpdate 负责可见过程
VerificationReport 负责完成判断
DriftCheck 负责防偏离
RuntimePolicy 负责安全
```

这样才能让桌面智能体既能回答问题，也能稳定执行开发任务。
