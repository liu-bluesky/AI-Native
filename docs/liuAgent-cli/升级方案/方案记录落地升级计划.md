# 方案记录落地升级计划

## 目标

把 `方案记录.txt` 中的通用 Agent 流程，落到当前系统已有的桌面本地智能体链路中：

```text
ProjectChat.vue
  -> native-desktop-bridge.js
  -> Tauri command: liuagent_start_local_chat
  -> src-tauri/src/liuagent_core/runtime.rs
  -> 本地模型请求 / 本地工具 / 权限 / 状态 / outbox
  -> ProjectChat 运行轨迹展示与 requirement 记录
```

这不是重写一套新 Agent，而是在现有 `liuAgent Core` 上补齐：

- 结构化上下文构建
- Prompt 栈解析
- 计划与澄清门
- 工具调度与 observation 合并
- 验证与重试路由
- 响应格式化
- 记忆/状态分类写入
- 可恢复、可审计的事件闭环

## 当前系统基线

### 已有能力

| 能力 | 当前落点 |
|---|---|
| 桌面入口 | `web-admin/frontend/src/views/projects/ProjectChat.vue` 的 `runDesktopLocalAgentChat` |
| 原生桥 | `web-admin/frontend/src/utils/native-desktop-bridge.js` |
| Tauri 命令 | `web-admin/frontend/src-tauri/src/main.rs` 的 `liuagent_start_local_chat` |
| 本地 Runtime | `web-admin/frontend/src-tauri/src/liuagent_core/runtime.rs` |
| 本地工具 | `liuagent_core/mod.rs` 聚合 file、command、network、mcp 工具 |
| 权限控制 | `liuagent_core/permission.rs` 与工具执行前的 permission decision |
| 事件记录 | `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/transcript.jsonl` |
| 当前状态 | `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/state.json` |
| 审计 | `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/audit.jsonl` |
| requirement 本地记录 | `.ai-employee/requirements/<project_id>/<chat_session_id>.json` |
| 同步 outbox | `.ai-employee/query-mcp/outbox/<project_id>__<chat_session_id>.jsonl` |

### 主要缺口

| `方案记录.txt` 节点 | 当前缺口 |
|---|---|
| Parse Request | 只有用户原文和历史传入，没有独立结构化解析结果 |
| Build Working Context | 上下文主要来自 `historyRows` 和附件，没有分层 context package |
| Resolve Prompt Stack | 只拼接 `systemPrompt + desktopAgentGlobalPrompt`，没有 prompt source 清单和冲突检查 |
| Execution Planning | 主要靠 prompt 约束模型输出，没有 Runtime 级 Plan 对象 |
| Need Clarification | 没有结构化 clarity assessment 和 confirmation gate |
| Execution Scheduler | 目前是模型 step + tool loop，缺少明确 scheduler 状态对象 |
| Context Merge | 工具结果作为 tool observation 继续给模型，但没有 observation 压缩与污染隔离 |
| Verification Engine | 没有 Fact / Logic / Constraint / Goal 四类验证对象 |
| Retry Router | 有有限失败处理，未形成 Search / Plan / Tool / Draft 分流 |
| Memory Classifier | requirement/outbox 已有，但没有 Long / Short / Project 分类策略 |

## 总体落地架构

```text
Receive Request
  -> RequestParser
  -> ContextBuilder
  -> PromptStackResolver
  -> PlanningGate
  -> RuntimeScheduler
  -> ModelStep / ToolStep
  -> ObservationMerger
  -> VerificationEngine
  -> RetryRouter
  -> ResponseFormatter
  -> MemoryClassifier
  -> StateWriter + OutboxWriter
```

落地原则：

1. **不替换现有链路**：继续从 `ProjectChat.vue` 进入 Tauri Runtime。
2. **先加结构，不改体验**：新增结构化对象先写入 state/transcript，不立刻大改 UI。
3. **Prompt 负责判断，Runtime 负责约束**：模型可以做语义判断，状态转移、权限和副作用由代码保证。
4. **诊断与模型历史隔离**：运行时错误、provider 诊断、工具审计不能直接作为普通 assistant 历史污染下一轮。
5. **每阶段都可验收**：每阶段至少有 Rust 单测或前端构建/回放验证。

## 新增核心对象

### AgentRunContext

保存一次运行的结构化上下文。

```json
{
  "project_id": "proj-d16591a6",
  "chat_session_id": "chat-session-xxx",
  "run_id": "local_xxx",
  "workspace_path": "/Volumes/work_mac_1_5T/self/ai-employee",
  "user_request": {
    "raw": "用户原文",
    "normalized": "标准化后的本轮目标",
    "attachments": []
  },
  "history_context": {
    "conversation_summary": "",
    "recent_user_messages": [],
    "recent_assistant_messages": [],
    "excluded_diagnostics": []
  },
  "project_context": {
    "project_name": "",
    "chat_settings": {},
    "workspace_snapshot": {}
  },
  "runtime_context": {
    "provider_id": "",
    "model_name": "",
    "model_capabilities": [],
    "available_tools": []
  }
}
```

落点：

- Rust：`liuagent_core/types.rs`
- 生成：`runtime.rs::build_agent_run_context(...)`
- 持久化：`state.json.model_runtime.context`

### PromptStack

记录本轮 prompt 的来源和优先级，避免之后排查时只看到一坨 system prompt。

```json
{
  "items": [
    {
      "source": "project_chat_settings.system_prompt",
      "priority": 100,
      "content_hash": "sha256:...",
      "content_preview": "..."
    },
    {
      "source": "system_config.desktop_agent_global_prompt",
      "priority": 80,
      "content_hash": "sha256:...",
      "content_preview": "..."
    },
    {
      "source": "runtime_policy",
      "priority": 60,
      "content_hash": "sha256:...",
      "content_preview": "..."
    }
  ],
  "resolved_system_prompt": "...",
  "warnings": []
}
```

落点：

- 前端仍由 `buildLocalLiuAgentSystemPrompt()` 传入现有 prompt。
- Runtime 端新增 `resolve_prompt_stack()`，至少把来源拆开记录。
- 后续再把前端传入从 string 升级为 `systemPromptParts`。

### ClarityAssessment

实现“Need Clarification?” 的结构化判断。

```json
{
  "score": 4,
  "task_type": "query | implementation | deployment | destructive | ambiguous",
  "goal_clear": true,
  "target_clear": true,
  "scope_clear": true,
  "expected_result_clear": false,
  "requires_confirmation": true,
  "interpretation": "用户要基于方案记录设计可落地升级计划",
  "ambiguities": []
}
```

落地策略：

- 第一版不新增一次额外模型请求，先用规则 + 轻量 schema 判断：
  - 有写入、修改、部署、删除、运行命令：`requires_confirmation=true`
  - 只有查询/解释：可直接回答
  - 目标文件路径明确且用户直接要求写入文档：视为已确认写文档
- 第二版再引入模型结构化分类。

### PlanState

把 `Execution Planning` 从纯文本转成可恢复对象。

```json
{
  "plan_id": "plan_xxx",
  "status": "draft | approved | running | completed | blocked | failed",
  "root_goal": "设计可落地升级计划",
  "nodes": [
    {
      "node_id": "context-audit",
      "title": "核对当前桌面智能体实现和现有设计文档",
      "status": "done",
      "verification_result": "已读取对比区别、编排、规划、状态存储文档"
    }
  ],
  "current_node_id": "write-plan",
  "destructive_steps": [],
  "created_at_epoch_ms": 0,
  "updated_at_epoch_ms": 0
}
```

复用现有 requirement 的 `task_tree/current_task_node` 字段，但 Runtime 内部也要在 `state.json` 保存同一份 `PlanState`，避免前端/服务端不可用时丢失。

### Observation

隔离工具结果、模型内容和运行诊断。

```json
{
  "observation_id": "obs_xxx",
  "source": "tool_result | model_output | runtime_error | user_input",
  "visibility": "model_context | user_visible | audit_only",
  "summary": "",
  "content_ref": "",
  "tool_call_id": "",
  "error_code": "",
  "created_at_epoch_ms": 0
}
```

关键规则：

- `runtime_error` 默认不进入下一轮普通 assistant history。
- 工具结果进入模型时先压缩为 observation。
- 用户可见失败摘要仍显示在 UI，但不等于下一轮模型上下文。

### VerificationReport

实现 Fact / Logic / Constraint / Goal 四类验证。

```json
{
  "verification_id": "verify_xxx",
  "target_node_id": "write-plan",
  "checks": [
    {
      "type": "fact",
      "status": "passed",
      "summary": "目标文件已创建"
    },
    {
      "type": "constraint",
      "status": "passed",
      "summary": "未修改代码，只新增文档"
    }
  ],
  "overall_status": "passed | failed | blocked",
  "created_at_epoch_ms": 0
}
```

### MemoryWritePlan

实现 Long / Short / Project 分类，但第一版只做本地 requirement/outbox 分类，不引入新数据库。

```json
{
  "items": [
    {
      "scope": "short",
      "target": "state.json",
      "content": "本轮已读取相关设计文档"
    },
    {
      "scope": "project",
      "target": "requirements/<project>/<session>.json",
      "content": "本轮交付升级计划文档"
    },
    {
      "scope": "long",
      "target": "deferred",
      "content": "形成通用智能体编排升级路线"
    }
  ]
}
```

## 分阶段实施计划

### 阶段 0：基线收敛

目标：先确认当前链路没有分叉状态和隐式身份修补。

改动：

- 梳理 `ProjectChat.vue` 到 `runtime.rs` 的参数映射。
- 明确 `historyRows`、`systemPrompt`、`modelRuntime`、`attachments` 当前进入模型的格式。
- 列出 runtime 产生的事件、state、checkpoint、audit、outbox。
- 在文档中固定 canonical 状态路径：
  - `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/state.json`
  - `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/transcript.jsonl`
  - `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/audit.jsonl`
  - `.ai-employee/requirements/<project_id>/<chat_session_id>.json`

验收：

- 输出一份现状清单，能说明每个字段从哪里来、写到哪里。
- `runtime.rs` 不再出现按 provider/model 强制改写身份回答的逻辑。

### 阶段 1：上下文构建与污染隔离

目标：把 Build Working Context 落成 `AgentRunContext`，避免运行诊断污染模型历史。

改动文件：

| 文件 | 改动 |
|---|---|
| `web-admin/frontend/src/views/projects/ProjectChat.vue` | 保持传参兼容，增加历史消息的元信息预留字段 |
| `web-admin/frontend/src-tauri/src/liuagent_core/types.rs` | 新增 `AgentRunContext`、`HistoryContext`、`Observation` |
| `web-admin/frontend/src-tauri/src/liuagent_core/runtime.rs` | 新增 `build_agent_run_context`，把 history、附件、provider、工具清单整理成结构化上下文 |

关键实现：

1. `LocalChatMessage` 增加可选字段：

```rust
pub source_kind: Option<String>,
pub diagnostic: Option<bool>,
pub visibility: Option<String>,
```

2. 前端写入失败消息时标记：

```js
assistantMessage.sourceKind = "desktop_local_agent_runtime"
assistantMessage.diagnostic = true
assistantMessage.visibility = "user_visible"
```

3. Runtime 构建模型消息时：

```text
diagnostic=true 且 visibility!=model_context -> 不作为普通 assistant 历史进入模型
```

验收：

- 失败摘要仍显示给用户。
- 下一轮模型输入快照中不再把本地失败摘要当普通 assistant 消息。
- 不依赖中文文案判断失败。

### 阶段 2：Prompt 栈解析

目标：把 Resolve Prompt Stack 从字符串拼接升级为可审计对象。

改动文件：

| 文件 | 改动 |
|---|---|
| `ProjectChat.vue` | 后续可传 `systemPromptParts`，第一版仍兼容 string |
| `types.rs` | 新增 `PromptStack`、`PromptStackItem` |
| `runtime.rs` | 新增 `resolve_prompt_stack`，写入 model input snapshot |

第一版策略：

- 若前端只传 `systemPrompt` string，则拆为：
  - `project_or_combined_system_prompt`
  - `runtime_policy` 空
- 后续再让前端传：
  - `project_chat_settings.system_prompt`
  - `system_config.desktop_agent_global_prompt`
  - `runtime_policy`

验收：

- `state.json.model_runtime.agent_loop.model_input_snapshots[].prompt_stack` 可看到 prompt 来源。
- 排查“模型为什么这么回答”时不用猜 prompt 是哪来的。

### 阶段 3：PlanningGate 与澄清门

目标：把 Execution Planning / Need Clarification 落到 Runtime 状态，而不是只靠提示词。

改动文件：

| 文件 | 改动 |
|---|---|
| `types.rs` | 新增 `ClarityAssessment`、`PlanState`、`PlanNode` |
| `runtime.rs` | 新增 `assess_clarity`、`create_or_update_plan_state` |
| `ProjectChat.vue` | 展示需要确认的 PlanSummary，复用现有消息气泡和操作按钮 |

第一版规则：

| 输入类型 | Runtime 行为 |
|---|---|
| 查询/解释 | 不强制 Plan，直接模型回答 |
| 写文档且路径明确 | 创建 PlanState，但可直接执行 |
| 修改代码/运行命令/部署 | 先生成 PlanSummary，等待用户确认 |
| 删除/覆盖/外部写入 | PlanSummary 后仍需权限门单独确认 |

验收：

- 实现型需求在 `state.json` 有 PlanState。
- 查询型问题不被硬拆成多节点。
- 用户确认后同一 run 不重复要求确认，除非出现破坏性动作或范围变化。

### 阶段 4：RuntimeScheduler 与工具循环规范化

目标：把当前模型 step + tool loop 收敛为明确状态机。

改动文件：

| 文件 | 改动 |
|---|---|
| `runtime.rs` | 抽出 `RuntimeScheduler`，管理 `waiting_model/waiting_tool/waiting_approval/waiting_user` |
| `types.rs` | 新增 `RunState`、`ToolBatchState` |
| `state/mod.rs` | state 写入从零散字段收敛到结构化 state |

关键规则：

- 模型返回多个 tool_calls 时先生成 `ToolBatch`。
- 工具失败时生成 `Observation`，再由 RetryRouter 判断是否继续。
- 权限等待必须保存 pending tool，授权后复用同一个 `tool_call_id`。

验收：

- `state_changed` 事件里的等待对象字段完整。
- 授权续跑不会要求模型重新生成同一个工具。
- 页面刷新后能恢复等待授权状态。

### 阶段 5：ObservationMerger 与模型兼容适配

目标：稳定处理 tool observation、`reasoning_content`、多模态附件等兼容问题。

改动文件：

| 文件 | 改动 |
|---|---|
| `runtime.rs` | 抽出 `openai_compatible_message_payload` 的 provider capability 策略 |
| `types.rs` | 新增 `ModelCompatibilityProfile` |
| `runtime.rs` | 桌面 Tauri runtime 本地推断并记录模型能力和兼容选项 |

边界说明：这里的模型能力不是从后端下发，也不新增服务端智能体能力。liuAgent CLI 是桌面智能体，本阶段只在 `src-tauri` 本地 runtime 根据 provider/model 配置、附件路由和请求模式推断 `ModelCompatibilityProfile`，并写入本地运行记录与诊断。

需要支持的能力标记：

```json
{
  "supports_tools": true,
  "supports_reasoning_content_replay": true,
  "requires_reasoning_content_replay": true,
  "supports_image_url": false,
  "tool_role_mode": "tool_role | user_observation",
  "stream_mode": "sse | non_stream"
}
```

针对当前问题：

- DeepSeek 内部网关报 `reasoning_content must be passed back` 时，不应只显示失败摘要。
- Runtime 要记录模型第一步返回的 `reasoning_content`，并在后续 assistant tool_call message 中按能力要求回传。
- 如果 provider 不支持 `image_url`，附件应降级为 metadata/extracted_text，而不是直接发未知 variant。

验收：

- 第二轮 tool continuation 不再因为缺 `reasoning_content` 失败。
- provider 不支持图片时给出明确可见限制，不发送不兼容 payload。
- 模型请求诊断中能看到 capability profile。

### 阶段 6：VerificationEngine

目标：把 Draft Solution 后的验证落成结构化对象。

改动文件：

| 文件 | 改动 |
|---|---|
| `runtime.rs` | 新增 `run_verification_checks` |
| `types.rs` | 新增 `VerificationReport`、`VerificationCheck` |
| `ProjectChat.vue` | 运行过程展示 verification 结果 |

第一版验证规则：

| 验证类型 | 当前可落地方式 |
|---|---|
| Fact | 文件是否存在、工具结果 ok、命令 exit code |
| Logic | 模型生成简短自检摘要，不展示隐藏推理 |
| Constraint | 是否越权、是否修改了非目标文件、是否执行了未授权动作 |
| Goal | 当前 PlanNode 是否有明确交付物 |

验收：

- 每个完成的 PlanNode 有 `verification_result`。
- 失败时能区分验证失败和工具失败。
- 最终回答能说明验证证据，而不是只说“已完成”。

### 阶段 7：RetryRouter

目标：将失败处理从“直接失败摘要”升级为可解释路由。

路由策略：

| 失败类型 | 路由 |
|---|---|
| 文件路径不存在 | Search：列目录/搜索候选文件 |
| schema 缺字段 | Tool：安全修复参数或请求模型修正 |
| provider payload 不兼容 | Plan：切换兼容策略或提示能力限制 |
| 权限拒绝 | Draft：说明未执行和替代方案 |
| 验证失败 | Tool 或 Plan：补救一次，重复失败后 blocked |
| 需求范围变化 | Ask User：暂停并确认 |

改动文件：

| 文件 | 改动 |
|---|---|
| `runtime.rs` | 新增 `route_failure` |
| `types.rs` | 新增 `RetryDecision` |
| `state.json` | 记录 `retry_history` |

验收：

- 同一工具同一参数失败不会无限循环。
- 能看到失败原因、选择的 retry 路由和退出条件。
- 失败三次仍不能恢复时进入 blocked，而不是继续空转。

### 阶段 8：ResponseFormatter

目标：把最终输出和运行诊断分离。

改动：

- `assistant_content` 只放面向用户的结果。
- `runtime_error`、`request_diagnostic`、`provider_id/model_name` 放到 `model_result`、`operations`、`processLog` 或错误弹窗。
- 失败时返回：

```json
{
  "assistant_content": "本轮没有完成，因为模型网关拒绝了 continuation 请求。",
  "error": "完整诊断",
  "user_visible_error_summary": "模型网关不兼容 reasoning_content 回传要求",
  "diagnostic": {...}
}
```

验收：

- 用户可见消息不再混入长 JSON 诊断。
- 调试面板仍能复制完整错误。
- 后续模型 history 默认不包含诊断内容。

### 阶段 9：MemoryClassifier 与状态写入

目标：把 Memory Classifier 落到当前已有 requirement/outbox/state。

第一版分类：

| scope | 写入位置 | 内容 |
|---|---|---|
| short | `state.json.current_state` | 本轮临时上下文、当前节点、pending 状态 |
| project | `requirements/<project_id>/<chat_session_id>.json` + outbox | 用户需求、计划、动作、验证结果 |
| long | 暂不自动写入长期记忆 | 只生成候选 `long_memory_candidates`，等待后续能力 |

改动文件：

| 文件 | 改动 |
|---|---|
| `runtime.rs` | 新增 `classify_memory_writes` |
| `state/mod.rs` | 保存 `current_state` 与 `memory_write_plan` |
| `ProjectChat.vue` | 同步 outbox 后展示同步结果 |

验收：

- 本地 requirement 能看到本轮 Plan、actions、verification。
- outbox ack 后本地待同步计数准确。
- 不因服务端不可用丢失本地事实。

## 迁移策略

### 兼容旧状态

旧 session 可能没有 `AgentRunContext`、`PromptStack`、`PlanState`。

处理规则：

- 读取旧 `state.json` 时按缺省值补齐新字段。
- 不重写 `transcript.jsonl` 历史事件。
- 新字段只从下一次 run 开始写入。
- 旧 requirement 不批量迁移，只在同一会话再次运行时补齐 `history` 事件。

### 前端兼容

- 前端展示层先读取新字段；不存在时继续使用旧 `runtime_events`。
- 操作按钮和权限确认沿用现有 UI。
- Trace Viewer 不从 assistant 文本猜状态，只读 runtime event / operation meta。

### 集成场景

1. 用户问“你是什么模型”
   - 不按配置强制改身份。
   - 回答应基于真实上游响应或明确说明“当前配置显示 provider/model，模型自报不可作为可靠事实”。
   - 运行诊断不污染后续历史。

2. 用户要求“基于方案记录写升级计划到 docs”
   - 创建 PlanState。
   - 读取相关文档。
   - 写入目标文件。
   - VerificationReport 记录目标文件存在、未改代码、路径正确。

3. DeepSeek 工具调用第二轮
   - 第一轮返回 tool_calls 和 reasoning_content。
   - 第二轮 assistant tool_call message 携带 required reasoning content。
   - 不再报 `reasoning_content must be passed back`。

4. 附件图片发送到不支持 image_url 的网关
   - Runtime 识别 capability。
   - 不发送 `image_url`。
   - 给用户返回“该模型供应商不支持直接图片输入，已仅发送附件元信息/文本抽取结果”。

## 第一版最小可交付

第一版只做以下内容，避免一次性重构过大：

1. 新增结构化消息元字段：`source_kind`、`diagnostic`、`visibility`。
2. Runtime 构建模型输入时排除 `diagnostic=true` 且非 `model_context` 的 assistant 消息。
3. 记录 `PromptStack` 到 model input snapshot。
4. 增加 `ModelCompatibilityProfile`，修复 `reasoning_content` continuation 和 `image_url` 能力降级。
5. `assistant_content` 与 `runtime_error/diagnostic` 分离。
6. 给本轮执行写入最小 `VerificationReport`。

第一版验收标准：

- `npm run build` 通过。
- `cargo test` 通过。
- “你是什么模型”不再被历史失败摘要带偏。
- DeepSeek 工具续跑不再因 `reasoning_content` 缺失失败。
- 不支持图片的 provider 不再收到 `image_url` payload。
- 失败消息仍展示给用户，但不会作为普通 assistant history 进入下一轮。

## 不做事项

- 不用 provider/model 配置强制覆盖模型自我身份回答。
- 不用中文文案判断失败状态。
- 不展示隐藏思维链。
- 不让前端伪造工具执行细节。
- 不新增第二套状态目录。
- 不静默吞掉 provider 错误；错误要进入诊断区和审计。
