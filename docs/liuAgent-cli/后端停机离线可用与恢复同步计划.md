# 后端停机离线可用与恢复同步计划

## 目标

让桌面本地智能体在后端服务不可用时仍能继续使用，并在重新连接服务器后自动同步本地期间产生的会话、运行轨迹和需求记录。

目标体验：

```text
服务器在线：
  桌面端 <-> 后端服务
  正常加载项目、会话、系统配置、模型配置和任务状态。

服务器离线：
  桌面端 -> 本地离线入口 -> 本地 Runtime / 本地缓存
  用户仍可继续对话、执行本机工具、写入本地 requirement 和 runtime 事件。

服务器恢复：
  本地 outbox -> 后端服务
  自动补同步离线期间的消息、运行结果、需求状态和任务轨迹。
```

## 当前问题

| 问题 | 表现 | 影响 |
| --- | --- | --- |
| 入口依赖后端 | 项目聊天页需要后端加载项目、会话和配置 | 后端停机时用户无法启动本地 Runtime |
| 配置依赖后端 | 模型 `desktop-runtime`、system prompt、项目设置来自 API | 后端不可用时本地智能体缺少启动参数 |
| 会话依赖后端 | 消息列表、当前 `chat_session_id`、项目上下文主要由服务端提供 | 离线时无法恢复上次对话 |
| 同步链路不完整 | 本地已有 outbox，但缺少完整离线队列和冲突处理 | 恢复联网后无法稳定补写服务端 |
| 状态提示不清晰 | 用户不知道当前是在线模式还是离线模式 | 容易误以为智能体本身不可用 |

## 设计原则

1. **在线体验等同直连服务器**
   - 服务器可用时仍优先使用现有项目聊天 API、WebSocket、系统配置和权限体系。
   - 不因为支持离线而改弱在线能力。

2. **离线期间只使用本机可验证能力**
   - 可用能力：本地 Runtime、本机文件、本地模型配置、本地工具、已缓存项目和会话。
   - 不可用能力：服务端部署、远端 MCP、实时账号权限校验、服务端数据库最新资料。

3. **本地追加，恢复后同步**
   - 离线产生的事实写入本地 `outbox`。
   - 服务端确认写入成功后才能 ack 删除。
   - 不用静默覆盖服务端数据掩盖冲突。

4. **状态源收敛**
   - 本地 Runtime 状态仍使用 `.ai-employee/agent-runtime-v2/`。
   - query-MCP 状态仍使用 `.ai-employee/query-mcp/active-sessions/`、`session-history/`、`outbox/`。
   - 不新增多个平行的 current-session 或临时 fallback 状态。

## 总体架构

```text
Desktop Shell
  -> Connectivity Detector
  -> Online Project Chat Adapter
      -> 后端项目 / 会话 / 配置 / WebSocket / requirement API
  -> Offline Project Chat Adapter
      -> Local Project Cache
      -> Local Session Cache
      -> Local Model Runtime Cache
      -> Tauri liuAgent Runtime
      -> Local Outbox
  -> Sync Worker
      -> 后端恢复后补同步 outbox
      -> 成功 ack，本地保留审计和 transcript
```

## 本地缓存范围

### Project Cache

保存桌面端最近使用项目的最小可离线信息：

```json
{
  "project_id": "proj_xxx",
  "name": "项目名称",
  "workspace_path": "/path/to/workspace",
  "last_chat_session_id": "chat-session-xxx",
  "updated_at": "2026-06-27T00:00:00Z"
}
```

### Session Cache

保存项目聊天可恢复所需的消息和状态：

- `project_id`
- `chat_session_id`
- 用户消息和助手消息
- 消息附件摘要
- 当前运行状态
- 最近 requirement 记录路径
- 最近 runtime event id
- 未完成授权请求

### Runtime Config Cache

保存本地启动模型所需的配置快照：

- provider id
- model name
- base url
- 本地可用 API key 或安全引用
- temperature
- system prompt
- desktop agent global prompt

API key 不应明文写入项目工作区；优先使用系统钥匙串、Tauri secure storage 或本机用户级配置。

### Outbox

保存等待同步到服务端的事件：

```json
{
  "event_id": "evt_xxx",
  "project_id": "proj_xxx",
  "chat_session_id": "chat-session-xxx",
  "source_kind": "desktop_local_agent",
  "event_type": "requirement_record_upsert",
  "payload": {},
  "payload_hash": "sha256:...",
  "created_at": "2026-06-27T00:00:00Z",
  "sync_status": "pending"
}
```

## 阶段 1：后端可用性检测

目标：桌面端明确知道当前是在线模式、离线模式还是恢复同步中。

要做：

- 增加后端健康检查：短超时请求 `/health` 或已有轻量接口。
- 在 `ProjectChat.vue` / Desktop Shell 中维护 `backendStatus`。
- API 请求失败时不直接让页面整体不可用，而是进入离线候选状态。
- 顶部显示模式标识：在线、离线、本地运行中、同步中、同步失败。

验收：

- 后端关闭时，桌面端不会卡在空白或无限 loading。
- 用户能看到“当前离线，使用本地缓存继续”的状态。

## 阶段 2：离线入口

目标：后端不可用时，用户仍能进入本地项目聊天。

要做：

- 在桌面端提供本地模式入口：
  - 最近项目列表
  - 最近会话列表
  - 工作区路径
  - 本地模型状态
  - 新建本地会话
- 后端加载项目失败时，自动展示本地缓存项目。
- 本地入口只允许打开已经缓存或用户明确选择工作区的项目。

涉及文件：

| 文件 | 改动 |
| --- | --- |
| `web-admin/frontend/src/views/projects/ProjectChat.vue` | 增加离线模式入口和状态切换 |
| `web-admin/frontend/src/utils/native-desktop-bridge.js` | 增加本地缓存读写桥接方法 |
| `web-admin/frontend/src-tauri/src/liuagent_core/state/` | 增加项目/会话缓存读写 |

验收：

- 后端停机时可以从桌面端打开最近项目。
- 没有缓存项目时提示用户选择本机工作区，而不是显示后端错误堆栈。

## 阶段 3：本地会话缓存

目标：离线时可以恢复历史对话并继续生成。

要做：

- 在线模式下，每次消息更新后写入本地 session cache。
- 本地 Runtime 返回结果后，立即追加本地消息和 runtime events。
- 保存当前 `chat_session_id`，避免离线期间生成多个无法合并的会话。
- 恢复时优先读取：
  1. 本地 session cache
  2. `.ai-employee/agent-runtime-v2/sessions/<chat_session_id>/state.json`
  3. transcript / checkpoint
  4. query-MCP active/session-history

验收：

- 后端关闭后重新打开桌面端，可以看到最近一次本地对话。
- 上一轮如果停在等待授权，恢复后仍显示授权状态。

## 阶段 4：本地模型配置缓存

目标：离线时本地 Runtime 仍能拿到模型调用参数。

要做：

- 在线时缓存 `desktop-runtime` 的非敏感字段。
- API key 使用安全存储或用户级配置引用，不写入项目目录。
- 离线启动前校验：
  - base url 是否存在
  - model name 是否存在
  - API key 或安全引用是否可用
- 配置过期时提示用户重新联网刷新或手动选择本地模型。

验收：

- 后端关闭后，只要本机已有有效模型配置，仍可启动本地对话。
- 缺少模型配置时给出明确提示，而不是报“后端不可用”。

## 阶段 5：离线运行与本地记录

目标：离线期间所有动作都能本地落盘并可追踪。

要做：

- 本地对话继续调用 `liuagent_start_local_chat`。
- 运行轨迹写入 `.ai-employee/agent-runtime-v2/`。
- requirement 写入 `.ai-employee/requirements/<project_id>/<chat_session_id>.json`。
- 同步事实写入 `.ai-employee/query-mcp/outbox/<project_id>__<chat_session_id>.jsonl`。
- 高风险工具仍走本地授权，不因离线跳过权限。

验收：

- 离线执行后刷新桌面端，运行结果和执行轨迹仍在。
- 未同步条目保留在 outbox，服务端未确认前不会删除。

## 阶段 6：恢复后同步

目标：服务器恢复后把离线期间产生的状态补写回服务端。

同步顺序：

1. 重新验证登录态和用户权限。
2. 拉取服务端项目和会话最新状态。
3. 读取本地 outbox。
4. 按 `created_at` 和 `event_id` 顺序提交。
5. 服务端返回成功后 ack 本地 outbox。
6. 发生冲突时保留本地条目并提示用户处理。

第一批同步对象：

- 项目聊天消息摘要
- requirement record
- runtime result summary
- task tree / current node 状态
- verification result

验收：

- 后端恢复后，离线期间的对话结果能出现在服务端项目聊天中。
- 重复点击同步不会重复写入同一条事件。
- 同步失败的条目不会丢失。

## 阶段 7：同步后自动清理

目标：同步成功后自动清理已确认的本地同步缓存，避免 outbox、会话缓存和临时快照无限增长。

清理原则：

- 服务端确认成功前不删除本地待同步事实。
- 已同步 outbox 条目必须自动 ack 并从待同步队列移除。
- 缓存清理只删除可重建或已确认同步的数据，不删除审计日志和必要 transcript。
- 清理动作本身要写入本地维护日志，便于排查“为什么本地看不到旧待同步项”。

清理范围：

| 类型 | 同步成功后动作 | 保留策略 |
| --- | --- | --- |
| `query-mcp/outbox/*.jsonl` | 按 `event_id` ack 删除已同步行 | 失败、冲突、blocked 条目继续保留 |
| session cache | 标记 `sync_status=synced`，压缩旧消息快照 | 最近 N 个会话保留完整缓存，其余只保留摘要 |
| project cache | 刷新 `last_synced_at` 和最近会话指针 | 最近项目继续保留，长期未访问项目可过期 |
| runtime state | 保留最后状态和 checkpoint | 旧中间快照可按 TTL 清理 |
| transcript / audit | 不随同步自动删除 | 只允许按独立日志保留策略归档或清理 |
| provider/runtime config cache | 不因消息同步删除 | 按配置过期时间或用户操作更新 |

建议默认策略：

- outbox：同步成功立即 ack。
- session cache：保留最近 20 个会话完整缓存，超出后只保留摘要和服务端引用。
- runtime 临时快照：保留最近 7 天或最近 20 次运行。
- transcript / audit：默认保留，后续单独提供“清理本地运行日志”设置。
- conflict / failed / blocked：不自动清理，必须用户处理或下一次同步成功后再清理。

同步后的本地状态示例：

```json
{
  "chat_session_id": "chat-session-xxx",
  "sync_status": "synced",
  "last_synced_at": "2026-06-27T00:00:00Z",
  "server_message_ids": ["msg_1", "msg_2"],
  "pending_outbox_count": 0,
  "cache_retention": "full"
}
```

UI 行为：

- 顶部状态从 `待同步 3 条` 变为 `已同步`。
- 已同步后隐藏“立即同步”按钮。
- 详情面板显示“已清理 3 条同步缓存”。
- 如果部分成功，显示 `已同步 2 条，失败 1 条`，只清理成功条目。

验收：

- 同步成功的 outbox 不会在下次启动时再次出现。
- 同步失败或冲突条目仍保留，方便重试或人工处理。
- 多次离线使用后，本地缓存不会无限增长。
- 清理不会删除 transcript / audit 中用于追溯权限和工具执行的记录。

## 阶段 8：冲突与幂等

目标：避免恢复同步时重复写、错写或覆盖服务端最新状态。

规则：

- 每条 outbox 必须有稳定 `event_id` 和 `payload_hash`。
- 服务端按 `project_id + chat_session_id + event_id` 幂等处理。
- 同一 `event_id` 但 `payload_hash` 不同，标记冲突。
- 服务端会话已删除或无权限时，本地条目进入 `blocked`，不自动丢弃。
- 同步只追加事实，不静默覆盖服务端消息正文。

验收：

- 同一 outbox 重放两次，服务端只产生一条记录。
- 冲突可见，用户可以选择保留本地、创建新会话或放弃同步。

## 阶段 9：测试计划

### 单元测试

- 本地 project cache 读写。
- session cache 读写和 schema 迁移。
- outbox append / list / ack。
- payload hash 和 event id 幂等。
- 离线模型配置校验。
- 同步成功后 outbox 自动清理。
- session cache 保留上限和摘要压缩。

### 集成测试

- 后端在线：正常走原项目聊天。
- 后端离线：进入本地入口并启动 Runtime。
- 后端恢复：outbox 自动同步并 ack。
- 同步成功：已同步条目自动从待同步队列清理。
- 同步失败：outbox 保留并显示错误。
- 冲突事件：不覆盖服务端状态。

### 手工验收

1. 启动桌面端和后端，打开项目聊天。
2. 发送一条消息，确认本地 session cache 已写入。
3. 停止后端。
4. 重新打开项目聊天，确认进入离线入口。
5. 离线发送消息并执行本机工具。
6. 重启后端。
7. 确认离线消息和 requirement 被同步到服务端。
8. 重新打开桌面端，确认已同步条目不再重复出现，失败或冲突条目仍保留。

| 里程碑 | 交付物 | 验收重点 |
| --- | --- | --- |
| M1 | 后端可用性检测和离线状态 UI | 后端停机不空白、不误报 |
| M2 | 离线入口和最近项目缓存 | 可打开本地项目 |
| M3 | 本地会话缓存和恢复 | 可继续历史对话 |
| M4 | 模型配置缓存 | 离线可调用本地模型 |

| M6 | 同步后自动清理 | 已同步缓存不重复出现、不无限增长 |
| M7 | 冲突和幂等 | 不重复写、不静默覆盖 |
| M8 | 全链路测试 | 在线、离线、恢复三段闭环 |

## 与现有设计的关系

- 复用 `docs/liuAgent-cli/design/06-state-storage.md` 中的 `.ai-employee/agent-runtime-v2/`、`query-mcp/outbox/` 和 requirement 存储布局。
- 复用桌面端 Tauri `liuagent_core` 的本地 Runtime，不另起一套智能体执行器。
- 在线模式继续使用现有后端项目聊天链路。
- 离线模式只补齐入口、缓存、恢复和同步，不改变本地工具权限模型。
