# 桌面本地智能体需求理解与记录计划

## 目标

把用户的一次输入处理成后端两段式内部流程：

1. 先解析用户当前输入，明确本轮重点和需要什么上下文。
2. 再按解析结果检索项目上下文、需求链、规则、技能和历史经验。
3. 最后带上下文处理任务，并记录本轮结果。

用户感知上仍然只发送一次消息；内部可以有多次模型请求和工具检索。

## 需求记录

需求记录采用类似技能 frontmatter 的结构，但字段面向需求：

```yaml
id: req_xxx
name: requirement-title
description: 简短描述
version: 1.0.0
tags: [requirement, context, task-tree]
```

正文记录：

- `original_request`: 用户原文
- `intent_analysis`: 需求理解循环输出，忠实提取用户原文里的目标对象、目标能力、关键词和上下文检索方向
- `related_context`: 命中的项目上下文、规则、技能、历史经验
- `contextual_plan`: 任务处理循环输出，包含处理目标、执行边界和计划
- `model_input_snapshots`: 真实发送给大模型的输入快照，按本轮内部循环分别记录
- `actions_taken`: 本轮实际做了什么
- `current_state_delta`: 本轮对当前产物状态的增量更新

同时维护会话级 `current_state`：

- 当前已经完成了什么
- 当前产物有哪些能力和限制
- 最近关键文件、页面、接口或任务树节点
- 下一轮需求需要优先继承的状态

## 处理流程

```text
用户输入
  -> 保存 original_request
  -> 创建或更新 requirement 记录
  -> 需求理解循环：生成 intent_analysis，并装配任务处理用上下文包
  -> 记录需求理解循环的 model_input_snapshot
  -> 任务处理循环：生成 contextual_plan，并完成回答、代码、文档或工具操作
  -> 记录任务处理循环的 model_input_snapshot
  -> 记录 actions_taken 和 current_state_delta
  -> 更新 current_state、任务树和验证信息
```

## 两个内部循环

用户只发送一次消息，后端内部拆成两个循环。

```text
用户输入
  -> 需求理解循环
  -> 任务处理循环
  -> 返回用户
```

| 段落 | 名称 | 做什么 | 边界 |
| --- | --- | --- | --- |
| 第一段 | 需求理解循环 | 先只理解用户当前需求，再按本轮重点检索候选上下文，最后生成任务处理用上下文包 | 只准备上下文，不执行任务 |
| 第二段 | 任务处理循环 | 带着上下文包真正处理任务，回答问题或调用工具完成修改 | 只围绕本轮重点处理 |

### 第一段内部循环：需求理解循环

职责是理解本轮需求，并准备任务处理需要的上下文。

```text
理解循环输入：
- 用户当前原文

理解循环步骤：
1. 只基于用户当前原文解析本轮重点和任务类型
2. 忠实提取原文中的目标对象、目标能力和关键词，不提前展开实现细节
3. 根据本轮重点生成上下文检索条件
4. 检索候选上下文，例如 current_state、最近需求链摘要、相关文件、规则或历史记录
5. 基于候选上下文标记关系：related / unrelated / uncertain
6. 生成任务处理用上下文包

理解循环输出：
- 本轮重点
- 当前需求结构化理解
- 需求关系判断
- 需要继承的历史需求
- 任务处理用上下文包
```

### 第二段内部循环：任务处理循环

职责是带着上下文包真正处理事情。

```text
任务循环输入：
- 用户当前原文
- 任务处理用上下文包
- 需求理解循环输出

任务循环步骤：
while 未完成 && 未超限:
  1. 模型确认本轮处理目标和边界
  2. 模型决定下一步
  3. 调用工具或生成回答
  4. 接收工具结果
  5. 继续处理，直到完成或需要用户确认

任务循环输出：
- 给用户的最终回答
- 本轮实际动作
- 修改文件或调用工具记录
- 验证结果
- current_state 增量更新
```

结果记录发生在任务处理循环结束后，用于更新 requirement、任务树和 current_state。

## 任务处理用上下文包

任务处理用上下文包必须简洁，只告诉 AI “当前要做什么”和“之前做过什么”。不要把历史需求原文、完整聊天、完整执行日志全部塞进去。

简洁不等于压扁需求。简化描述只是一句话标题，不能替代结构化理解。结构化理解要保留用户原文里的关键对象、动作和约束；没有从原文或上下文得到的信息，只能作为检索方向或待上下文确认项，不能当成已经确定的需求内容。

上下文包建议形态：

```text
当前需求简化描述：
- <把用户本轮长需求压缩成一句清楚目标>

当前需求结构化理解：
- 目标对象：<用户原文里的对象>
- 目标能力：<用户原文里的能力或动作>
- 关键词：<用于检索上下文的关键词>
- 上下文检索方向：<需要从项目里确认的页面、接口、文件、规则或历史状态>

相关历史工作摘要：
- <之前做过什么，简短描述>
- <当前产物已有能力或状态>

本轮要做内容：
- 要做：<本轮重点>
```

示例：

```text
用户历史需求：创建登录注册页面
AI 理解后的历史摘要：已创建登录注册页面。

用户当前需求：给注册页面添加短信验证码逻辑
当前需求简化描述：在已有注册页面上添加短信验证码逻辑。

当前需求结构化理解：
- 目标对象：注册页面
- 目标能力：手机短信验证码逻辑
- 关键词：注册页面、手机、短信验证码、注册流程
- 上下文检索方向：现有注册页面、注册提交流程、项目里是否已有短信验证码接口或约定

传给任务处理循环的上下文包：
- 之前已创建登录注册页面。
- 本轮要在注册页面上添加手机短信验证码逻辑，具体实现方式以检索到的现有注册流程和短信接口约定为准。
```

也就是说，历史记录进入任务处理循环前必须先变成简短工作摘要。它的作用是说明“之前做过什么”，不是让 AI 重读所有历史。

## 需求关系判断

同一窗口里，不是所有需求都天然相关。新输入需要完成关系判断，但不能早于当前原文理解：

关系判断不能发生在用户原文解析之前。必须先独立理解用户当前需求，再按本轮重点检索候选上下文，最后基于候选上下文判断关系。

```json
{
  "relationship": "related | unrelated | uncertain",
  "related_objects": ["页面", "模块", "文件", "功能"],
  "reason": "为什么相关或不相关",
  "context_strategy": "continue_chain | new_chain | ask_for_clarification"
}
```

示例：

- `创建登录页` -> `给登录页添加手机短信验证码逻辑`：`related`，继续当前需求链。
- `创建登录页` -> `做订单列表页`：`unrelated`，在同一窗口中新建需求链。
- `创建登录页` -> `按钮改成蓝色`：`uncertain` 或 `likely_related`，优先结合最近需求链理解，必要时确认。

## 上下文策略

长对话不能把所有需求都发给模型。模型上下文只使用处理后的工作上下文：

```text
current_state
  + 最近 1-3 条相关需求摘要
  + 按本轮重点检索出的少量历史记录
  + 本次用户原文
```

真实聊天记录和模型上下文分开：

- `raw_conversation_log`: 完整保存真实聊天，供回放、审计和排查。
- `model_context`: 从需求记录、任务树、当前状态和相关历史里加工出的结构化上下文。

用户问“最近做什么了”时，这本身是历史回顾需求，应按时间检索最近需求记录，而不是按功能相关性检索。

## 模型输入快照

服务器需求记录和真实发送给大模型的数据不是同一种东西。

服务器记录偏向需求摘要、任务树、工作事实和项目记忆，用来跨窗口检索、恢复任务和回忆最近工作。真实发送给大模型的数据要在本地需求记录里单独保存为 `model_input_snapshots`。

`model_input_snapshots` 记录每次内部模型请求实际使用的上下文。需求理解循环和任务处理循环都可能调用模型，所以要分别记录，不能只用一个字段覆盖。

建议结构：

```json
{
  "model_input_snapshots": [
    {
      "turn_id": "turn_xxx",
      "message_id": "用户消息ID",
      "loop": "requirement_understanding | task_processing",
      "prompt_version": "xxx",
      "model_name": "xxx",
      "input_summary": "本次发给模型的简短说明",
      "context_package": {
        "current_request_summary": "",
        "related_work_summary": [],
        "current_state_used": "",
        "task_focus": ""
      },
      "messages_hash": "xxx",
      "messages_redacted": [],
      "created_at": "2026-06-24T00:00:00Z"
    }
  ]
}
```

字段含义：

- `loop`: 标记这条输入快照属于需求理解循环还是任务处理循环。
- `input_summary`: 用一句话说明这次模型请求解决什么问题。
- `context_package`: 记录进入模型前加工后的上下文包。
- `messages_hash`: 记录最终 messages 的 hash，用于确认是否和实际请求一致。
- `messages_redacted`: 可选保存脱敏后的最终 messages，便于调试和复盘。

完整 messages 默认只保存在本地，不默认同步到服务器。服务器只同步摘要、索引字段、`turn_id`、`message_id` 和 `messages_hash`。

检索规则：

```text
同窗口继续处理：
  读取本地 requirement、current_state 和 model_input_snapshots

新窗口或跨设备恢复：
  读取服务器需求摘要、任务树、工作事实和项目记忆

需要复盘当时模型为什么这样理解：
  按 chat_session_id + turn_id 回查本地 model_input_snapshots
```

进入下一次模型请求前，仍然要把检索结果重新压缩成任务处理用上下文包，不能把历史快照原样全部塞给模型。

## 提示词约束

提示词必须围绕两个内部循环设计。不要拆成碎片化提示词，也不要使用次数式表述。

对用户展示时仍然是一轮请求、一次处理过程；循环细节只进调试日志和审计记录。

## 提示词模板

### 需求理解循环提示词

```text
你负责完成本轮用户输入的需求理解循环。

输入：
- 用户当前原文

要求：
- 只理解和准备上下文，不回答最终结果，不改代码，不调用高风险工具。
- 先只基于用户当前原文提取本轮重点、任务类型和关联对象。
- 把用户当前原文压缩成一句简洁的当前需求描述。
- 简化描述不能替代结构化理解；结构化理解必须保留原文里的目标对象、目标能力、关键词和约束。
- 不把原文和候选上下文都没有提供的实现细节写成本轮需求。
- 根据本轮重点生成上下文检索条件。
- 拿到候选上下文后，再判断本轮需求与当前需求链的关系：related / unrelated / uncertain。
- 根据关系判断生成任务处理用上下文包：
  - related：继承 current_state 和相关需求链的简短工作摘要。
  - unrelated：新建需求链，只保留必要项目基础上下文。
  - uncertain：保留候选上下文但降低置信度，必要时要求确认。
  - history_recall：按时间使用最近需求记录。
- 不全量注入聊天历史、工具日志、长 diff 或无关需求。

输出 JSON：
{
  "current_request_summary": "",
  "current_request_understanding": {
    "target_object": "",
    "target_capability": "",
    "keywords": [],
    "constraints": [],
    "context_lookup": []
  },
  "focus": "",
  "task_type": "new_feature | follow_up | modification | history_recall | question | other",
  "relationship": "related | unrelated | uncertain",
  "related_objects": [],
  "context_strategy": "continue_chain | new_chain | retrieve_recent | ask_for_clarification",
  "context_queries": [],
  "context_include": [],
  "needs_user_confirmation": false,
  "reason": ""
}
```

### 任务处理循环提示词

```text
你负责完成本轮用户输入的任务处理循环。

输入：
- 用户当前原文
- 需求理解循环输出
- 任务处理用上下文包：当前需求简化描述、相关历史工作摘要、本轮要做内容

要求：
- 基于上下文包确认本轮真正要处理的目标和执行边界。
- 只围绕本轮重点处理任务，不重新全量理解历史，不主动扩大范围。
- 历史摘要只用于了解之前做过什么，不得把历史内容当成本轮全部任务重新执行。
- 可以回答问题，也可以在允许范围内调用工具完成代码、文档或任务树操作。
- 涉及删除、覆盖、部署、发布、外部写入或高风险操作时必须暂停确认。
- 结束时输出给用户的结果摘要，并给结果记录提供结构化信息。

输出 JSON：
{
  "final_intent": "",
  "execution_focus": "",
  "context_used": [],
  "actions_taken": [],
  "changed_files": [],
  "verification": [],
  "current_state_delta": [],
  "next_current_state": "",
  "user_response": ""
}
```

## 落点

- 本地记录：`.ai-employee/requirements/<project_id>/<chat_session_id>.json`
- 会话状态：`.ai-employee/query-mcp/active-sessions/<chat_session_id>.json`
- 历史索引：`.ai-employee/query-mcp/session-history/<project_id>__<chat_session_id>.json`
- 服务端同步：复用现有 query MCP 工作事实、任务树和项目记忆能力
- 模型输入快照：写入本地 requirement 的 `model_input_snapshots`

## 存储与检索来源

需求记录采用本地优先、服务端同步。

```text
本轮执行中：
  优先读写本地 requirement、current_state 和 model_input_snapshots

本轮结束后：
  同步关键摘要、索引和 hash 到服务端任务树、工作事实和项目记忆

后续同窗口继续：
  优先读取本地 current_state、本地 requirement 和模型输入快照

新窗口、跨设备、历史回顾：
  优先从服务端检索已同步的需求摘要、任务树和项目记忆
```

本地记录负责支撑桌面智能体当前会话的连续执行，服务端记录负责支撑跨窗口、跨设备、历史回顾和项目级检索。

检索时不要把本地完整聊天或服务端完整历史全部塞给模型。无论来源是本地还是服务端，都要先转成简短工作摘要，再进入任务处理用上下文包。

## 关键约束

- 同一条用户需求只能有一个稳定 `requirement_id`。
- 重试、续跑、上下文补充都必须幂等更新同一条记录。
- 上下文检索必须记录命中来源和原因，不能全量注入无关规则。
- 任务处理循环要明确说明相比需求理解循环新增、修正或确认了什么。
- 进入代码修改、部署、删除、发布、共享知识库写入前仍需用户确认。

## 验证

- 发送一条新需求后，本地 requirement 文件立即生成。
- 文件中能看到用户原文、需求理解循环输出、上下文命中和任务处理循环输出。
- 文件中能看到本轮内部模型请求对应的 `model_input_snapshots`。
- 同一需求重复执行不会生成第二条记录。
- 换一个完全不同需求时，命中的上下文应明显不同。
- 同一窗口内相关需求会续写当前需求链，不相关需求会新建需求链。
- 中断后恢复时，能通过 `chat_session_id` 找回同一条记录和任务树。

## 实施顺序

1. 先补需求记录 schema 和写入入口。
2. 再接入上下文检索结果写回。
3. 最后把任务处理循环和任务树状态联动。
