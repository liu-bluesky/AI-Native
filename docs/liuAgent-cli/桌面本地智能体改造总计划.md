# 桌面本地智能体改造总计划

## 目标

把当前桌面本地智能体从“粗粒度工具执行器”升级为“AI 自主决策 + Runner 提供可靠工具和可见执行轨迹”的本地 Agent Runtime。

核心目标：

- AI 是大脑：任务理解、方案选择、工具编排、失败后的下一步，由模型自主决定。
- Runner 是手和安全层：提供文件、命令、MCP、浏览器、权限、状态、审计等能力，不替 AI 做业务决策。
- 用户能看到执行过程：展示进度播报、工具调用、文件修改、命令输出、失败恢复，不展示隐藏思维链。
- 工具调用可恢复：工具参数错误、重复失败、会话恢复、日志重复等问题要由 Runtime 处理，而不是直接把失败抛给用户。

## 当前问题

| 问题 | 表现 | 影响 |
|---|---|---|
| 执行轨迹太粗 | 只显示“正在本机创建会话并执行本地工具” | 用户看不到 AI 到底做了什么 |
| 进度播报不稳定 | 缺少类似 Codex / Claude 的 `Explored`、`Ran`、`Edited`、`Computing` | 长任务体验像黑盒 |
| 工具失败反馈弱 | 例如 `write_file` 缺少 `path` 后反复失败 | 模型无法根据错误快速修正 |
| 工具契约太薄 | schema 有 required，但缺少示例、纠错提示和高阶工具 | 模型容易生成不完整参数 |
| 计划选择未结构化 | 用户选择 “login.html 风格 + register.html” 后没有固化成执行状态 | 后续步骤只能靠聊天历史猜 |
| 前端日志重复 | 失败恢复后重复展示 read/write 事件 | 用户难以判断真实执行链路 |
| 工作流过度固定风险 | 强制“计划先行”可能压制模型自主工具选择 | AI 的推理和行动空间被缩小 |

## 设计原则

1. **不做隐藏思维链展示**
   - 不展示模型内部 chain-of-thought。
   - 只展示模型主动输出的可见说明、计划、阶段性发现、下一步动作。

2. **固定事件类型，动态事件内容**
   - `event.type`、`event.status`、`event_id` 是固定协议。
   - `summary`、`next_action`、`detail` 由 AI 或 Runner 动态生成。

3. **模型决定怎么做，Runtime 负责让它做得稳**
   - 不把业务路径写死在 Runner。
   - Runtime 只负责工具契约、权限、安全、重试、去重、状态持久化。

4. **失败先修复，再暂停**
   - schema 错误、缺字段、参数名轻微错误，先尝试安全修复或给模型明确纠错指令。
   - 只有相同错误多次重复且无法修复时，才暂停自动循环。

5. **前端只渲染真实事件**
   - 前端不伪造 `Ran`、`Edited`、`Explored`。
   - 这些内容必须来自 Runner、工具执行层或模型可见进度事件。

## 总体架构

```text
用户输入
  -> ProjectChat.vue / Desktop Adapter
  -> Local Agent Runtime
  -> Model Adapter
  -> 模型生成可见说明 + tool_calls
  -> Runtime 校验 / 修复 / 权限门
  -> 本地工具执行
  -> AgentTraceEvent 流
  -> 前端 Trace Viewer 渲染
  -> JSONL 持久化用于恢复
```

## 阶段 1：先修复会卡死体验的问题

目标：让当前智能体不再因为简单工具参数错误反复失败。

要改：

- `write_file` 缺少 `path` 时，从最近用户消息中提取目标文件名，例如 `register.html`。
- `tool.schema_invalid` 失败反馈改成模型能理解的纠错说明。
- 工具定义补充必填字段说明和 JSON 示例。
- 重复失败检测保留，但在暂停前先尝试一次安全修复。

涉及文件：

| 文件 | 改动 |
|---|---|
| `web-admin/frontend/src-tauri/src/liuagent_core/runtime.rs` | 工具参数修复、失败恢复提示、重复失败处理 |
| `web-admin/frontend/src-tauri/src/liuagent_core/definitions.rs` | 强化工具 schema 描述 |
| `web-admin/frontend/src-tauri/src/liuagent_core/tools/file.rs` | 保持真实工具校验，不绕过必填字段 |

验收：

- 用户说 `浅色渐变商务风。register.html`，模型漏传 `path` 时 Runtime 能补全。
- 同类失败不再无限重复。
- 测试覆盖 `register.html` 目标文件推断。

## 阶段 2：建立 Agent Trace 事件协议

目标：让桌面智能体输出和 Codex / Claude 类似的可见执行转录。

第一批必须支持的事件：

| 事件 | 用途 |
|---|---|
| `run_started` | 本轮运行开始 |
| `progress_update` | AI 生成的阶段性播报 |
| `model_call_started` | 模型开始计算 |
| `model_call_finished` | 模型返回内容或工具调用 |
| `tool_call_started` | 工具开始执行 |
| `tool_call_finished` | 工具执行完成 |
| `tool_call_failed` | 工具执行失败 |
| `file_read_finished` | 文件读取完成 |
| `file_diff_generated` | 生成文件 diff 摘要 |
| `file_written` | 文件写入完成 |
| `command_started` | 命令开始执行 |
| `command_output_chunk` | 命令输出片段 |
| `command_finished` | 命令完成 |
| `permission_required` | 需要用户授权 |
| `schema_repaired` | Runtime 自动修复工具参数 |
| `run_finished` | 本轮运行完成 |

涉及文件：

| 文件 | 改动 |
|---|---|
| `web-admin/frontend/src-tauri/src/liuagent_core/adapters/protocol.rs` | 定义 `AgentTraceEvent` |
| `web-admin/frontend/src-tauri/src/liuagent_core/runtime.rs` | 在模型循环和工具循环中发事件 |
| `web-admin/frontend/src-tauri/src/liuagent_core/types.rs` | 补充事件结构体和状态枚举 |

验收：

- 前端能显示 `Computing...`、`Read(...)`、`Edit(...)`、`Bash(...)`、`Result`。
- 每个事件有稳定 `event_id`，刷新或恢复后不重复。

## 阶段 3：AI 进度播报

目标：让“我接下来要做什么”由 AI 生成，而不是写死固定文本。

设计：

```json
{
  "type": "progress_update",
  "status": "running",
  "summary": "我已经确认目标文件是 register.html，参考风格来自 login.html。",
  "next_action": "接下来读取 login.html 的布局和样式，再创建 register.html。",
  "metadata": {
    "target_file": "register.html",
    "source_style_file": "login.html"
  }
}
```

要求：

- 进度播报是可展示内容，不是隐藏思维链。
- 模型每轮可选择输出，也可以由 Runtime 在关键节点请求模型生成简短播报。
- 前端只展示，不改写核心含义。

涉及文件：

| 文件 | 改动 |
|---|---|
| `web-admin/frontend/src/views/projects/ProjectChat.vue` | 展示 AI 进度播报 |
| `web-admin/frontend/src-tauri/src/liuagent_core/runtime.rs` | 捕获模型可见文本并转成 `progress_update` |
| `web-admin/frontend/src-tauri/src/liuagent_core/adapters/protocol.rs` | 定义 `ProgressUpdatePayload` |

验收：

- 运行中能看到 AI 自己写的阶段性说明。
- 不出现“这是我的思考过程”这类隐藏推理文本。

## 阶段 4：前端 Trace Viewer

目标：把 Runtime 事件渲染成用户能读懂的执行过程。

显示映射：

| Trace 事件 | 前端展示 |
|---|---|
| `model_call_started` | `Computing...` |
| `file_searched` / `file_read_finished` | `Explored` / `Read(...)` |
| `command_started` | `Ran <command>` |
| `file_diff_generated` / `file_written` | `Edited <file> (+x -y)` |
| `mcp_tool_started` | `Called <tool>` |
| `plan_updated` | `Updated Plan` |
| `tool_call_finished` | `Result` |
| `output_collapsed` | `+65 lines` 折叠提示 |

必须实现：

- 按 `event_id` 去重。
- 同一工具调用按 span 分组。
- 长输出折叠。
- running / completed / failed / blocked 状态可见。
- 失败恢复后不重复展示旧日志。

涉及文件：

| 文件 | 改动 |
|---|---|
| `web-admin/frontend/src/views/projects/ProjectChat.vue` | Trace Viewer 渲染 |
| `web-admin/frontend/src/utils/native-desktop-bridge.js` | 原生事件订阅和去重 |

验收：

- 用户能从界面看出 AI 做了哪一步、调用了哪个工具、结果是什么。
- `write_file missing path` 这类错误能显示具体失败和修复动作。

## 阶段 5：结构化保存用户选择

目标：用户选择方案后，不只存在聊天文本里，而是成为稳定执行状态。

示例：

```json
{
  "task": "create_register_page",
  "selected_path": "A",
  "source_style_file": "login.html",
  "target_file": "register.html",
  "visual_style": "浅色渐变商务风（南京嘉华运营平台）"
}
```

要改：

- 从用户补充中提取目标文件、风格来源、所选路径。
- 注入下一轮模型上下文。
- 作为工具参数修复和失败恢复的主要依据。

验收：

- 用户说“这个风格，register.html”，后续所有写文件动作都能稳定知道目标文件。
- 恢复会话后仍能读取同一结构化状态。

## 阶段 6：高阶文件工具

目标：不要强迫模型每次完整生成大文件。

新增工具建议：

| 工具 | 用途 |
|---|---|
| `copy_file` | 复制现有页面作为模板 |
| `replace_in_file` | 局部替换文件内容 |
| `create_file_from_template` | 基于模板和指令生成新文件 |
| `read_file_outline` | 提取文件结构摘要，减少大文件上下文压力 |
| `apply_patch_checked` | 带路径校验和 diff 摘要的补丁工具 |

验收：

- 创建 `register.html` 这类页面时，可走 `copy_file(login.html, register.html)` + 局部替换。
- 文件修改成功率高于整文件 `write_file`。

## 阶段 7：本地 Trace 持久化与恢复

目标：刷新页面、恢复失败会话时，执行轨迹不丢、不重、不乱。

建议路径：

```text
.ai-employee/agent-runtime-v2/task-runs/<run_id>/trace.jsonl
.ai-employee/agent-runtime-v2/task-runs/<run_id>/state.json
```

要求：

- 每行一个完整 `AgentTraceEvent`。
- 用 `event_id` 去重。
- 恢复时先 replay JSONL，再继续追加。
- 旧失败会话只能恢复一次，不重复注入同一批历史工具结果。

验收：

- 页面刷新后还能看到之前执行过的工具调用。
- 失败恢复不会重复出现三四遍相同 `read_file` / `write_file`。

## 阶段 8：测试与验收矩阵

| 测试 | 目的 |
|---|---|
| Rust 单测：工具参数修复 | 验证 `register.html` 路径推断 |
| Rust 单测：事件序列 | 验证模型调用和工具调用都产生 Trace |
| Rust 单测：JSONL replay | 验证恢复不重复 |
| 前端测试：事件映射 | 验证 `Ran`、`Edited`、`Read` 展示 |
| 前端测试：折叠输出 | 验证长命令输出不会撑爆 UI |
| 集成测试：创建注册页 | 验证从用户选择到文件创建完整闭环 |
| 集成测试：工具失败修复 | 验证 `schema_invalid` 能纠错或明确暂停 |

## 优先级

| 优先级 | 内容 | 原因 |
|---|---|---|
| P0 | 工具失败修复、schema 纠错、失败反馈 | 直接影响任务能不能完成 |
| P0 | Agent Trace 最小事件集 | 直接影响用户是否能看懂执行过程 |
| P1 | AI 进度播报 | 接近 Codex / Claude 体验 |
| P1 | 前端 Trace Viewer 去重和折叠 | 解决日志重复和输出噪音 |
| P2 | 结构化用户选择 | 提高多轮任务稳定性 |
| P2 | 高阶文件工具 | 提高复杂文件改动成功率 |
| P3 | 完整 67 类事件协议 | 后续扩展 CLI / Web / Desktop 一致性 |

## 不做的事

- 不展示隐藏思维链。
- 不把业务流程全部写死成固定步骤。
- 不让前端伪造工具执行细节。
- 不通过静默降级掩盖 Runtime 状态不一致。
- 不绕过权限门执行本机命令或写文件。

## 最小可交付版本

第一版只要求做到：

1. 模型工具调用前后都有可见 Trace。
2. AI 能输出可见进度播报。
3. `write_file` 缺 `path` 这类简单错误能修复或明确提示。
4. 前端能展示 `Computing`、`Read`、`Edit`、`Bash`、`Result`。
5. 失败恢复日志不重复。
6. 创建 `register.html` 的案例能从选择方案到写入文件跑通。

完成这 6 点后，再扩展完整事件协议、高阶工具和跨端一致性。
