# 桌面智能体附件上传与模型适配方案

## 目标

让桌面智能体在项目对话中支持任意文件上传，并根据不同大模型的能力决定附件如何进入模型上下文。

核心结论：

- 上传层不限制文件类型。
- 系统自己的 `attachment_id` 只用于附件管理，不能直接让模型理解文件。
- 模型要理解附件，必须拿到模型可用的输入：模型侧 `file_id`、图片输入、base64、可访问 URL、解析后的文本、OCR、转写内容、检索片段或结构化摘要。
- 附件处理不能只按文件后缀决定，必须按“模型能力 + 附件内容 + 用户问题”共同决策。

## 总体链路

```text
用户输入文本 + 附件
  -> 桌面端一次发送
  -> 系统保存附件，生成 attachment_id
  -> 识别当前模型能力
  -> 选择附件处理策略
  -> 必要时上传到模型厂商，拿到 provider_file_id
  -> 必要时本地解析 / OCR / 转写 / 分块 / 检索
  -> 组装模型请求
  -> 模型回答
```

用户体验上是“一次发送”。实现上可以分为附件上传、解析、模型请求三步，但必须绑定到同一条用户消息。

## 模型能力分层

不同模型对附件的支持不同，不能用一套固定逻辑处理所有模型。

| 模型能力 | 典型能力 | 附件处理方式 |
|---|---|---|
| 原生文件输入 | 支持 `file_id`、`input_file`、文件 URL、base64 文件 | 上传到模型厂商或传模型支持的文件输入 |
| 原生图片视觉 | 支持 `input_image`、图片 URL、base64 图片、图片 `file_id` | 图片直接作为视觉输入，必要时补 OCR |
| 文本模型 | 只支持文本 prompt | 系统侧先解析文件，再把文本片段拼进 prompt |
| 检索增强模型 | 支持向量库、文件检索、工具检索 | 文件入库分块，先检索，再把命中片段给模型 |
| 工具调用模型 | 支持 tool call，但不直接吃文件 | 模型调用读取、解析、搜索工具获取附件内容 |
| 本地/受限模型 | 能力不稳定或只支持局部格式 | 走本地解析器和降级策略，尽量不给原始二进制 |

## OpenAI 适配示例

如果使用 OpenAI Responses API，附件可以按能力进入模型：

- 普通文件：上传到 Files API，拿到 `file_id`，请求里用 `input_file` 引用。
- 图片：用 `input_image`，可以传图片 URL、base64，或模型侧文件 `file_id`。
- 文本模型或不适合直接输入的文件：先由系统侧解析，再把内容作为 `input_text`。

示例结构：

```json
{
  "input": [
    {
      "role": "user",
      "content": [
        {
          "type": "input_text",
          "text": "用户备注：请分析这个文件里的问题"
        },
        {
          "type": "input_file",
          "file_id": "file_xxx"
        }
      ]
    }
  ]
}
```

图片示例：

```json
{
  "input": [
    {
      "role": "user",
      "content": [
        {
          "type": "input_text",
          "text": "用户备注：看这张截图，说明页面哪里异常"
        },
        {
          "type": "input_image",
          "file_id": "file_xxx"
        }
      ]
    }
  ]
}
```

这只是 OpenAI 的适配方式。系统设计上要抽象成 `ModelAttachmentAdapter`，不要把业务逻辑写死到某一家模型 API。

## 附件路由决策

每条带附件消息都先走附件路由。

```text
attachment_id
  -> 获取附件元数据
  -> 读取当前模型能力
  -> 判断是否支持原生文件输入
  -> 判断是否支持图片视觉输入
  -> 判断是否需要系统侧解析
  -> 判断是否需要检索
  -> 生成 model_payload_parts
```

### 决策优先级

1. 如果模型支持原生文件输入，且该文件适合直传，优先上传到模型厂商。
2. 如果是图片且模型支持视觉输入，优先走图片输入。
3. 如果模型不支持文件或视觉输入，系统侧先解析成文本。
4. 如果文件很大，先分块入库，检索相关片段。
5. 如果文件无法解析，不伪装理解，只传元数据和失败原因。

## 系统附件对象

系统内部至少保存两类 id：

```json
{
  "attachment_id": "att_local_001",
  "message_id": "msg_001",
  "filename": "example.pdf",
  "mime_type": "application/pdf",
  "size": 123456,
  "sha256": "xxx",
  "local_path": "/local/path/example.pdf",
  "provider_files": [
    {
      "provider": "openai",
      "model_family": "responses",
      "provider_file_id": "file_xxx",
      "purpose": "user_data"
    }
  ],
  "derived_artifacts": {
    "text": "extracted_text_id",
    "ocr": "ocr_result_id",
    "transcript": "transcript_id",
    "chunks": "chunk_index_id"
  }
}
```

说明：

- `attachment_id` 是系统自己的附件 id。
- `provider_file_id` 是模型厂商上传接口返回的 id。
- 调模型时不能只传 `attachment_id`。
- 一个附件可以对应多个厂商的 `provider_file_id`。

## ModelAttachmentAdapter

建议为不同模型实现统一适配接口。

```ts
interface ModelAttachmentAdapter {
  provider: string
  supportsFileInput(model: string): boolean
  supportsImageInput(model: string): boolean
  supportsAudioInput(model: string): boolean
  supportsRetrieval(model: string): boolean
  uploadFile(input: LocalAttachment): Promise<ProviderFileRef>
  buildPayload(parts: AttachmentContextPart[]): ModelPayloadPart[]
}
```

适配器负责：

- 查询模型能力
- 上传到模型厂商
- 复用已有 `provider_file_id`
- 把附件转成该模型 API 需要的请求结构
- 在不支持原生文件时返回系统侧解析内容

## Prompt 拼接规则

有附件时，不要只把用户文本直接发给模型。应拼成稳定结构：

```text
用户原始问题：
<用户输入>

附件上下文：
1. attachment_id: att_local_001
   filename: example.pdf
   provider: openai
   model_input: input_file:file_xxx
   status: ready
   summary: <附件摘要>

可读内容：
<如果模型不能直接读取文件，则放解析文本 / OCR / 转写 / 检索片段>

任务：
请基于用户问题和附件内容回答。不要声称读取了不可读的附件。
```

规则：

- 用户备注必须放在最前。
- 附件按发送顺序编号。
- 同一条用户消息里的文本和附件必须绑定。
- 对模型原生文件输入，要在上下文里说明文件已经作为模型输入提供。
- 对系统侧解析内容，要说明内容来源于哪个附件。
- 解析失败时，要把失败原因写进上下文，禁止静默忽略。

## 不同模型下的同一附件示例

用户发送：

```text
请总结这个 PDF，并指出截图里的错误。
附件：
1. report.pdf
2. screenshot.png
```

### 支持文件和图片输入的模型

```text
input_text: 用户问题
input_file: report.pdf 的 provider_file_id
input_image: screenshot.png 的 provider_file_id
```

### 只支持图片输入的模型

```text
input_text: 用户问题 + report.pdf 的解析摘要和片段
input_image: screenshot.png
```

### 只支持文本的模型

```text
input_text:
- 用户问题
- report.pdf 解析文本片段
- screenshot.png OCR 结果
- screenshot.png 图片摘要
```

### 支持检索的模型

```text
input_text: 用户问题
retrieval_context:
- report.pdf 命中的段落
- screenshot.png OCR 命中的文字
```

## 失败与降级

附件处理失败时必须显式反馈，不要假装模型看过文件。

常见状态：

- `uploaded`: 系统已保存附件
- `provider_uploaded`: 已上传到模型厂商
- `parsed`: 已完成系统侧解析
- `indexed`: 已完成分块和检索索引
- `model_ready`: 可进入当前模型请求
- `unsupported_for_model`: 当前模型不能直接读取该附件
- `parse_failed`: 系统侧解析失败
- `provider_upload_failed`: 模型厂商上传失败

降级顺序：

1. 原生文件输入失败，尝试系统侧解析。
2. 图片视觉输入失败，尝试 OCR。
3. 大文件直传失败，尝试分块检索。
4. 完全无法解析时，只保留元数据，并明确告诉用户。

## 验收标准

1. 任意文件都可以作为附件提交，不因后缀被上传层拒绝。
2. 模型请求里不能只出现系统自己的 `attachment_id`。
3. 支持文件输入的模型能收到 `provider_file_id` 或等价文件输入。
4. 支持视觉的模型能收到图片输入。
5. 只支持文本的模型能收到解析后的文本、OCR、转写或检索片段。
6. 多附件不会串联错位，用户备注和附件保持同一消息绑定。
7. 解析或上传失败时，用户能看到明确原因，模型上下文里也有失败状态。

## 结论

这套方案的核心不是“把所有文件无脑丢给大模型”，而是：

- 上传层允许任何文件。
- 附件管理层保存系统自己的 `attachment_id`。
- 模型适配层按不同模型能力决定是否上传到模型厂商。
- 上下文组装层保证模型拿到它能理解的输入。

只有做到这四层，桌面智能体才能稳定支持文件上传和多模态理解。
