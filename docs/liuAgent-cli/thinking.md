• 对，你这个判断是对的：不能用“时间到了”直接判断失败。

正确设计应该是加一个“任务状态”，让系统先记录事实，再让 AI 根据事实判断。

不是：
超时 = 失败

而是：
长时间没返回 = 仍在运行 / 暂无信号 / 需要判断

应该分成几种状态：

running 正在执行
no_signal 很久没新消息，但不代表失败
stalled 可能卡住，需要 AI 或用户判断
succeeded 有明确完成证据
failed 有明确失败证据
cancelled 用户取消

判断成功不能靠时间，应该靠“完成证据”：

命令退出码是 0
文件真的生成了
测试真的通过了
部署接口返回成功
模型返回了最终 answer
AI 检查结果后确认满足目标

判断失败也要靠“失败证据”：

命令退出码非 0
接口明确报错
模型连接断开
权限被拒绝
用户取消
结果缺失且重试后仍不满足目标

如果只是：

模型思考很久
网络慢
电脑卡
命令 20 分钟没输出

这只能叫：

no_signal / still_running

不能直接叫失败。

所以更合理的机制是：

每个任务节点有状态
每个工具执行有状态
每隔一段时间写 heartbeat
前端显示“仍在运行 / 暂无新输出”
AI 只在拿到结果或状态证据后判断成功失败

一句话：

时间只能说明“等得久”，不能说明“失败”。失败必须有失败证据，成功必须有成功证据。
