# liuAgent CLI 智能体的手规划

## 相关文档

- [项目设计](./PROJECT_DESIGN.md)

## 定位

liuAgent CLI 的“手”是一个受控工具执行层。模型只负责提出工具调用意图，CLI 负责校验、授权、执行和把结果回传给模型。

```text
模型 -> tool_call -> 权限检查 -> 工具执行 -> observation -> 模型继续推理
```

## 手的最小架构

### 1. Tool Registry

登记所有工具，让模型知道有哪些能力可以用。

每个工具保留：

- `name`: 工具名
- `description`: 用途说明
- `input_schema`: JSON Schema 参数定义
- `action`: 权限策略动作名，例如 `file.read`、`file.write`、`command.run`
- `risk`: 风险等级
- `requires_approval`: 默认是否需要用户确认
- `scope`: 工具允许访问的资源范围
- `execute`: 执行函数

### 2. Tool Router

接收模型的 `tool_call`，按工具名找到对应工具。

Router 只负责分发，不直接执行危险动作。

### 3. Permission Gate

所有工具执行前都要过权限闸门。

主要检查：

- 路径是否在 workspace 内
- 参数是否符合 `input_schema`
- 命令是否高风险
- 是否涉及删除、覆盖、部署、发消息、网络写入
- 是否可能读取或外传本地凭据
- `approve_session` 只在当前 `chat_session_id` 内缓存同类 action 授权，不能跨会话复用

### 4. Tool Executor

真正执行工具，并统一返回结果。

```json
{
  "tool_result_id": "result_001",
  "tool_call_id": "call_001",
  "ok": true,
  "content": "读取 120 行，命中 3 处。",
  "summary": "读取文件成功",
  "data": {},
  "error": null,
  "audit_id": "audit_001",
  "created_at": "2026-06-20T07:00:00Z"
}
```

### 5. Observation Adapter

把工具结果整理成模型能继续使用的 observation。

要求：

- 长文件只回传相关片段
- 长命令输出要截断
- 错误也要回传给模型
- 结构化结果优先保留 JSON

## 第一批工具

### 文件工具

- `list_files`: 列目录
- `read_file`: 读文件
- `search_text`: 搜索文本
- `apply_patch`: 修改文件
- `write_file`: 新建文件，默认需要确认

### 命令工具

- `run_command`: 执行本地命令
- `check_command_risk`: 判断命令风险

默认限制在 workspace 内执行。删除、安装依赖、部署、写系统目录必须确认。

### 网络工具

- `http_get`: 读取公开 URL
- `http_post`: 调用外部 API，默认需要确认
- `download_file`: 下载文件到 workspace

网络工具不能自动携带本地凭据。

### MCP 工具

- `list_mcp_tools`: 发现 MCP 工具
- `call_mcp_tool`: 调用 MCP 工具
- `read_mcp_resource`: 读取 MCP 资源

MCP 工具也必须经过同一套权限闸门。

## 权限等级

| 等级     | 例子                                 | 策略             |
| -------- | ------------------------------------ | ---------------- |
| safe     | 读文件、列目录、搜索                 | 自动执行         |
| medium   | 写 workspace 文件、运行只读命令      | 可配置确认       |
| high     | 删除、安装依赖、网络写入             | 必须确认         |
| critical | 部署、发消息、外传密钥、系统目录写入 | 默认禁止或强确认 |

## 最小执行循环

```text
1. 用户输入
2. 调模型
3. 模型返回最终答案：结束
4. 模型返回工具调用：
   - 校验工具名
   - 校验参数
   - 经过权限闸门
   - 执行工具
   - 回传 observation
5. 回到第 2 步
```

## 第一版边界

第一版只把“手”做稳，不先做复杂多智能体。

优先保证：

- 工具可被模型发现
- 参数可校验
- 执行有权限控制
- 结果能回传模型
- 失败后模型还能继续处理

后续再扩展长期记忆、多人协作、远程执行和任务树。

1. 模型能力
   - 用什么模型、上下文长度、函数调用能力、推理能力、多轮稳定性。
   - 现在桌面端已接 direct-openai-compatible 主线，能走标准 tool_calls。

2. 上下文质量
   - 项目手册、规则、需求历史、当前文件、任务树、运行状态能不能准确喂给模型。
   - 上下文越准，越不像“瞎猜”。

3. 工具能力
   - 能读文件、搜代码、写文件、跑命令、调 MCP、访问网络、操作项目数据。
   - 工具越完整，智能体越能真正完成事，而不是只聊天。

4. 工具选择与执行循环
   - 模型能不能自己决定该调用哪个工具。
   - 工具结果能不能回喂模型继续推理。
   - 这就是 user -> model -> tool_calls -> tool_results -> model continuation 的 Agent Loop。

5. 记忆与恢复
   - 能不能记住这个项目之前怎么做、当前做到哪、失败在哪里。
   - 包括 requirement、task tree、transcript、checkpoint、outbox。

6. 权限与安全边界
   - 越聪明越危险，所以必须有权限门。
   - 写文件、命令、网络写入、MCP 写入都要可确认、可审计、可恢复。

7. 规划与任务分解
   - 能不能把需求拆成正确步骤，并持续推进。
   - 不只是固定“分析/实现/验证”，而是按任务类型生成真实计划。

8. 反馈学习
   - 测试失败、用户纠正、项目经验能不能沉淀成规则或记忆。
   - 这决定它以后会不会重复犯错。

一句话：模型决定上限，工具和上下文决定能不能落地，记忆/任务树/权限决定能不能长期可靠地做事。

• 当前最薄弱的主要是这些：

1. 产品化权限体验
   - 权限门已经有了，但授权详情、撤销 session grant、审计摘要查看还不够完整。
   - 现在偏“能用”，还没到“用户一眼就放心”的程度。

2. CLI 独立形态
   - Core 已经有 CLI adapter 边界和 NDJSON 输出。
   - 但还没完整抽成独立 crate，也没有成熟的 CLI resume/replay 命令。

3. 真实模型效果验证
   - 已经接了 OpenAI-compatible tool_calls 主线，也补了本地 stub 测试。
   - 但还缺更多真实模型场景压测，比如多工具链、失败恢复、长上下文、复杂修改任务。

4. 长期记忆质量
   - requirement、task tree、transcript、checkpoint、outbox 都有基础。
   - 但“哪些记忆该长期保留、如何压缩、如何避免污染上下文”还需要继续打磨。

5. 自动规划能力
   - 任务树和计划链路可用了。
   - 但复杂任务下，计划质量仍依赖提示词和模型本身，缺少更强的计划健康检查和自动纠偏闭环。

6. 工具生态厚度
   - 文件、命令、网络、MCP 基础工具有了。
   - 但更高阶工具还薄，比如代码索引、项目级语义搜索、测试失败自动定位、数据库/浏览器/设计资源深度操作。

7. 异常恢复体验
   - runtime state 恢复、waiting approval 恢复已有。
   - 但用户侧“断了以后点一下继续”的体验还可以更顺，尤其是多轮工具执行中断、模型失败重试。

结论：核心执行链路已经通了，薄弱点主要在产品化、复杂任务稳定性、长期记忆质量和工具生态深度。
