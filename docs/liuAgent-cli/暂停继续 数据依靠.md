# 暂停继续 数据依靠

## 1. 目标

桌面智能体执行时间可能很长，也可能遇到手动暂停、网络超时、模型报错或应用重启。

暂停继续机制需要解决三个问题：

1. 暂停时把已经完成的工作保存下来。
2. 继续时不要重复执行已经成功的写文件、命令或外部操作。
3. 关闭并重新打开桌面应用后，仍然能够找到上一次的暂停状态。

## 2. 一句话说明

桌面智能体暂停后，会把运行状态、工具结果和执行记录写入当前工作区的 `.ai-employee` 目录。

点击“继续执行”时，不是让旧进程从原代码位置继续，而是启动一个新的智能体循环，读取上次保存的数据，判断下一步应该做什么。

## 3. 暂停流程

```text
用户点击暂停
    ↓
前端向 Tauri Runtime 发送暂停信号
    ↓
Runtime 立即禁止新的模型请求和工具调度
    ↓
本地状态函数读取已经持续落盘的最新 Runtime 事件
    ↓
直接记录当前工作节点、未完成操作和中断原因
    ↓
原子更新 state.json、checkpoint.json 和会话索引
    ↓
界面显示“任务已暂停，可以继续执行”
```

暂停和 checkpoint 写入不调用大模型，也不等待模型生成总结。网络连接失败同样进入暂停状态，并使用相同的本地恢复数据。

## 4. 数据保存在哪里

数据保存在本次执行使用的工作区目录中：

```text
<工作区>/.ai-employee/desktop-agent-runtime/sessions/<chat_session_id>/
├── state.json
├── checkpoint.json
├── transcript.jsonl
└── audit.jsonl
```

例如：

```text
/Volumes/work_mac_1_5T/self/ai-employee/
└── .ai-employee/desktop-agent-runtime/sessions/chat-session-xxxx/
```

### 4.1 state.json

这是恢复运行的主要数据，包含：

- 当前状态，例如 `paused`、`failed`、`waiting_approval`。
- 原始任务目标。
- 当前计划和调度状态。
- 已执行工具及其结果。
- 已完成和未完成的操作。
- 验证结果、重试建议和恢复判断。
- 后台任务信息。

### 4.2 checkpoint.json

这是一次完整的恢复快照。

它包含当前 `state.json` 的完整内容，并记录项目、会话、Runtime session 和最新状态。

### 4.3 transcript.jsonl

这是执行过程事件流，按行追加记录：

- 模型步骤。
- 工具调用。
- 命令输出。
- 工具执行结果。
- 状态变化。
- 授权等待。

该文件会在运行过程中持续追加，主要用于恢复界面轨迹和排查问题。

### 4.4 audit.jsonl

记录工具执行和权限审计信息，用于判断执行过什么操作、结果是否成功。

### 4.5 会话索引

系统还会写入会话索引，帮助重新打开应用后定位 Runtime 状态：

```text
<工作区>/.ai-employee/query-mcp/active-sessions/<chat_session_id>.json
<工作区>/.ai-employee/query-mcp/session-history/<project_id>__<chat_session_id>.json
```

## 5. 继续执行依靠什么数据

点击“继续执行”时，桌面端会读取以下数据：

| 数据 | 用途 |
| --- | --- |
| `run_state` | 判断上次是暂停、失败还是等待授权 |
| `current_state` | 获取任务目标、计划、观察结果和验证结果 |
| `tool_results` | 判断哪些工具已经执行成功或失败 |
| `operations` | 判断哪些操作已完成、哪些仍未完成 |
| `background_jobs` | 判断是否还有后台任务正在运行 |
| `resume_judgement` | 判断应该继续、等待还是先检查结果 |
| 聊天历史 | 恢复用户原始需求和前后文 |

系统会把这些数据整理成恢复快照，再启动一个新的模型循环。

新的模型循环必须遵守：

1. 不重复执行已经成功的写文件、删除、命令和外部写入。
2. 对结果不确定的操作先检查状态，不直接重放。
3. 后台任务仍在运行时，先刷新状态或等待。
4. 根据原始目标继续未完成部分，而不是重新开始整个任务。

## 6. 关闭重新打开会不会丢失

### 6.1 不会丢失的情况

满足以下条件时，关闭并重新打开桌面应用后可以恢复：

- 界面已经显示“任务已暂停，可以继续执行”。
- 重新打开后使用相同的工作区。
- 使用相同的项目和聊天会话。
- 工作区中的 `.ai-employee` 目录没有被删除。

桌面应用重新进入会话时，会读取 `state.json`、checkpoint 和 transcript，重新生成可恢复状态和“继续执行”按钮。

### 6.2 可能丢失的情况

以下情况可能只能恢复到最后一条成功写入的 Runtime 事件：

- 删除了工作区中的 `.ai-employee` 目录。
- 更换了工作区目录。
- 使用了不同的 `chat_session_id`。
- 工作区文件损坏或没有写入权限。

## 7. 当前机制的边界

Runtime 每收到一条模型、计划、工具、命令或状态事件，都会追加 `transcript.jsonl` 并滚动更新 checkpoint。暂停函数只负责把当前事件节点冻结为 `paused`：

```text
点击暂停
    ↓
禁止新的模型请求和工具调度
    ↓
本地原子 checkpoint 写入完成
```

已经发送到模型服务的 HTTP 请求不能撤回，但暂停后 Runtime 不会发起新的模型请求，也不会把该请求返回的工具计划继续调度。

另外，“继续执行”属于基于数据的重新规划，不是恢复旧进程的内存和调用栈。因此不能绝对保证外部系统操作的 exactly-once。

如果某个外部操作在中断前已经发出，但结果没有成功记录，恢复后必须先查询外部状态，再决定是否重试。

## 8. 前端聊天数据

除了 Runtime checkpoint，聊天界面还会保存消息、操作卡和执行日志：

- 桌面 WebView 的 `localStorage` 保存本地快照。
- 后端聊天 Runtime 接口保存服务端快照。
- 数据量过大时，本地快照可能被截断，完整消息以服务端历史为准。

前端聊天缓存用于恢复界面展示，真正决定能否继续运行的核心数据仍然是工作区中的 Runtime 状态文件。

## 9. 用户操作建议

1. 点击暂停后，界面会直接进入可恢复状态。
2. 重新打开应用时，选择原来的工作区和聊天会话。
3. 不要手动删除 `.ai-employee/desktop-agent-runtime`。
4. 如果恢复后提示结果不确定，先检查文件、命令或外部任务状态。

## 10. 后续可增强项

为了进一步降低异常退出造成的数据损失，可以继续增加：

1. 对仍在运行的本地子进程增加平台级挂起与恢复。
2. 对远程外部写入统一增加幂等键和结果查询接口。
3. 对损坏的最后一条 JSONL 事件增加自动截断修复。
