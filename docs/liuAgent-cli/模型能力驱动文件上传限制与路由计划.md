# 模型能力驱动的文件上传限制与路由计划

## 背景

当前项目对话和桌面本地智能体已经支持附件上传，但文件处理逻辑仍偏全局化：

- 项目设置里有统一的 `allowed_file_types`、`max_file_size_mb` 和上传数量限制。
- 桌面本地智能体会先尝试在前端解析文件，再把 `extractedText` 交给本地 Runtime。
- 远端对话会把可解析文本拼进 prompt，图片转成 base64。
- 不同模型的原生文件能力没有统一建模，导致“用户能上传”不等于“当前模型能直接理解”。

更合理的方向是按“供应商 + 模型”的能力决定上传限制和处理策略：支持文件直传的模型走 provider file upload；不支持的模型再走本地解析、OCR、转写或拒绝。

## 目标

1. 为每个模型建立明确的文件输入能力配置。
2. 前端上传入口按当前模型能力动态限制文件类型、大小和数量。
3. Runtime 根据能力路由附件，不再固定依赖本地解析。
4. 支持模型原生文件输入时，优先让大模型自己解析文件。
5. 保留本地解析作为兜底，且解析失败时显式提示，不静默忽略。

## 非目标

- 不在第一阶段实现所有供应商的文件上传 API。
- 不把所有文件无条件上传到模型厂商。
- 不移除本地解析能力。
- 不让前端单独决定最终能力，后端和桌面 Runtime 必须再次校验。

## 能力模型

新增统一的模型附件能力描述，建议命名为 `model_attachment_capabilities`。

```json
{
  "provider": "openai",
  "model": "gpt-5.5",
  "file_input": {
    "mode": "provider_file",
    "max_files": 10,
    "max_file_size_mb": 25,
    "max_total_size_mb": 100,
    "mime_types": [
      "application/pdf",
      "text/plain",
      "image/png"
    ],
    "extensions": [
      ".pdf",
      ".txt",
      ".png"
    ],
    "supports_image": true,
    "supports_document": true,
    "supports_audio": false,
    "supports_video": false
  }
}
```

### `mode` 说明

| mode | 含义 | 处理方式 |
|---|---|---|
| `provider_file` | 模型支持原生文件输入 | 上传到供应商，拿 `provider_file_id` 后引用 |
| `inline_image` | 模型只支持图片输入 | 图片走 base64 / image_url，文档走兜底 |
| `local_extract` | 模型只支持文本 | 本地解析文本后拼进上下文 |
| `retrieval` | 模型或系统支持检索增强 | 文件分块入库，按问题检索片段 |
| `unsupported` | 当前模型不支持附件 | 前端阻止上传或发送前提示 |

## 路由规则

附件发送时按以下顺序决策：

```text
当前 provider + model
  -> 读取附件能力
  -> 校验文件数量、大小、扩展名、MIME
  -> 判断是否能 provider_file 直传
  -> 判断是否能 inline image
  -> 判断是否需要 local_extract
  -> 判断是否需要 retrieval
  -> 生成 model_payload_parts
```

优先级：

1. 支持 `provider_file` 且文件类型匹配时，优先直传模型厂商。
2. 图片且模型支持视觉输入时，优先走图片输入。
3. 文档但模型不支持文件输入时，走本地解析或检索。
4. 文件过大时，优先进入检索/摘要流程，不能直接塞进 prompt。
5. 不支持的文件类型在发送前拦截，并说明当前模型限制。

## 前端改造

### 1. 能力加载

当前选择模型变化时，请求模型附件能力：

```text
GET /llm/providers/{provider_id}/models/{model_name}/attachment-capabilities
```

返回结果用于：

- 上传按钮 accept
- 上传数量限制
- 单文件大小限制
- 文件列表提示
- 发送前校验
- “当前模型处理方式”展示

### 2. 上传校验

上传组件不再只看项目全局设置，而是合并：

```text
effective_limit = min(project_limit, model_limit)
effective_types = intersection(project_allowed_types, model_allowed_types)
```

如果模型没有附件能力：

- 隐藏附件入口，或禁用上传按钮。
- 提示“当前模型不支持文件输入，请切换模型或粘贴文本内容”。

### 3. 发送前提示

发送前展示每个附件的处理方式：

```text
report.pdf -> 将上传给 OpenAI 文件接口
screenshot.png -> 将作为图片输入
data.xlsx -> 当前模型不支持表格直传，将本地解析为文本
archive.zip -> 当前模型不支持，无法发送
```

## Runtime 改造

### 1. 统一附件对象

扩展现有 `LocalChatAttachment`：

```json
{
  "attachmentId": "att_001",
  "name": "report.pdf",
  "mimeType": "application/pdf",
  "size": 123456,
  "kind": "document",
  "routingMode": "provider_file",
  "provider": "openai",
  "providerFileId": "file_xxx",
  "extractionStatus": "not_required",
  "extractedText": "",
  "error": ""
}
```

### 2. Provider Adapter

新增 `ModelAttachmentAdapter`：

```ts
interface ModelAttachmentAdapter {
  provider: string
  supports(model: string): boolean
  getCapabilities(model: string): AttachmentCapabilities
  uploadFile(input: LocalAttachment): Promise<ProviderFileRef>
  buildPayload(parts: AttachmentContextPart[]): unknown[]
}
```

第一阶段至少实现：

- `OpenAIResponsesAttachmentAdapter`
- `OpenAICompatibleFallbackAdapter`
- `LocalExtractAttachmentAdapter`

### 3. 本地 Runtime 路由

桌面 Runtime 不再默认要求前端先解析文档。

```text
LocalChatRequest.attachments
  -> resolve_model_attachment_capability
  -> route_attachment
  -> provider upload / inline image / local extract / reject
  -> build model request
```

前端仍可提供 `extractedText` 作为缓存或兜底，但不能作为唯一来源。

## 后端改造

### 1. 能力配置来源

短期可以配置在服务端模型供应商模板中：

```json
{
  "attachment_capabilities": {
    "default": {
      "mode": "local_extract",
      "max_files": 6,
      "max_file_size_mb": 15,
      "extensions": [".txt", ".md", ".pdf", ".docx", ".xlsx", ".png", ".jpg"]
    },
    "gpt-5.5": {
      "mode": "provider_file",
      "max_files": 10,
      "max_file_size_mb": 25,
      "extensions": [".pdf", ".txt", ".png", ".jpg"]
    }
  }
}
```

长期应沉淀到独立表或配置中心，并支持：

- 按 provider 默认
- 按 model 覆盖
- 按项目覆盖
- 按用户或环境关闭直传

### 2. 安全策略

文件直传模型厂商前必须经过：

- 文件大小校验
- MIME 和扩展名校验
- 敏感文件名拦截，例如 `.env`、密钥、证书
- 用户可见的处理方式提示
- 审计记录：上传到哪个供应商、返回哪个 `provider_file_id`

## 数据结构

建议新增附件处理记录：

```json
{
  "attachment_id": "att_001",
  "message_id": "msg_001",
  "provider": "openai",
  "model": "gpt-5.5",
  "routing_mode": "provider_file",
  "status": "ready",
  "provider_file_id": "file_xxx",
  "derived_text_id": "",
  "error": "",
  "created_at": "2026-06-25T00:00:00Z"
}
```

作用：

- 复用同一附件的供应商文件上传结果。
- 排查“文件上传了但模型没解析”的问题。
- 给用户展示真实处理状态。
- 支持后续清理供应商侧文件。

## 阶段计划

### 阶段一：能力表和前端限制

- 新增模型附件能力 schema。
- 在模型设置或供应商模板中维护默认能力。
- 前端上传组件读取当前模型能力。
- 上传前按模型限制拦截文件。
- 发送前展示附件处理方式。

验收：

- 切换模型后，上传限制实时变化。
- 不支持附件的模型无法上传文件。
- 超过模型限制时提示清楚，不进入发送流程。

### 阶段二：Runtime 路由抽象

- 新增附件路由器。
- 新增 `routingMode`、`providerFileId`、`error` 等字段。
- 桌面本地 Runtime 不再只依赖前端解析。
- 本地解析能力改成一种 fallback adapter。

验收：

- 本地模型只支持文本时，文档自动解析为文本。
- 支持图片的模型能收到图片输入。
- 解析失败时，模型上下文明确包含失败原因。

### 阶段三：Provider 文件直传

- 实现 OpenAI Responses 文件上传适配。
- 建立 `provider_file_id` 缓存和审计。
- 调模型时引用 provider 文件对象。
- 增加失败回退：直传失败可选择本地解析或阻止发送。

验收：

- 支持直传的模型收到 provider 文件输入。
- 同一附件重复发送可复用 `provider_file_id`。
- 供应商上传失败有明确错误和兜底路径。

### 阶段四：大文件与检索

- 大文件进入分块与索引。
- 根据用户问题检索相关片段。
- 对表格、PDF、长文档做摘要缓存。

验收：

- 大文档不会直接撑爆上下文。
- 模型回答能引用命中文档片段。
- 检索失败时不伪装已经读取。

## 回归测试

至少覆盖：

- 不支持附件模型：上传入口禁用。
- 只支持图片模型：图片可传，PDF 被拒绝或走解析。
- 支持文件模型：PDF 走 provider upload。
- 文本模型：`.txt/.md/.csv/.pdf/.docx/.xlsx` 走本地解析。
- Excel 解析失败时返回明确错误。
- 文件超限时阻止发送。
- 切换模型后旧上传队列重新校验。
- 桌面 Runtime 收到 `providerFileId` 时不重复解析。

## 风险

- 不同供应商文件上传接口差异大，不能用一个请求格式硬套。
- 模型宣传支持文件，不代表所有文件类型都稳定可解析。
- 直传给供应商涉及隐私边界，必须给用户可见提示。
- 本地解析和模型直传结果可能不一致，需要在上下文里标注来源。
- 文件能力配置过期会导致误拦截或误放行，需要可维护的更新机制。

## 最小落地结论

第一版不要追求所有模型都能文件直传。先完成“模型能力表 + 前端动态限制 + Runtime 路由抽象”，让系统知道当前模型到底能不能吃文件。之后再逐个供应商接 `provider_file` 上传能力。
