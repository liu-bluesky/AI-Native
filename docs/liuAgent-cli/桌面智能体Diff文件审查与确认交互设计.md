# 桌面智能体 Diff 文件审查与确认交互设计

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档状态 | 设计方案，待开发 |
| 设计日期 | 2026-07-22 |
| 适用页面 | 项目 AI 对话页、桌面本地智能体执行界面 |
| 核心目标 | 让用户集中查看 AI 修改的文件，并逐个或批量确认保存、放弃或撤回 |
| 推荐实现 | Monaco Diff Editor 按需加载 + 统一文件变更抽屉 |

## 2. 背景与现状

当前系统已经具备部分文件变更能力，但入口和决策流程较分散：

- 已有项目工作区文件列表、文件读取和服务端保存接口。
- 前端已维护 `workspaceFileOriginal`、`workspaceFileDraft`、`workspaceDiffPreview` 等状态。
- 桌面端已有 Git 工作区 Diff 预览，可读取 `git status --short`、`git diff --stat` 和 `git diff`。
- 桌面端已有写入前准备和风险确认逻辑，但当前只记录确认，不执行真实写入。
- 消息审计区域能展示“工作区变更”和“变更审查”，但不适合作为长期、集中管理多个文件的主入口。

因此本设计不再新增一个孤立的 Diff 弹窗，而是把现有能力收敛为一个统一的 **文件变更审查中心**。

## 3. 设计目标

### 3.1 用户目标

用户需要随时知道：

1. AI 本轮修改了哪些文件。
2. 每个文件具体增加、删除、修改了什么。
3. 哪些修改尚未确认，哪些已保存，哪些已放弃或撤回。
4. 是否可以只保存部分文件，而不是一次接受所有修改。
5. 保存后发现问题时，能否安全撤回。

### 3.2 产品目标

- 文件变更有唯一、稳定、容易发现的入口。
- “查看差异”和“修改决策”在同一个界面完成。
- 保存、放弃、撤回三个概念边界清楚。
- 不依赖颜色传达状态，颜色之外同时提供图标和文字。
- 高风险文件和外部冲突必须阻止静默覆盖。
- 不要求用户理解 Git 概念，也能完成审查。

## 4. 核心概念定义

### 4.1 保存修改

把当前候选内容写入工作区文件。保存后该文件从“待确认”变为“已保存”。

### 4.2 放弃修改

针对尚未保存的候选修改，丢弃候选内容，工作区文件保持原样。

放弃只作用于“尚未写入磁盘”的内容，不修改 Git 历史，也不修改已经保存的文件。

### 4.3 撤回修改

针对本轮已经保存的文件，将文件恢复到本轮修改前的快照。

撤回不是直接执行 `git checkout` 或 `git restore`，因为：

- 文件可能在修改前就包含用户未提交的改动。
- 直接恢复 Git 版本可能误删用户原有工作。
- 非 Git 工作区也需要支持撤回。

正确方式是：首次写入前保存本轮基线快照，撤回时恢复该快照；如果检测到文件已被其他进程再次修改，则进入冲突处理，不允许直接覆盖。

## 5. 页面入口设计

### 5.1 主入口位置

在项目对话页右下角、输入框上方操作区增加常驻入口：

```text
┌──────────────────────────────────────────────┐
│ 对话消息区域                                  │
│                                              │
│                        ┌──────────────────┐  │
│                        │ 文件变更  5       │  │
│                        │ +126  -38        │  │
│                        └──────────────────┘  │
├──────────────────────────────────────────────┤
│ 输入框                                       │
└──────────────────────────────────────────────┘
```

入口采用“文件图标 + 文件变更 + 数量徽标”的形式：

- 无变更：入口弱化显示，可不展示数字。
- AI 正在修改：显示动态状态“正在生成变更…”。
- 有待确认文件：使用主色描边，并显示橙色数量徽标。
- 全部已处理：短暂显示“变更已处理”，随后恢复弱化状态。
- 存在冲突或失败：显示红色状态点和“需要处理”。

不建议只把入口放进设置中心、消息审计卡片或文件树中，因为这些位置不够稳定，用户容易错过待确认修改。

### 5.2 辅助入口

以下位置可以跳转到同一个文件变更抽屉，但不创建第二套界面：

- AI 消息中的“已修改 5 个文件”摘要。
- 执行审计中的“工作区变更”。
- 任务完成提示中的“审查修改”。
- 工作区文件树中文件右侧的变更状态图标。

所有入口最终调用同一个动作：`openFileChangesDrawer({ selectedPath })`。

## 6. 文件变更抽屉

### 6.1 展开方式

桌面端使用右侧抽屉，建议宽度为页面的 `72%`，最小 `840px`，最大 `1440px`。

窄屏或窗口宽度小于 `1100px` 时，切换为全屏覆盖层，不强行维持双栏。

选择抽屉而不是普通弹窗的原因：

- 用户需要在对话内容和文件修改之间来回核对。
- 多文件列表需要稳定的纵向空间。
- Diff 编辑器需要较宽区域。
- 抽屉关闭后能自然回到对话，不打断当前任务上下文。

### 6.2 桌面布局

```text
┌──────────────── 文件变更审查 ───────────────────────────────────────┐
│ 5 个文件待处理  +126 -38       [仅看待处理] [刷新] [关闭]           │
├──────────────────┬───────────────────────────────────────────────────┤

│                  │ 已修改 · +42 -11 · 中风险                         │
│ ● 待确认 3       ├───────────────────────────────────────────────────┤
│ ✓ 已保存 1       │ [并排对比] [单栏对比] [忽略空白] [上一处][下一处] │
│ ↶ 已撤回 1       ├───────────────────────────────────────────────────┤
```

### 6.3 左侧文件列表

#### 分组

文件按状态分组，而不是单纯按目录排列：

1. **待确认**：默认展开，最重要。
2. **存在冲突**：置顶并使用错误状态。
3. **已保存**：默认折叠。
4. **已放弃**：默认折叠。
5. **已撤回**：默认折叠。

#### 文件项信息

每个文件项展示：

- 文件名。
- 相对路径，文件名重复时必须显示完整路径。
- 状态图标和文字：新增、修改、删除、重命名、冲突。
- 变更统计：`+42 -11`。
- 风险等级：仅中风险和高风险展示标签。
- 选择框：用于批量保存或批量放弃。

文件状态建议：

| 状态 | 标识 | 说明 |
|---|---|---|
| 新增 | `A` + 绿色 | 本轮创建的新文件 |
| 修改 | `M` + 蓝色 | 修改已有文件 |
| 删除 | `D` + 红色 | 准备删除文件，必须二次确认 |
| 重命名 | `R` + 紫色 | 展示旧路径和新路径 |
| 冲突 | `!` + 红色 | 基线之后发生外部修改 |
| 已保存 | 勾选图标 | 已写入工作区 |
| 已放弃 | 禁止图标 | 候选修改已丢弃 |
| 已撤回 | 回退图标 | 已恢复本轮修改前快照 |

#### 列表交互

- 点击文件项：在右侧打开该文件 Diff。
- `Shift` 连选、`Command/Ctrl` 多选。
- 鼠标悬停显示快捷操作，但危险操作不能只用图标。
- 切换文件前，如果右侧还有未同步到候选状态的编辑，先自动保存到内存草稿，不直接写磁盘。
- 文件较多时支持按路径搜索和按状态筛选。

### 6.4 右侧 Diff 区域

采用 Monaco Diff Editor，按需加载。

默认设置：

- 左侧原文件只读。
- 右侧候选修改可编辑。
- 默认并排模式；空间不足自动切换单栏 Diff。
- 显示行号、语法高亮、变更导航和迷你地图关闭。
- 支持“上一处变更 / 下一处变更”。
- 支持“忽略空白变化”，但默认关闭。
- 右侧内容编辑后，文件保持“待确认”状态并重新计算统计。

顶部信息栏显示：

- 文件相对路径。
- 文件状态。
- 新增和删除行数。
- 风险等级。
- 是否存在外部修改。

#### 大文件和二进制文件

- 文本文件大于 1MB：不加载完整编辑器，展示统计和“使用系统编辑器打开”。
- 图片：提供修改前后图片对比，可切换并排和滑动对比。
- 二进制文件：只展示文件大小、哈希和修改时间，不展示文本 Diff。
- Diff 被截断时必须明确显示“内容已截断”，不能让用户误以为已查看全部修改。

## 7. 决策流程设计

### 7.1 单文件保存

点击“保存当前文件”后：

1. 重新读取磁盘文件的修改时间和内容哈希。
2. 与该文件基线快照比较。
3. 无冲突时展示保存确认摘要。
4. 用户确认后执行真实写入。
5. 写入完成后再次读取并验证内容哈希。
6. 文件状态更新为“已保存”。
7. 写入审计记录和本轮变更记录。

确认框示例：

```text
保存这个文件？

src/views/projects/ProjectChat.vue
修改 53 行：新增 42 行，删除 11 行
风险等级：中风险

保存后仍可在本轮任务结束前撤回到修改前版本。

[返回审查]                  [确认保存]
```

低风险文件可以简化确认内容，但不能跳过确认；高风险文件需要显示风险原因。

### 7.2 批量保存

点击“保存所选”后，先进入批量确认页，不立即写入：

```text
确认保存 5 个文件？

新增文件  1
修改文件  3
删除文件  1
总变更    +126 -38

高风险文件
- web-admin/api/auth.py：认证逻辑变更
- config/prod.env：环境配置变更

□ 我已查看高风险文件

[返回审查]                  [确认保存 5 个文件]
```

批量保存要求：

- 保存前逐个执行冲突检查。
- 任一文件冲突时，不静默跳过。
- 默认采用“安全批次”：存在冲突则整批停止，用户处理后再保存。
- 后续如需支持“保存无冲突文件”，应作为用户显式选择，不作为默认行为。

### 7.3 放弃未保存修改

点击“放弃修改”表示丢弃候选内容，不改磁盘文件。

单文件提示：

```text
放弃这个文件的候选修改？

文件不会被写入，当前候选内容将被清除。
该操作在关闭本轮任务后不可恢复。

[取消]                      [放弃修改]
```

批量放弃时必须列出文件数量和文件名；删除、新增文件同样适用。

建议在当前页面生命周期内保留一次内存级“撤销放弃”机会，显示 8 秒 Toast：

```text
已放弃 ProjectChat.vue 的候选修改。    [撤销]
```

该撤销仅恢复候选草稿，不写入磁盘。

### 7.4 撤回已保存修改

已保存文件在本轮任务未结束前显示“撤回本轮修改”。

点击后：

1. 读取当前磁盘内容和哈希。
2. 确认当前内容仍等于本轮保存后的快照。
3. 如果一致，展示修改后版本与修改前基线的反向 Diff。
4. 用户确认后恢复基线快照。
5. 重新读取文件验证恢复结果。
6. 状态更新为“已撤回”。

如果文件在保存后又被用户、IDE 或其他智能体修改：

- 禁止一键撤回。
- 文件进入“冲突”状态。
- 提供“查看当前内容”“复制基线内容”“手动合并”三个动作。
- 不提供“强制覆盖”作为普通按钮；如未来需要，必须走高风险确认。

撤回确认示例：

```text
撤回本轮对这个文件的修改？

将恢复到 AI 修改前的版本。
检测到文件保存后没有新的外部修改，可以安全撤回。

[取消]                      [确认撤回]
```

### 7.5 关闭抽屉

关闭抽屉不等于放弃修改。

- 候选修改继续保留。
- 主入口继续显示待确认数量。
- 用户离开页面或切换项目时，如果还有待确认修改，必须弹出离开提醒。
- 刷新页面后需要从本轮变更会话恢复候选状态；不能仅依赖 Vue 内存状态。

## 8. AI 执行结束后的交互

当 AI 完成代码修改但还有文件未确认时，不直接把任务展示为“完全完成”。

推荐状态：

```text
AI 已完成修改，等待你审查 5 个文件
[查看文件变更]
```

任务状态建议区分：

| 状态 | 含义 |
|---|---|
| 执行中 | AI 仍在读取、修改或验证 |
| 待审查 | AI 已完成候选修改，等待用户处理文件 |
| 部分确认 | 部分文件已保存，仍有待处理文件 |
| 已确认 | 所有候选文件已保存或明确放弃 |
| 已撤回 | 本轮已保存修改全部撤回 |
| 冲突阻塞 | 文件发生外部修改，需要人工处理 |

只有所有文件都进入终态后，文件审查阶段才算结束。终态包括：

- 已保存。
- 已放弃。
- 已撤回。

## 9. 状态模型

建议不要继续只依赖单个 `workspaceFileOriginal` 和 `workspaceFileDraft`。多文件审查需要独立的变更会话模型。

```ts
type FileChangeStatus =
  | "pending"
  | "saving"
  | "saved"
  | "discarded"
  | "reverting"
  | "reverted"
  | "conflict"
  | "failed";

interface FileChangeItem {
  id: string;
  path: string;
  changeType: "added" | "modified" | "deleted" | "renamed";
  previousPath?: string;
  language: string;
  baselineContent: string;
  baselineHash: string;
  baselineModifiedAt: number;
  draftContent: string;
  draftHash: string;
  savedContent?: string;
  savedHash?: string;
  additions: number;
  deletions: number;
  riskLevel: "low" | "medium" | "high";
  riskReasons: string[];
  status: FileChangeStatus;
  error?: string;
}

interface FileChangeSession {
  id: string;
  projectId: string;
  chatSessionId: string;
  workspacePath: string;
  createdAt: string;
  updatedAt: string;
  status: "collecting" | "awaiting_review" | "partially_resolved" | "resolved" | "blocked";
  files: FileChangeItem[];
}
```

## 10. 数据与安全策略

### 10.1 基线快照

- 第一次准备修改某个文件时保存基线内容、哈希和修改时间。
- 同一轮任务再次修改同一文件，不覆盖第一次基线。
- 基线快照是撤回能力的依据。
- 任务归档后按保留策略清理，不长期保存敏感源码副本。

### 10.2 并发冲突

保存前至少校验：

- 文件是否仍存在。
- 当前磁盘哈希是否等于基线哈希。
- 修改时间是否发生变化。
- 新增文件路径是否已被其他进程占用。

撤回前至少校验：

- 当前磁盘哈希是否等于本轮保存后的哈希。
- 文件是否在本轮保存后被再次改动。

### 10.3 路径安全

- 所有文件路径必须经过工作区范围校验。
- 禁止 `../` 逃逸、符号链接逃逸和写入工作区外路径。
- 删除、重命名、权限变更单独提升风险等级。
- `.env`、凭据、认证、部署配置等敏感路径默认中高风险。

### 10.4 非 Git 工作区

文件审查功能不能依赖 Git 才可使用：

- Git 工作区：Git Diff 用于补充工作区整体变化。
- 非 Git 工作区：使用基线内容与候选内容计算 Diff。
- 撤回始终使用本轮基线快照，不直接依赖 Git 恢复。

## 11. 组件拆分建议

建议从 `ProjectChat.vue` 中拆出独立模块，避免继续增加单文件复杂度。

| 组件或模块 | 职责 |
|---|---|
| `FileChangesTrigger.vue` | 对话页常驻入口、数量和状态提示 |
| `FileChangesDrawer.vue` | 抽屉布局、状态编排和批量操作 |
| `FileChangeList.vue` | 文件分组、筛选、多选和状态展示 |
| `FileDiffEditor.vue` | Monaco Diff Editor 包装、语言识别和变更导航 |
| `FileChangeConfirmDialog.vue` | 保存、放弃、撤回确认 |
| `useProjectChatFileChanges.js` | 多文件状态、选中项和决策动作 |
| `projectChatFileChangesApi.js` | 变更会话、保存、撤回和冲突检查接口 |

现有 `useProjectChatWorkspaceFiles.js` 继续负责文件树浏览，不承担完整的多文件审查会话。

## 12. 后端与桌面端能力建议

现有单文件读写和 Git Diff 接口可以复用，但建议新增语义明确的文件变更会话接口：

```text
POST   /projects/{project_id}/file-change-sessions
GET    /projects/{project_id}/file-change-sessions/{session_id}
POST   /projects/{project_id}/file-change-sessions/{session_id}/files/{file_id}/save
POST   /projects/{project_id}/file-change-sessions/{session_id}/files/{file_id}/discard
POST   /projects/{project_id}/file-change-sessions/{session_id}/files/{file_id}/revert
POST   /projects/{project_id}/file-change-sessions/{session_id}/save-selected
```

桌面端桥接能力建议补齐：

- `create_workspace_file_change_session`
- `prepare_workspace_file_change`
- `save_workspace_file_change`
- `discard_workspace_file_change`
- `revert_workspace_file_change`
- `inspect_workspace_file_conflict`

所有写操作必须返回：

- 是否执行成功。
- 写入前后哈希。
- 文件修改时间。
- 实际变更统计。
- 冲突信息。
- 可用于审计的操作 ID。

## 13. 视觉规范

### 13.1 色彩

- 新增：绿色，但同时显示 `A` 或“新增”。
- 删除：红色，但同时显示 `D` 或“删除”。
- 修改：蓝色，但同时显示 `M` 或“修改”。
- 待确认：橙色状态点。
- 冲突：红色错误图标和文字。
- 已保存：中性绿色勾选。
- 已放弃、已撤回：灰色图标和明确文字。

### 13.2 层级

- 第一视觉焦点是当前文件 Diff。
- 第二视觉焦点是底部主要决策按钮。
- 文件列表使用中性背景，避免多种高饱和颜色争抢注意力。
- 主操作每个状态只保留一个：待确认时为“保存”，已保存时为“撤回”。

### 13.3 按钮

- 主按钮：`确认保存` 或 `保存所选`。
- 次按钮：`返回审查`、`取消`。
- 危险按钮：`放弃修改`、`确认撤回`，使用危险样式但不与主按钮并排抢占视觉焦点。
- 所有仅图标按钮必须有 Tooltip 和 `aria-label`。
- 点击区域不小于 40px，触屏环境不小于 44px。

## 14. 键盘与无障碍

- `Command/Ctrl + Shift + D`：打开或关闭文件变更抽屉。
- `Command/Ctrl + S`：在抽屉内触发“保存当前文件”的确认流程，不直接静默保存。
- `Alt + ↑/↓`：上一处或下一处变更。
- `Escape`：关闭当前确认框；第二次才关闭抽屉。
- 文件状态不能只用红绿颜色表达。
- 抽屉打开后焦点进入标题，关闭后返回触发按钮。
- Monaco 加载失败时提供纯文本 Unified Diff 降级视图。

## 15. 异常与边界状态

| 场景 | 处理方式 |
|---|---|
| 工作区不是 Git 仓库 | 使用基线与候选内容计算 Diff |
| 文件保存前被外部修改 | 标记冲突，禁止覆盖 |
| 保存后又被外部修改 | 禁止一键撤回，进入手动合并 |
| 文件被删除 | 显示“文件已不存在”，提供按冲突处理 |
| 新增文件路径被占用 | 标记冲突，不覆盖现有文件 |
| 二进制文件 | 展示元信息，不展示文本 Diff |
| 文件大于限制 | 展示统计并允许外部编辑器打开 |
| Monaco 加载失败 | 降级为 Unified Diff 只读视图 |
| 保存部分失败 | 显示逐文件结果，不把整批标记为成功 |
| 页面刷新 | 恢复当前变更会话和待确认状态 |
| 切换项目 | 有待确认修改时先提醒用户 |

## 18. 验收标准

### 入口与列表

- AI 修改文件后，用户不展开消息详情也能看到待确认数量。
- 点击入口可查看完整修改文件列表。
- 文件状态、路径和增删行数准确。

### Diff

- 用户能查看原文件和候选文件的并排差异。
- 新增、删除和修改均有颜色、图标或文字双重表达。
- 可跳转上一处和下一处变更。

### 保存

- 保存前有清晰摘要和确认。
- 用户可以只保存一个文件或保存所选文件。
- 保存后重新读取并校验结果，不能只根据接口返回判断成功。

### 放弃

- 放弃未保存修改不会改动磁盘文件。
- 批量放弃前明确显示影响文件。
- 页面生命周期内可撤销最近一次放弃。

### 撤回

- 已保存文件可以恢复到本轮修改前快照。
- 保存后发生外部修改时，一键撤回被阻止。
- 撤回成功后重新读取并验证内容。

### 状态闭环

- 所有文件进入已保存、已放弃或已撤回后，待审查状态结束。
- 页面刷新后仍能恢复待确认文件和处理状态。
- 所有保存、放弃和撤回动作均进入审计记录。

## 19. 最终设计决策

本功能采用：

> **常驻文件变更入口 + 右侧统一审查抽屉 + Monaco Diff Editor + 基线快照撤回机制。**

不采用以下方案：

- 不只在 AI 消息里放一个 Diff 链接，因为多文件管理和后续恢复不稳定。
- 不只使用临时确认弹窗，因为用户无法完整查看和逐个处理文件。
- 不用 Git 恢复代替撤回，因为可能覆盖用户在任务开始前已有的未提交修改。
- 不在候选内容生成时立即写磁盘，因为用户需要先审查并明确确认。

该方案能把“AI 修改文件”从一次不可见的写入动作，变成可查看、可选择、可确认、可放弃、可撤回的完整审查流程。
