# AI 项目执行系统目标定义

> 本文用于重新定义 ai员工工厂的产品方向与阶段目标。核心结论：不要把 ai员工工厂做成另一个 Codex 或 Hermes，而要做成管理 Codex、Hermes、Claude Code、MCP、本地工具和业务流程的项目级执行系统。

## 一句话定位

ai员工工厂是面向真实项目交付的 Agent 工作流控制台。

它负责把用户目标转化为可追踪、可恢复、可验证、可交付的执行过程；Codex、Hermes、Claude Code、本地 MCP、飞书、部署脚本和业务工具，都是它可以调度的执行能力。

## 不做什么

ai员工工厂不应定位为以下产品：

- 通用 AI 聊天工具
- 另一个 Codex
- 另一个 Hermes
- 单纯 MCP 管理后台
- 单纯 AI 员工展示系统
- 只记录任务但不能真实执行的项目管理工具

这些方向要么已经有强竞争者，要么无法体现项目自己的差异化价值。

## 要做什么

ai员工工厂要成为：

- AI 项目执行系统
- 项目级 Agent OS
- Agent 工作流控制台
- AI 项目经理与执行编排平台

它的核心价值不是“模型会回答”，而是“系统能带着项目上下文完成真实任务，并清楚告诉用户做到哪、卡在哪、验证了什么、谁能接着做”。

## 核心用户价值

用户不只是让 AI 回答问题，而是把一个真实目标交给系统。

系统需要做到：

- 理解当前项目上下文
- 生成任务树
- 选择合适的 AI 员工、工具或外部 Agent
- 执行真实命令、MCP、脚本、接口或业务动作
- 遇到授权、确认、验证码等人工动作时明确暂停
- 用户完成操作后自动恢复并继续原目标
- 对关键步骤做真实验证
- 输出交付报告
- 沉淀项目记忆、经验规则和后续可复用流程

## 与 Codex / Hermes 的关系

Codex 是优秀的代码执行员工。

Hermes 是优秀的多工具、多平台 Agent 框架。

ai员工工厂不是替代它们，而是管理它们。

推荐关系：

```text
ai员工工厂
  -> 管理项目目标、任务树、状态、权限、交付、记忆
  -> 调度 Codex / Hermes / Claude Code / MCP / 本地工具
  -> 验证结果并沉淀经验
```

也就是说：

```text
Codex / Hermes / Claude Code = 执行员工
ai员工工厂 = 项目经理 + 调度系统 + 状态中枢 + 交付系统
```

## 第一阶段目标

第一阶段只做一个可靠闭环：

```text
用户目标
-> 任务树
-> 真实工具执行
-> 人工操作暂停
-> 自动恢复续跑
-> 结果验证
-> 交付记录
-> 项目记忆
```

这个闭环必须优先于员工市场、技能广场、多平台扩展和复杂 UI。

如果这个闭环不可靠，系统功能越多，体验越差。

## 第一阶段必须做硬的能力

### 1. 任务状态必须可信

系统不能把“等待用户授权”显示成“已完成”。

每个状态都要有明确含义：

- `pending`：尚未开始
- `running`：正在真实执行
- `waiting_user_action`：等待用户动作
- `resuming`：用户动作完成后正在恢复
- `verifying`：正在验证结果
- `completed`：已完成并有验证结果
- `failed`：失败，并给出失败证据
- `timeout`：等待超时，需要重新发起

### 2. 人工操作必须可恢复

例如 `/lark-cli 登录` 这类任务，不能只展示授权链接。

正确流程应该是：

```text
发起授权
-> 保存 device_code、task_id、原始用户目标
-> 等待用户在浏览器授权
-> 授权完成后自动检测登录状态
-> 登录有效后继续原始任务
-> 输出最终结果
```

如果用户授权完成后系统不能自动继续，这个能力就不算完成。

### 3. 工具执行必须可验证

系统不能只输出“我已经完成”。

关键结果必须至少包含一种验证证据：

- 命令输出
- API 返回
- 测试结果
- 文件变更
- 日志片段
- 构建结果
- 截图或运行状态
- 人工确认记录

### 4. 项目上下文必须统一

以下状态不能分裂：

- `project_id`
- `chat_session_id`
- `session_id`
- 任务树
- requirement 本地对象
- 工作事实
- 项目记忆
- 交付记录

同一个用户目标只能有一条清晰的任务线索。

### 5. 前端必须围绕执行状态设计

前端第一屏应优先回答这些问题：

- 当前目标是什么
- 当前执行到哪一步
- 当前是否卡住
- 如果卡住，需要用户做什么
- 用户做完以后系统会继续什么
- 哪些步骤已真实验证
- 哪些只是模型总结

运行轨迹不能把“过程消息”包装成“最终结果”。

## 第一阶段暂不做

为了避免继续变成大杂烩，第一阶段暂时不优先做：

- 员工市场
- 复杂技能广场
- 多平台大规模接入
- 替代 Codex / Hermes 的底层 Agent Loop
- 大量新增视觉模块
- 复杂权限矩阵
- 面向所有场景的通用自动化

这些能力可以作为第二阶段或第三阶段扩展，但不能压过主闭环。

## MVP 场景

最小可行产品用一个场景验证：

```text
用户输入：修复飞书登录后不自动继续的问题
```

系统必须展示：

- 当前任务树
- 正在执行的真实命令或工具
- 等待用户授权的明确提示
- 用户授权完成后的自动续跑记录
- 登录状态验证结果
- 修复改动
- 测试或验证结果
- 交付总结
- 本次经验沉淀

如果这个场景跑通，说明系统具备项目执行平台的核心价值。

## 成功标准

第一阶段成功标准：

- 用户能把一个真实项目任务交给系统
- 系统能生成可信任务树
- 每一步都能看到真实执行证据
- 人工等待不会被误报为完成
- 用户动作完成后系统能自动恢复
- 最终交付有验证记录
- 后续会话能恢复任务和经验

## 产品口号

推荐口号：

```text
让 AI 不只是回答，而是交付。
```

或：

```text
把 AI 员工组织成一个真正能完成项目的执行系统。
```

## 后续路线

### 阶段 1：执行闭环

聚焦任务树、工具执行、人工等待、自动恢复、验证和交付记录。

### 阶段 2：外部 Agent 编排

接入 Codex、Hermes、Claude Code、本地 MCP 和项目工具，把它们作为可调度执行员工。

### 阶段 3：员工与技能体系

沉淀员工角色、技能包、经验规则和项目级能力复用。

### 阶段 4：团队化与平台化

扩展多人协作、权限、审计、飞书/企业微信/消息平台、长期任务和运营面板。

## 当前最重要的设计原则

状态真实比功能丰富更重要。

自动恢复比一次性执行更重要。

验证结果比模型总结更重要。

项目上下文统一比到处兼容更重要。

先把一个任务做完整，再让系统看起来强大。
