## 核心结论

机器人不应只作为“第三方平台消息接收器”或“客服自动回复入口”。机器人要成为 AI 对话页的远程操作入口。

用户在 AI 对话页能看到、能选择、能继续、能授权的项目执行能力，机器人也应能获取和操作同一份状态：

```text
AI 对话页
  -> 项目
  -> 对话窗口 / 任务窗口
  -> chat_session_id / session_id / requirement
  -> 任务树
  -> 当前进行中的任务
  -> 外部智能体 / 本机 Runner
  -> 审批队列
  -> 文件、日志、验证和交付

机器人
  -> 同一套项目
  -> 同一套任务窗口
  -> 同一套状态查询
  -> 同一套外部智能体执行
  -> 同一套授权审批
  -> 同一套交付记录
```

也就是说，机器人不是另一套执行系统。机器人只是把飞书、QQ、微信等平台消息映射到项目执行主链。

## 用户目标

用户希望在群里或私聊里完成这些操作：

- 选择项目。
- 查看当前项目有哪些正在进行的任务。
- 查看当前任务执行到哪一步。
- 创建一个新的任务会话窗口。
- 切换到已有任务会话窗口。
- 像 AI 对话页一样发起开发需求。
- 强制使用项目配置的外部智能体执行开发任务。
- 通过机器人完成命令、写入、部署、登录、授权等审批。
- 让机器人控制本机 Runner 继续执行，而不是只返回文字建议。
- 获取 AI 对话页中用户自己可以获取的信息，包括任务树、运行日志、文件变更、验证结果、阻塞原因和交付摘要。

## 产品边界

### 机器人可以做什么

机器人可以：

- 读取用户在 AI 对话页有权限查看的项目、任务和运行状态。
- 在用户授权范围内创建新的项目任务会话。
- 将平台消息写入对应 `chat_session_id` 的项目对话历史。
- 触发外部智能体 Runner 执行。
- 展示当前 Runner 状态、等待用户动作、审批请求和运行结果。
- 将用户在平台上的按钮操作转成系统审批决定。
- 将最终交付、验证结果和风险同步回平台。

### 机器人不能绕过什么

机器人不能：

- 绕过项目权限读取别人的项目或任务。
- 绕过 AI 对话页的任务树、requirement、工作会话和审计记录。
- 直接让 Docker 服务端控制用户 Mac / Windows。
- 在本机 Runner 不在线时假装已经执行。
- 在未配置外部智能体时执行开发任务。
- 在未授权时执行写文件、删除、部署、发外部消息等高风险动作。
- 把“已发送给外部智能体”当作“任务已完成”。

## 与 AI 对话页的能力等价

机器人要等价的不是 UI，而是状态和操作能力。

| AI 对话页能力 | 机器人等价能力 |
| --- | --- |
| 选择项目 | 机器人命令或卡片选择项目 |
| 新对话 | 创建新任务窗口 |
| 切换对话 | 切换当前任务窗口 |
| 查看历史对话 | 查看任务窗口列表和摘要 |
| 查看任务树 | 返回当前任务树和当前节点 |
| 查看正在进行任务 | 返回运行中任务、等待审批任务、Runner 会话 |
| 选择外部 Agent | 使用项目配置或机器人卡片选择执行器 |
| 发送开发需求 | 平台消息进入 external_agent 执行链 |
| 查看 Runner 日志 | 机器人返回摘要、可选展开日志片段 |
| 审批命令 | 机器人按钮批准一次、本会话批准、拒绝 |
| 查看文件变更 | 返回改动文件列表和 diff 摘要 |
| 查看验证结果 | 返回测试、构建、命令输出或人工验证 |
| 继续任务 | 发送“继续”或点击继续按钮恢复同一任务 |
| 取消任务 | 发送取消或点击取消按钮终止 Runner |

## 任务窗口模型

机器人必须支持任务窗口，而不是把一个群永远绑定到一个会话。

### 三层会话

```text
平台会话
  connector_id + platform + external_chat_id + thread_id

项目任务窗口
  project_id + chat_session_id

工作执行会话
  session_id + runner_session_id + task_tree_id
```

平台会话只表示消息从哪里来；项目任务窗口才表示用户当前在处理哪个需求；工作执行会话表示外部智能体或 Runner 正在执行哪一次任务。

### 创建新任务窗口

用户可以通过这些方式创建新任务窗口：

```text
/新任务 修复登录页错误
/新对话
/创建任务
```

系统行为：

- 生成新的 `chat_session_id`。
- 创建新的 requirement。
- 创建或绑定新的任务树。
- 创建新的工作会话 `session_id`。
- 将平台来源写入 `source_context`。
- 将当前平台会话的 active task window 指向新窗口。

### 切换任务窗口

用户可以通过这些方式切换：

```text
/任务列表
/切换任务 3
/切换 chat-session-xxx
```

机器人返回任务窗口列表时，必须展示：

- 序号。
- 项目名。
- 任务标题。
- 当前状态。
- 当前节点。
- 是否有 Runner 正在运行。
- 是否等待审批。
- 最近更新时间。

切换后，用户的普通消息默认进入新的 active task window。

### 防止任务串线

机器人处理普通消息前必须解析 active task window。

如果当前平台会话没有 active task window：

- 私聊：提示用户选择项目或创建任务。
- 群聊：只在 @ 机器人或命令触发时响应。
- 如果机器人配置有默认项目，可以创建临时项目任务窗口，但必须在回复里明确当前项目。

如果消息看起来是新任务，但当前已有运行中任务：

- 不应直接覆盖当前窗口。
- 应返回“继续当前任务 / 创建新任务 / 切换任务”的选择卡片。

## 当前正在进行任务查询

机器人必须支持查询当前正在进行的任务，这是远程入口的第一优先级能力。

### 查询命令

建议支持：

```text
/当前任务
/进行中
/任务状态
/审批
/Runner
```

### 查询范围

机器人要按用户权限返回：

- 当前平台会话绑定的 active task window。
- 当前项目所有进行中任务。
- 当前用户相关任务。
- 当前等待用户审批的任务。
- 当前本机 Runner 运行中的会话。
- 最近可恢复的 blocked / waiting_user_action 任务。

### 返回内容

最小返回格式：

```text
当前任务：修复登录页服务端地址配置
项目：ai员工工厂
状态：waiting_user_action
当前节点：等待命令审批
执行器：Codex CLI
Runner：running
审批：需要批准一次命令
最近事件：准备执行 npm run build
可操作：批准一次 / 本会话批准 / 拒绝 / 打开详情 / 取消
```

如果有多个任务：

```text
进行中任务：
1. 修复登录页服务端地址配置 · 等待审批 · Codex CLI
2. 接入飞书机器人任务窗口 · 执行中 · Hermes
3. 登录页动画优化 · 已验证待归档
```

## 项目信息获取范围

“用户自己在 AI 对话页能获取的信息，机器人也可以获取到”必须落成权限规则。

机器人可读信息包括：

- 项目列表和项目详情摘要。
- 项目成员和当前选中员工。
- 项目规则、技能、手册摘要。
- 项目对话历史。
- 当前 `chat_session_id` 的任务树。
- 当前 requirement。
- 当前工作会话。
- 运行中 Runner 会话。
- 最近 Runner 日志摘要。
- 等待审批请求。
- 文件变更摘要。
- 验证证据。
- 交付报告。

机器人不可直接暴露：

- 用户无权限访问的项目。
- 原始密钥、token、cookie、App Secret。
- 未脱敏的敏感日志。
- 超出当前项目工作区的本机文件。
- 其他平台会话中的私聊内容，除非用户本身在 AI 对话页也能看到。

## 外部智能体前置要求

机器人发起开发、修复、执行命令、改文件等任务前，必须检查：

- 项目已选择。
- 项目有工作区。
- 项目配置了外部智能体类型：`codex_cli`、`hermes` 或 `claude_code`。
- 桌面端或本机 Runner 在线。
- Runner 对该项目工作区可访问。
- 当前用户有执行权限。
- 当前平台会话已经绑定任务窗口。

如果缺失外部智能体配置，机器人应返回：

```text
当前项目还不能通过机器人执行开发任务。
缺少：外部智能体配置。
请在项目 AI 对话页或项目设置中选择 Codex CLI / Hermes / Claude Code，并完成本机 Runner 检测。
```

## 执行链路

### 用户发起开发任务

```text
飞书群 @机器人 修复登录页按钮 loading 问题
  -> 机器人解析平台来源
  -> 读取 active task window
  -> 如无窗口，引导选择项目或创建新任务
  -> 检查项目外部智能体和 Runner
  -> 创建/恢复 chat_session_id + session_id + requirement
  -> 生成任务树
  -> 构造 external_agent 任务包
  -> 投递给本机 Runner
  -> Runner 启动 Codex / Hermes / Claude Code
  -> 事件回写项目执行主链
  -> 机器人持续返回关键状态
```

### Runner 返回事件

Runner 事件要映射为平台可读消息：

- `started`：机器人提示任务已开始。
- `tool_call`：只摘要关键命令，不刷屏。
- `waiting_user_action`：发送审批卡片。
- `file_changed`：返回文件变更摘要。
- `verifying`：提示正在验证。
- `completed`：返回交付摘要。
- `failed`：返回失败原因和下一步操作。

## 授权审批模型

机器人授权要对齐 AI 对话页和本机 Runner 的审批队列。

### 审批类型

至少支持：

- 命令审批。
- 写文件审批。
- 删除/覆盖审批。
- 部署/发布审批。
- 外部系统写入审批。
- 登录/授权等待。
- 验证码或人工输入等待。

### 平台交互

飞书第一阶段使用交互卡片。

按钮：

- 批准一次。
- 本会话批准。
- 拒绝。
- 查看详情。
- 取消任务。

点击按钮后：

- 写入权限决定。
- 记录到工作会话。
- 更新任务树节点状态。
- 通知 Runner 继续或终止。
- 回发操作结果。

### 审批记录

每条审批必须保留：

- `project_id`
- `chat_session_id`
- `session_id`
- `runner_session_id`
- `connector_id`
- `external_chat_id`
- `request_id`
- `decision`
- `decision_scope`
- `operator_user_id`
- `command_or_action`
- `risk_level`
- `created_at`
- `decided_at`

## 机器人命令设计

第一阶段建议支持这些命令：

| 命令 | 作用 |
| --- | --- |
| `/项目列表` | 查看可选择项目 |
| `/选择项目 <名称或序号>` | 设置当前平台会话默认项目 |
| `/新任务 <目标>` | 创建新的任务窗口并开始处理 |
| `/任务列表` | 查看当前项目或当前用户相关任务窗口 |
| `/切换任务 <序号>` | 切换 active task window |
| `/当前任务` | 查看当前任务窗口状态 |
| `/进行中` | 查看进行中的任务 |
| `/审批` | 查看等待审批的请求 |
| `/继续` | 恢复当前等待任务 |
| `/取消任务` | 取消当前 Runner 或任务窗口 |
| `/交付` | 查看当前任务交付摘要 |
| `/帮助` | 查看机器人可用命令 |

普通自然语言消息：

- 如果 active task window 存在，默认进入当前任务窗口。
- 如果没有 active task window，机器人先引导选择项目或创建任务。
- 如果识别为新任务，提示创建新任务窗口，避免覆盖旧任务。

## 与 Hermes 的参考关系

Hermes 的参考价值在于 gateway 和 approval 模型：

- 消息平台不是聊天 UI，而是 Agent 入口。
- 会话需要保存来源、目标、线程和上下文。
- 审批要有 pending approval 和 respond approval 的结构。
- 平台按钮比纯文本 `/approve` 更可靠。
- Agent 执行和消息投递要解耦。

ai员工工厂不需要复制 Hermes 的项目状态模型。项目状态仍由 ai员工工厂掌握：

```text
Hermes 参考：平台 gateway、approval buttons、session routing
ai员工工厂保留：project_id、chat_session_id、session_id、requirement、任务树、工作事实、交付报告
```

## 分阶段落地

### 阶段 1：机器人任务窗口和状态查询

目标：机器人先能像 AI 对话页一样看见当前任务。

工作项：

- 增加平台会话到 active task window 的绑定表。
- 支持选择项目、创建新任务、切换任务。
- 支持 `/当前任务`、`/进行中`、`/任务列表`。
- 返回任务树、当前节点、工作会话、Runner 状态和审批状态。
- 机器人查询必须按用户权限过滤。

验收标准：

- 用户能在飞书里创建新的项目任务窗口。
- 用户能切换任务窗口。
- 用户能看到当前正在进行的任务。
- 同一群里多个任务不会串线。

### 阶段 2：机器人接入 external_agent 执行链

目标：机器人发起的开发任务和 AI 对话页同链路执行。

工作项：

- 将机器人消息转换成项目对话请求。
- 开发类任务强制 `chat_mode=external_agent`。
- 复用 AI 对话页的任务包结构。
- 检查外部智能体、工作区和 Runner 在线状态。
- 将 Runner 事件回写到当前任务窗口。

验收标准：

- 从飞书发起开发任务后，AI 对话页能看到同一条消息、任务树和 Runner 会话。
- AI 对话页创建的任务，机器人能查询和继续。
- 机器人和 AI 对话页看到的状态一致。

### 阶段 3：机器人授权审批

目标：机器人能承担远程审批入口。

工作项：

- 增加审批请求服务端队列。
- Runner 等待审批时推送平台卡片。
- 平台按钮回调写入权限决定。
- Runner 根据审批继续、拒绝或取消。
- 审批记录进入工作会话和任务树。

验收标准：

- 命令审批能在飞书卡片里批准一次。
- 本会话批准能在当前 Runner 会话内生效。
- 拒绝后任务状态显示 blocked 或 cancelled。
- 审批记录在 AI 对话页和机器人里都可追溯。

### 阶段 4：文件、日志和交付等价

目标：机器人获取 AI 对话页可见的执行结果。

工作项：

- 返回文件变更摘要。
- 返回关键日志和命令输出摘要。
- 返回验证证据。
- 返回交付报告。
- 支持打开详情链接到 AI 对话页对应任务窗口。

验收标准：

- 用户能从机器人看到任务是否真的执行、验证和交付。
- 机器人不会把内部长日志刷屏到群里。
- 详情可回到 AI 对话页继续查看。

### 阶段 5：扩展 QQ / 微信

目标：在飞书闭环稳定后扩展其他平台。

工作项：

- 抽象平台卡片能力。
- 对不支持按钮的平台提供安全文本确认协议。
- 复用任务窗口、状态查询和审批服务。

验收标准：

- QQ / 微信不需要重写项目执行主链。
- 平台差异只存在于消息和按钮适配层。

## 需要改造的模块

### 后端

- 机器人连接器服务：增加默认项目、执行模式和外部智能体要求。
- 平台会话绑定服务：保存 active project 和 active task window。
- 项目聊天入口：允许机器人以平台来源触发 external_agent。
- 任务状态查询 API：按平台用户和项目权限返回进行中任务。
- Runner 状态聚合 API：返回本机 Runner、外部智能体和审批队列。
- 审批服务：统一 AI 对话页、本机 Runner 和机器人卡片审批。
- 飞书事件服务：处理命令、任务窗口、审批按钮和执行状态回推。

### 前端

- AI 对话页需要能显示“该任务来自机器人”。
- 任务窗口列表要展示平台来源。
- Runner 详情要能生成机器人可引用的详情链接。
- 机器人配置页增加执行能力配置：
  - 默认项目。
  - 是否允许创建任务。
  - 是否允许 external_agent 执行。
  - 可用执行器。
  - 审批策略。

### 桌面端 / Runner

- Runner 需要支持服务端下发来自机器人的任务。
- Runner 需要按 `project_id + chat_session_id + session_id` 上报事件。
- Runner 审批等待必须可被服务端转成平台卡片。
- Runner 不在线时，机器人必须返回明确阻塞状态。

## 风险与约束

### 风险 1：机器人绕过 AI 对话页状态

如果机器人自己维护一套对话和任务状态，会导致 AI 对话页看不到真实执行。

规避方式：

- 机器人只能写入项目执行主链。
- 所有任务必须有 `chat_session_id` 和 `session_id`。
- 任务树和 requirement 是 canonical 状态。

### 风险 2：群消息导致任务串线

同一个群可能同时讨论多个任务。

规避方式：

- 普通消息进入 active task window。
- 新任务必须显式创建或确认。
- 机器人展示当前 active task window。
- 支持按线程隔离任务窗口。

### 风险 3：服务端误以为能控制本机

Docker 服务端不能直接控制用户电脑。

规避方式：

- 所有电脑控制必须通过桌面端或本机 Runner。
- Runner 不在线时任务进入 blocked。
- 机器人只负责触发、查询、审批和反馈。

### 风险 4：审批过于随意

群里按钮审批可能被不该审批的人点击。

规避方式：

- 审批按钮回调必须校验平台用户身份。
- 只有项目授权用户或发起人可审批。
- 高风险操作必须展示命令、工作区、风险原因和作用范围。
- 所有审批写入审计。

## 第一阶段最小验收场景

```text
1. 用户在飞书群输入：/选择项目 ai员工工厂
2. 用户输入：/新任务 修复登录页按钮 loading 问题
3. 系统创建新的 chat_session_id / session_id / requirement
4. 系统生成任务树
5. 系统检查项目已配置 Codex CLI 和本机 Runner
6. Runner 开始执行
7. 用户输入：/当前任务
8. 机器人返回当前节点、Runner 状态、最近事件
9. Runner 请求执行 npm run build
10. 机器人发送审批卡片
11. 用户点击“批准一次”
12. Runner 继续执行
13. 机器人返回文件变更、验证结果和交付摘要
14. AI 对话页能看到同一任务窗口和同一状态
```

## 推荐优先级

1. 先做飞书。
2. 先做任务窗口和当前任务查询。
3. 再接 external_agent 执行链。
4. 再做审批卡片。
5. 再做文件、日志和交付摘要。
6. 最后扩展 QQ / 微信。

## 一句话路线图

先让机器人看见 AI 对话页同一套项目任务状态，再让机器人创建和切换任务窗口，随后把机器人消息接入 external_agent 和本机 Runner，最后用平台按钮完成授权审批，让机器人成为真实项目执行的远程控制入口。
