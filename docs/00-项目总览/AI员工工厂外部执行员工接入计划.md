# AI员工工厂外部执行员工接入计划

> 生成时间：2026-06-06
> 项目：ai员工工厂 / proj-d16591a6
> 关联文档：`docs/00-项目总览/AI项目执行系统目标定义.md`

## 核心结论

ai员工工厂后续方向不是再做一个 Codex、Claude Code 或 Hermes，而是把它们纳入统一调度。

更准确的定位是：

```text
Codex / Claude Code / Hermes = 外部执行员工
LlmProviderService = 轻量 AI 能力网关
Agent Runtime + 任务树 + 工作会话 = 项目执行主链
ai员工工厂 = 项目经理 + 调度系统 + 状态中枢 + 交付系统
```

也就是说，用户把真实项目目标交给 ai员工工厂；系统负责拆任务、选执行器、跟踪状态、等待人工动作、恢复续跑、验证结果和输出交付报告。Codex、Claude Code、Hermes 负责承担其中具体的执行工作。

## 背景

当前系统已经存在多类 AI 调用：

- 项目聊天、工具循环、任务执行中的模型调用。
- 总结、分类、润色、命名、草稿生成等轻量 AI 能力。
- 图片、视频、语音合成、语音转写等多模态能力。
- 本地 connector 转发的模型能力。
- 前端已有 `external_agent` / `codex_cli` 概念，后端也存在 Codex runner 雏形。

这些能力目前还没有完全收敛成一条清晰的“项目执行主链”。如果继续在各个业务入口零散新增 AI 调用，系统会越来越像普通 AI 功能集合，而不是项目级执行系统。

本计划的目标是把 AI 调用分层，并把真正会改变项目状态的任务统一收敛到外部执行员工机制。

## 能力分层

### 1. 轻量 AI 能力

轻量 AI 能力指不需要真实接管项目执行、不需要长时间状态管理、不需要操作文件或外部系统的模型调用。

典型场景：

- 意图分类。
- 内容总结。
- 文案润色。
- 标题生成。
- 结构化抽取。
- 反馈反思。
- 员工草稿生成。
- 模板命名和去重判断。
- 语音转文字。
- 文字转语音。
- 图片、视频等媒体生成。

这些能力继续走 `LlmProviderService` 或本地 connector，不需要派发给 Codex、Claude Code 或 Hermes。

### 2. 项目执行能力

项目执行能力指会影响项目状态、需要多步推进、需要验证和恢复的任务。

典型场景：

- 修改代码。
- 修复 bug。
- 新增功能。
- 执行测试。
- 构建或部署。
- 操作飞书、数据库、MCP、脚本或第三方系统。
- 读取项目状态后持续推进。
- 遇到登录、授权、验证码、确认等人工动作后恢复续跑。

这些任务应该进入 Agent Runtime、任务树和工作会话，再由系统选择合适的外部执行员工。

## 目标架构

```text
用户目标
  -> 项目入口
  -> 需求对象 / chat_session_id / session_id
  -> 任务树
  -> 调度器选择执行员工
  -> 外部执行员工执行
  -> 过程事件 / 文件改动 / 命令日志 / 工具结果
  -> 验证器
  -> 交付报告
  -> 项目记忆与经验沉淀
```

### 主链职责

ai员工工厂负责：

- 保存用户原始目标。
- 创建或恢复 requirement 记录。
- 生成任务树。
- 判断任务类型和风险。
- 选择执行员工。
- 提供项目上下文和可执行边界。
- 维护运行状态。
- 处理人工等待和恢复续跑。
- 收集执行证据。
- 执行验证。
- 生成交付报告。
- 沉淀项目记忆和经验规则。

外部执行员工负责：

- 在受控工作区内执行具体任务。
- 按协议回传过程事件。
- 返回文件变更、命令输出、测试结果和失败原因。
- 在需要人工动作时明确暂停。
- 支持恢复同一个任务。

## 外部执行员工类型

### Codex

定位：代码执行员工。

适用场景：

- 代码修改。
- 测试修复。
- 代码库结构分析。
- 小到中型功能实现。
- 工程化验证。

接入方式：

- 通过外部 runner 或本地 connector 启动 Codex。
- 传入项目路径、任务目标、任务节点、权限策略和上下文文件。
- 接收流式执行日志、文件改动摘要和验证结果。

### Claude Code

定位：代码分析与实现员工。

适用场景：

- 复杂代码理解。
- 大范围重构前分析。
- 多文件实现。
- 文档化设计和代码解释。

接入方式：

- 与 Codex 共用外部执行员工协议。
- 区分执行器类型、命令入口、上下文注入方式和输出解析器。
- 在任务配置中显式记录实际模型或执行器版本。

### Hermes

定位：多工具 Agent 框架或外部多平台代理执行器。

适用场景：

- 多平台工具编排。
- 跨工具自动化。
- 复杂 Agent 流程实验。
- 需要复用 Hermes 生态能力的任务。

接入方式：

- 不把 ai员工工厂改造成 Hermes。
- 通过标准执行员工协议调用 Hermes 能力。
- 可借鉴 Hermes 的 tool registry、skill boundary、runtime capability 设计，但项目主状态仍由 ai员工工厂掌握。

## 标准执行员工协议

外部执行员工必须通过统一协议接入，避免每个执行器各写一套状态逻辑。

### 任务输入

```json
{
  "project_id": "proj-d16591a6",
  "chat_session_id": "...",
  "session_id": "...",
  "requirement_id": "...",
  "task_tree_id": "...",
  "task_node_id": "...",
  "executor_type": "codex_cli",
  "workspace_path": "/path/to/workspace",
  "sandbox_mode": "workspace-write",
  "user_goal": "用户原始目标",
  "node_goal": "当前节点目标",
  "context_files": [],
  "allowed_tools": [],
  "risk_policy": {
    "require_confirmation_for_destructive_actions": true,
    "network_access": "restricted"
  }
}
```

### 过程事件

执行员工应持续回传结构化事件：

- `started`：已开始。
- `thinking`：分析过程摘要。
- `tool_call`：准备调用工具或命令。
- `tool_result`：工具结果。
- `file_changed`：文件变更。
- `waiting_user_action`：等待用户动作。
- `resuming`：恢复执行。
- `verifying`：正在验证。
- `completed`：完成。
- `failed`：失败。

### 任务输出

```json
{
  "status": "completed",
  "summary": "完成了什么",
  "changed_files": [],
  "commands": [],
  "verification": [],
  "risks": [],
  "next_steps": []
}
```

输出必须能被任务树、工作事实、交付报告和项目记忆复用。

## 状态机要求

外部执行员工接入后，任务状态不能只靠聊天文本判断。

建议统一状态：

- `pending`：尚未开始。
- `in_progress`：正在执行。
- `waiting_user_action`：等待用户动作。
- `resuming`：用户动作完成后恢复。
- `verifying`：正在验证。
- `completed`：已完成且有验证证据。
- `failed`：失败且有失败证据。
- `cancelled`：用户取消。
- `timeout`：等待或执行超时。

关键原则：

- 等待用户授权不能显示为完成。
- 未写入验证结果不能标记完成。
- 外部执行器异常退出必须有错误事件。
- 中断恢复必须复用同一条 `chat_session_id` 和 `session_id`。
- 文件变更、命令输出、测试结果必须能追溯到具体任务节点。

## 调度策略

调度器应先判断任务类型，再选择执行员工。

建议规则：

| 任务类型 | 推荐执行路径 |
| --- | --- |
| 总结、润色、分类、命名 | LlmProviderService |
| 语音、图片、视频 | LlmProviderService 多模态能力 |
| 代码修复、功能实现、测试 | Codex 或 Claude Code |
| 大范围代码理解、重构设计 | Claude Code 或 Codex |
| 多工具、跨平台、自动化流程 | Hermes 或 MCP 工具员工 |
| 飞书、数据库、部署脚本 | MCP / 本地工具 / 外部执行员工 |
| 高风险删除、覆盖、发布 | 先人工确认，再执行 |

调度决策必须记录：

- 为什么选择这个执行器。
- 为什么没有选择其他执行器。
- 当前权限和风险边界。
- 是否需要人工确认。
- 如何验证执行结果。

## 分阶段落地计划

### 阶段 1：盘点并收敛 AI 调用边界

目标：明确哪些调用继续走模型网关，哪些必须进入项目执行主链。

工作项：

- 建立 AI 调用清单。
- 标记每个调用的类型：轻量 AI / 项目执行 / 多模态 / 外部 Agent。
- 对项目执行类入口增加任务树和工作会话要求。
- 禁止新的项目执行能力绕过 Agent Runtime 直接调用模型。

验收标准：

- 能列出系统所有主要 AI 调用入口。
- 每个入口都有明确归属层级。
- 新增执行型能力有统一接入规范。

### 阶段 2：定义外部执行员工协议

目标：先稳定协议，再接入具体执行器。

工作项：

- 定义执行任务输入结构。
- 定义过程事件结构。
- 定义执行结果结构。
- 定义错误、暂停、恢复和取消语义。
- 定义文件变更和验证结果的记录格式。

验收标准：

- Codex、Claude Code、Hermes 可以共用同一套任务协议。
- 任务树能消费外部执行事件。
- 工作事实能记录执行证据。

### 阶段 3：Codex CLI 作为第一个执行员工

目标：把 Codex 接成可用的代码执行员工。

工作项：

- 明确 Codex runner 的启动、健康检查和权限策略。
- 接通 `external_agent` 到后端执行链路。
- 支持流式日志回传。
- 支持文件变更摘要。
- 支持命令和测试结果回传。
- 支持失败后保留上下文。

验收场景：

```text
用户输入：修复某个明确 bug
系统生成任务树
调度 Codex 执行
Codex 修改代码并运行验证
系统展示变更文件和验证结果
任务树节点完成
交付报告生成
```

### 阶段 4：Claude Code 接入同一协议

目标：验证协议不是 Codex 专用。

工作项：

- 新增 `executor_type=claude_code`。
- 复用任务输入、事件和输出协议。
- 单独实现启动命令、环境检查和输出解析。
- 在调度策略中支持手动选择和自动选择。

验收标准：

- 同一个任务可以在 Codex 和 Claude Code 之间切换。
- 任务树、状态、验证、交付报告不依赖具体执行器。
- 日志能明确记录实际执行器。

### 阶段 5：Hermes 作为多工具执行员工

目标：让 Hermes 承担多工具、多平台 Agent 任务，而不是替代 ai员工工厂主链。

工作项：

- 定义 Hermes executor adapter。
- 约束 Hermes 只负责执行，不掌握项目最终状态。
- 将 Hermes 工具事件映射到统一过程事件。
- 将 Hermes 输出映射到任务树验证和交付报告。

验收标准：

- Hermes 可以执行跨工具任务。
- ai员工工厂仍能看到任务状态、暂停点、恢复点和验证结果。
- Hermes 失败时不会导致项目状态丢失。

### 阶段 6：执行器选择与治理

目标：从“能调用”升级为“可治理、可追溯、可恢复”。

工作项：

- 为不同任务配置默认执行器策略。
- 增加人工选择执行器入口。
- 增加执行器能力表和健康状态。
- 增加执行器级别权限和风险策略。
- 增加执行成本、耗时、成功率、失败原因统计。

验收标准：

- 用户能知道当前任务由谁执行。
- 系统能解释为什么选择该执行器。
- 执行器失败后能切换或重试。
- 高风险操作必须有人工确认记录。

## 需要改造的系统模块

### 后端

- `LlmProviderService`：继续作为轻量 AI 与多模态模型网关。
- `Agent Runtime v2`：作为项目执行主链。
- `project_chat_execution_service`：负责项目聊天入口到执行主链的衔接。
- `local_connector_service`：作为本地模型和外部执行能力接入点之一。
- `agent_runner`：优先打磨为 Codex 外部执行员工 runner。
- 任务树服务：消费外部执行事件并推进节点状态。
- 工作事实和项目记忆：记录执行证据、验证结果和交付结论。

### 前端

- 项目聊天中明确区分“模型回复”和“执行员工运行中”。
- 外部执行员工需要有状态展示：启动中、执行中、等待用户、验证中、失败、完成。
- 展示执行器类型：Codex、Claude Code、Hermes、本地工具或 MCP。
- 展示关键证据：文件变更、命令输出、测试结果、人工等待说明。
- 用户可以在任务开始前或失败后切换执行器。

### 配置

- 执行器注册表。
- 执行器启动命令。
- 工作区权限策略。
- 网络访问策略。
- 高风险操作确认策略。
- 默认执行器选择策略。
- 日志和审计保留策略。

## 风险与约束

### 风险 1：把外部执行员工当成普通模型调用

如果只把 Codex、Claude Code、Hermes 包成一个聊天接口，系统仍然无法可靠处理文件变更、验证、等待用户和恢复续跑。

规避方式：

- 必须走任务协议。
- 必须有过程事件。
- 必须有验证结果。
- 必须绑定任务树节点。

### 风险 2：项目状态被外部执行器接管

Hermes 或 Codex 可以执行任务，但不能成为项目状态源。

规避方式：

- ai员工工厂保存 canonical 状态。
- 外部执行器只返回事件和结果。
- 任务树、工作事实、记忆、交付报告由 ai员工工厂写入。

### 风险 3：执行器差异导致协议失控

不同执行器输出格式、命令能力和权限模型不同。

规避方式：

- 先定义最小通用协议。
- 每个执行器只写 adapter。
- 不让业务入口直接依赖具体执行器输出。

### 风险 4：任务树继续模板化

外部执行员工接入后，如果任务树仍然只有“分析、实现、验证”，用户仍然看不清真实执行过程。

规避方式：

- 任务树必须按任务类型差异化生成。
- 执行员工事件必须反向补充任务树状态。
- 交付报告必须引用真实节点和真实证据。

## 验收标准

第一阶段最小验收：

- 系统能明确区分轻量 AI 调用和项目执行调用。
- Codex CLI 能作为外部执行员工被调度。
- 外部执行过程能回写任务树节点状态。
- 文件变更、命令输出和验证结果能进入交付报告。
- 等待人工动作不会被误报为完成。
- 中断后能通过同一 `chat_session_id` 和 `session_id` 恢复。

第二阶段验收：

- Claude Code 可以复用同一协议接入。
- 用户可以选择 Codex 或 Claude Code 执行同一类代码任务。
- 执行器选择、执行日志和验证结果可追溯。

第三阶段验收：

- Hermes 可以作为多工具执行员工接入。
- Hermes 执行过程能映射到 ai员工工厂任务树。
- ai员工工厂仍然掌握项目状态、恢复和交付闭环。

## 推荐优先级

优先级从高到低：

1. 收敛 AI 调用分层。
2. 定义外部执行员工协议。
3. 打通 Codex CLI runner。
4. 将任务树状态与外部执行事件绑定。
5. 完成交付报告和验证证据闭环。
6. 接入 Claude Code。
7. 接入 Hermes。
8. 做执行器选择策略、统计和治理能力。

## 一句话路线图

先保留 `LlmProviderService` 做轻量 AI 能力，再把真实项目执行统一交给 Agent Runtime 管理，最后把 Codex、Claude Code、Hermes 接成标准外部执行员工，由 ai员工工厂统一调度、跟踪、验证和交付。
