# 需求文档清单（独立模块）

## 1. 模块定位

- 名称：反馈驱动规则升级模块（Feedback Rule Upgrade Module）
- 形态：独立可插拔模块
- 启用方式：管理员手动选择启用（按项目、按员工）
- 默认策略：默认关闭，灰度启用

## 2. 需求范围清单

- `R1` 反馈采集
- `R2` AI 反思分析（直接原因 + 深层原因）
- `R3` 候选规则生成（非直接上线）
- `R4` 人工审核（通过/编辑后通过/拒绝）
- `R5` 规则版本发布与回滚
- `R6` 审计与追踪（全链路留痕）
- `R7` 指标看板（反馈处理时效、升级采纳率、回归率）
- `R8` 可配置风控（按风险域阈值与审批等级）
- `R9` 项目级隔离（当前项目反馈与反思结果单独存储）
- `R10` 前后端菜单与页面闭环（反馈录入、审核、版本追踪）

## 3. 交付物清单

- `D1` PRD 文档（本目录）
- `D2` 数据模型设计（反馈、分析、候选、审核、发布）
- `D3` API 设计（反馈提交、候选查询、审核动作、发布动作）
- `D4` 前后端设计文档（菜单、路由、页面、分层）
- `D5` 风控规则配置清单
- `D6` 监控指标定义与埋点清单
- `D7` 上线计划（灰度、回滚、SOP）

## 4. 验收标准清单

- `A1` 可在员工配置页手动启用/停用模块。
- `A2` 用户可提交结构化 bug 反馈，支持关联会话与规则。
- `A3` AI 可输出标准化反思报告（含证据链）。
- `A4` 规则升级必须先进入“待审候选”，不能绕过审核直接生效。
- `A5` 审核动作全量留痕，支持按人、按时间、按规则追溯。
- `A6` 发布后产生新规则版本，可一键回滚旧版本。
- `A7` 可查看升级前后质量指标对比（至少包含采纳率和回归率）。
- `A8` 反馈、反思、候选、审核数据均带 `project_id` 且无法跨项目访问。
- `A9` 前端提供可访问菜单与页面链路（反馈录入、反馈详情、候选审核、版本记录）。

## 5. 依赖与约束

- 依赖现有模块：
- `web-admin/api/routers/evolution.py`
- `mcp-evolution`
- `mcp-rules`
- `web-admin/frontend/src/views/evolution`
- 约束：
- 不影响现有 API 行为（向后兼容）
- 高风险规则必须人工审批
- 所有自动决策必须可解释、可审计
- 所有反馈链路写入必须强制绑定当前项目

## 6. 里程碑建议

- `M1` 文档冻结：需求、数据模型、接口定义确认
- `M2` 后端最小可用：反馈入库 + 候选生成 + 审核流转
- `M3` 前端最小可用：反馈录入页 + 审核台 + 版本记录
- `M4` 灰度发布：仅 1-2 个员工启用
- `M5` 全量发布：完成指标验证与回滚演练
