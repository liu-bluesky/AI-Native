# 前后端设计与菜单功能

## 1. 设计目标

- 明确“用户当前项目反馈 -> AI 反思 -> 规则候选 -> 审核发布”的端到端实现边界。
- 补齐前端菜单、页面、交互动作，以及后端接口、服务、存储分层。
- 强制 `project_id` 作为全链路主隔离键，避免跨项目数据污染。

## 2. 前端信息架构

## 2.1 菜单结构

- 一级：`员工管理`
- 二级动作（员工行内）：`进化`、`审核`、`反馈`
- 进化页 Tab：`反馈工单`、`候选审核`、`版本记录`

## 2.2 路由建议

- `/employees` 员工列表
- `/evolution/:employeeId` 进化报告
- `/review/:employeeId` 候选审核
- `/feedback/:employeeId` 反馈工单列表（新增）
- `/feedback/:employeeId/:feedbackId` 反馈详情（新增）

## 2.3 页面功能拆解

- `反馈工单列表`
- 功能：按状态、严重级别、时间、规则筛选；展示 AI 分析状态；支持新建反馈。
- `反馈提交`
- 功能：标题、现象、期望、严重级别、会话 ID、规则 ID，提交后进入 `new` 状态。
- `反馈详情`
- 功能：查看反馈正文、AI 反思结构化结果、证据链、候选规则 diff。
- `候选审核`
- 功能：approve/edit/reject，编辑后显示 diff 预览并二次确认。
- `规则详情版本记录`
- 功能：查看规则版本链、每个版本对应来源反馈与审核记录。

## 2.4 前端接口约束

- 所有请求必须使用项目级路径：`/api/projects/{project_id}/...`。
- 项目切换后清空页面缓存与本地筛选条件，避免读到旧项目数据。
- 返回 `403` 时提示“当前项目无权限访问该数据”。

## 3. 后端设计

## 3.1 分层与文件建议

- 路由层：`web-admin/api/routers/feedback_upgrade.py`
- 服务层：`web-admin/api/services/feedback_service.py`
- 服务层：`web-admin/api/services/analysis_service.py`
- 服务层：`web-admin/api/services/candidate_service.py`
- 存储层：`feedback_bug_store`、`feedback_analysis_store`、`feedback_candidate_store`

## 3.2 接口定义

- `POST /api/projects/{project_id}/feedback/bugs` 创建反馈
- `GET /api/projects/{project_id}/feedback/bugs` 反馈列表
- `GET /api/projects/{project_id}/feedback/bugs/{id}` 反馈详情
- `POST /api/projects/{project_id}/feedback/bugs/{id}/analyze` 触发反思
- `GET /api/projects/{project_id}/feedback/candidates` 候选列表
- `POST /api/projects/{project_id}/feedback/candidates/{id}/review` 审核动作
- `POST /api/projects/{project_id}/feedback/candidates/{id}/publish` 发布版本
- `POST /api/projects/{project_id}/feedback/candidates/{id}/rollback` 回滚版本

## 3.3 状态机

- 反馈状态：`new -> analyzing -> pending_review -> closed`
- 候选状态：`pending -> approved/rejected -> published -> rolled_back`
- 不允许跨级迁移，例如 `pending -> published`。

## 3.4 项目隔离规则

- 写入时：`project_id` 从路由参数读取并落库，不允许请求体覆盖。
- 查询时：SQL 必须包含 `WHERE project_id = :project_id`。
- 关联读取（反馈 -> 分析 -> 候选）必须校验三者 `project_id` 一致。

## 4. 数据落库要求

- `feedback_bug`：保存用户原始反馈与上下文快照。
- `feedback_analysis`：保存 AI 反思结构化结果与原始输出 JSON。
- `rule_upgrade_candidate`：保存规则升级候选、风险等级与置信度。
- `review_decision`：保存审核动作、编辑内容、操作者、时间。

## 5. 现状与缺口

- 已有：当前系统具备演化报告和候选审核页面能力。
- 缺失：反馈工单录入/详情页面、项目级反馈 API、项目隔离字段与校验。
- 结论：当前设计方向正确，但还不是可直接开发的完整前后端设计，本文件用于补齐缺口。
