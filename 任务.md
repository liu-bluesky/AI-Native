聊天功能
AgentTaskRuntime 被以下四个聊天入口直接创建和调用：

- 全局助手 WebSocket：/chat/global/ws，见 web-admin/api/routers/projects.py:12964
- 全局助手普通请求：POST /chat/global，见 web-admin/api/routers/projects.py:13677
- 项目聊天 WebSocket：/{project_id}/chat/ws，见 web-admin/api/routers/projects.py:21244
- 项目聊天 SSE：POST /{project_id}/chat/stream，见 web-admin/api/routers/projects.py:23167

因此删除后，受影响的不只是“智能体功能”，而是：

- 全局助手对话
- 项目聊天
- 流式模型输出
- 多轮工具调用
- 工具执行过程推送
- 聊天过程卡片和运行轨迹
- 任务暂停后继续执行

删除结论
如果只直接删除 services/agent_runtime/：

1. core/server.py 导入失败，API 大概率无法启动。
2. routers/projects.py 导入失败，项目路由无法加载。
3. 全局助手和项目聊天均无法执行。
4. 文件、命令、网络等内置工具失效。
5. 权限确认、工作区信任、暂停恢复失效。
6. 相关测试会大量失败。

所以它不能作为普通废弃目录直接删除。如果目标是“不要项目聊天使用智能体执行”，更合理的是拆除四个聊
天入口对 AgentTaskRuntime 的依赖并提供替代的纯 LLM 对话链路，之后才能安全删除该目录。

附注：本轮 MCP 自动任务树错误生成成了治理模板，因此任务树闭环未完成；以上结论来自本地代码的直接引
用检查。
