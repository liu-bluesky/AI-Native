Hermes 的核心做法是分层：

1. 工具层只保证契约和写入安全
   write_file(path, content) 仍然要求 content 是字符串，不接受对象。工具层负责原子写入、stale file 检查、并发写
   保护、BOM/换行保留等，不负责判断 package.json 业务上写得好不好。

2. 写完后走通用诊断层
   Hermes 在 write_file/patch 后接 lint 和 LSP diagnostics。代码里明确说，TS/Go/Rust 的单文件命令会产生假阳性，
   所以用 LSP 替代：tsserver、gopls、rust-analyzer。见：
   /Volumes/work_mac_1_5T/hermes/hermes-agent/tools/file_operations.py:397
   /Volumes/work_mac_1_5T/hermes/hermes-agent/tools/file_operations.py:1236

3. 语言支持靠 LSP registry，不靠每种语言写一个工具
   Hermes 有统一的 LSP server registry：pyright、typescript、vue-language-server、gopls、rust-analyzer、clangd
   等。新增语言本质是新增一个诊断 provider，不是新增一套 agent 工具。见：
   /Volumes/work_mac_1_5T/hermes/hermes-agent/agent/lsp/servers.py:831

4. 扩展靠 hooks，而不是改核心
   Hermes 支持 pre_tool_call、post_tool_call、pre_llm_call 等 hook。比如 post_tool_call 可以匹配 write_file|
   patch 后执行 formatter、policy checker、项目自定义校验。见：
   /Volumes/work_mac_1_5T/hermes/hermes-agent/cli-config.yaml.example:1091

所以正确设计不是“加 package.json validator”，而是：

你的 agent 要加的是 Verification Gate / Acceptance Gate。

它应该这样工作：

1. 项目发现
   自动识别当前项目有什么：package.json、Cargo.toml、go.mod、pyproject.toml、pom.xml、Makefile、CI 配置、已有脚
   本。

2. 生成验证计划
   不是让模型随便写“验证方案”，而是 runtime 根据项目事实生成：
   - 改了 JSON/YAML/TOML：先做语法解析。
   - 改了 TS/Vue：跑 LSP/typecheck/build。
   - 改了 Rust：跑 cargo check/test。
   - 改了 Go：跑 go test ./... 或 gopls。
   - 有项目脚本：优先使用项目已有 lint/typecheck/test/build。
   - 没有验证能力：明确标记 verification gap，不能说完成。

3. 执行后收集证据
   每次工具写入后记录 changed files，然后 gate 决定要跑哪些检查。检查结果回灌模型，不是只靠模型自述。

4. 语义质量用项目规范和 reviewer，不塞进工具
   你举的 package.json 里 "latest"、依赖分组不合理，这种很多时候 build 也能过。它应该由两类东西拦：
   - 项目/组织 policy hook，例如“禁止 latest 依赖”。
   - 独立 reviewer pass：拿用户目标、diff、诊断结果、项目现有风格，输出结构化 verdict。

5. 大型迁移要分批闭环
   对大型迁移，不能一次性“全项目改完再验证”。应该是：
   - 发现项目结构和验证能力。
   - 建 migration plan。
   - 分批改动。
   - 每批跑局部验证。
   - 最后跑全量验证。
   - 任何验证缺口都显式报告。

结论：你之前的问题不是“AI 不聪明”，也不是“提示词不够专业”。根因是 agent runtime 缺少通用的验证闭环。Hermes 的方
向是：工具严格、诊断通用、扩展走 hook、语义质量走项目规范和 reviewer。你这个 agent 也应该按这个架构修，而不是继
续给前端、后端、配置文件分别写硬编码 validator。
