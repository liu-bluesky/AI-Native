• 对，stdio MCP 只是 MCP 的一种传输类型。

可以这样理解：

MCP = 一套工具/资源/提示词调用协议
stdio / HTTP / SSE / Streamable HTTP = 这套协议跑在什么通道上

常见 MCP 类型大概是：

类型 运行位置 连接方式 典型场景
━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
stdio MCP 用户本机启动一个进程 stdin/stdout 本地文件、本地数据库、本地命令行工具
───────────────────── ────────────────────── ──────────────────── ──────────────────────────────────────
HTTP MCP 服务端或远程机器 HTTP 请求 后端统一 MCP、远程工具服务
───────────────────── ────────────────────── ──────────────────── ──────────────────────────────────────
SSE MCP 服务端或远程机器 SSE 长连接 + HTTP 老一些的远程 MCP 实现
───────────────────── ────────────────────── ──────────────────── ──────────────────────────────────────
Streamable HTTP MCP 服务端或远程机器 新版 HTTP 流式协议 新版远程 MCP 推荐方向

所以：

- stdio 不是“本地外部 MCP”的同义词，它只是 MCP 的一种 transport。
- 内部/外部 MCP 是产品管理概念，不是 MCP 协议概念。
- 项目 MCP、系统 MCP、用户配置 MCP、本机 MCP 都可以统一放在一个 MCP 配置中心里。
- 每个 MCP 配置项只需要标记它的 transport：stdio、http、sse 等。

更合理的用户视角是：

MCP 服务

- 名称
- 启用状态
- 作用范围：全局 / 项目 / 当前对话
- 传输类型：stdio / http / sse
- 连接配置：command 或 url

.ai-employee/mcp-adapter/servers.json 只对应其中一种情况：当某个 MCP 服务的 transport 是 stdio，需要桌面端在本机启动进程时，
桌面端派生出来的运行配置文件。
